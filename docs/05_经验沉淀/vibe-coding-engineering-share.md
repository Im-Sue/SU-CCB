# 从 Vibe Coding 到工程化管理

## 分享定位

- 主题：结合 CCB 的 Claude + Codex 协作设计，分享 Vibe Coding 的工程化管理思路和工作流逻辑
- 形式：对话式、互动式、交流式分享，不走纯 PPT 讲授
- 时长：约 60 分钟
- 目标：
  - 先让大家对齐当前 Vibe Coding 的真实痛点
  - 再把这些痛点上升为工程管理问题
  - 结合 CCB 讲清楚为什么需要角色分离、工作流和协议
  - 最后讲清楚什么场景值得工程化，什么场景不值得

## 一句话主结论

Vibe Coding 解决的是产出速度，工程化管理解决的是稳定做对。

## 开场方式

### 你可以这样开场

今天我不想讲怎么把 prompt 写得更花，也不想讲哪个模型更强。我更想和大家讨论一件事：AI 已经很会写代码了，为什么我们做项目的时候还是经常不放心？

### 开场互动问题

1. 大家现在用 AI 写代码，最大的好处是什么？
2. 最大的痛点是什么？
3. 有没有人遇到过：第一轮写得很好，第二三轮开始方向就偏了？

### 开场归纳

等大家回答以后，你可以收成一句：

> 我们今天的问题已经不是 AI 能不能写，而是 AI 能不能持续做对。

然后再接：

> 所以今天我想分享的不是 prompt 技巧，而是 Vibe Coding 如何做工程化管理。

## 第一部分：先让大家把问题说出来

### 互动目标

让现场先有共鸣，不要一上来就讲方案。

### 你可以引导大家说的几个问题

- AI 写得很快，但方向容易错
- 上下文会漂移
- 改着改着和原始需求脱节
- 返工成本高
- Review 成本大
- 很难复盘为什么这么改

### 你可以这样总结

大家刚刚说的这些问题，表面上看像是 AI 使用体验问题，但本质上其实是工程问题。

### 这里插入 CCB 的第一层讲解

我们现在做 CCB，其实不是为了让模型再聪明一点，而是想把 AI 协作从灵感式对话，变成一个更稳定的工程过程。

### 这里要落的核心观点

工程化管理不是为了限制 AI，而是为了解决三件事：

- 可控性
- 可复用性
- 可审计性

### 可以现场说的金句

> AI 已经解决了写得快，但还没有自动解决持续做对。

---

## 第二部分：为什么需要工程化管理

### 你可以这样过渡

如果一个人和 AI 的协作结果，最后还是靠人自己从头到尾盯着、兜着、修着，那它本质上还不是一个稳定的工程流程，它只是一个效率更高的助手。

### 互动问题

你们觉得，AI 协作里最贵的成本到底是生成代码，还是确认代码真的对？

### 你要引导出来的答案

不是生成最贵，确认最贵。真正贵的是：

- 方向走偏后的返工
- 上下文丢失后的重复沟通
- 审查时重新理解背景
- 团队协作时无法复盘

### 这里插入 CCB 的第二层讲解

CCB 的核心思路，是把原来散落在人脑里的 Planning、Execution、Review 这几个动作显式化，让它们变成一个可以重复、可以约束、可以审查的流程。

### 你可以画的极简流程

```text
先理解问题
  -> 再做方案
  -> 再切任务
  -> 再执行
  -> 再验证
  -> 再审查
```

### 这里要落的结论

工程化管理的意义，不是让过程更重，而是让高风险环节先被管住。

## 第三部分：为什么是 Claude + Codex，而不是单 Agent 或堆更多 Agent

### 这一段的讲法

不要讲成模型 PK，要讲成角色设计。

### 互动问题

如果让一个 agent 同时做需求理解、方案设计、任务拆解、编码、验证、审查，大家直觉觉得会更高效，还是更容易混乱？

### 你可以这样承接

我自己的结论是：不是 agent 越多越工程化，也不是一个 agent 全包就最省事。真正重要的是职责边界清不清楚。

### 这里插入 CCB 的角色设计

在 CCB 里，我们做了一个非常明确的分工：

- Claude 更像决策者
- Codex 更像执行者

### 你可以这样解释角色

Claude 负责：

- 需求理解
- 技术方案
- 任务切片
- 审查把关

Codex 负责：

- 落地实现
- 执行验证
- 遇到边界问题时主动回抛

### 这里要强调的重点

这个设计的关键不是模型偏好，而是工程角色分层。我们不是在比较哪个模型更强，而是在定义：

- 谁负责想清楚
- 谁负责做出来
- 谁负责在关键节点停下来确认

### 这里可以顺带穿插当前 CCB 的工作方式

比如在 CCB 里，`/su:plan` 本质上是先把需求、方案和任务边界想清楚；到了执行阶段，再把明确任务交给 Codex 去做实现和验证；执行完以后，再回到审查环节确认是不是按设计做对了。

### 这一段的金句

> 不是 agent 越多越工程化，而是边界越清晰越工程化。

## 第四部分：工程化管理的关键，是工作流和协议，不是提示词和 skills

### 这一段是全场重点

### 互动问题

如果 prompt 足够长、skills 足够多，AI 协作就一定稳定吗？

### 你可以这样回答

不一定。prompt 和 skills 提升的是能力上限，但工程化管理解决的是协作稳定性。

### 这里插入 CCB 的核心讲解

CCB 真正关键的不是某条提示词，而是工作流和协议。真正影响协作质量的，是下面这些东西：

- 阶段划分
- 输入输出契约
- 审批门
- 回抛规则
- 回执标准

### 这里建议你只讲 4 个 CCB 机制

#### 1. 三模式：consult / explore / execute

```text
consult：先协商，不改代码
explore：先勘探，理解现状和风险
execute：真正实施并验证
```

你可以解释：

- consult 适合在需求理解和方案设计阶段使用
- explore 适合理解遗留系统、判断影响面和风险
- execute 适合方案已经稳定后的实施阶段

#### 2. 审批门

不是 AI 想到什么就一路做下去，而是在关键节点确认方向，避免正确执行了错误目标。

#### 3. 回抛机制

执行过程中遇到边界问题，不要求 Codex 硬做，而是要求它主动回抛不确定性。这样可以避免做完才发现前提错了。

#### 4. 精简回执

执行结束后，不要求审查者重新读完整 diff 才知道发生了什么，而是先看一份结构化结论：改了什么、为什么改、怎么验证、有什么风险。

### 你可以现场口述的完整链路

```text
用户提出问题
  -> Claude 判断是否需要协商
  -> Codex consult / explore
  -> Claude 确认方案和任务切片
  -> Codex execute
  -> Codex 输出精简回执
  -> Claude review
```

### 这里要落的核心结论

如果没有协议，AI 协作就是高级聊天；有了协议，AI 协作才开始像工程。

### 关于 skills 的一句解释

skills 很重要，但它更像能力插件，类似 tools 的增强层。没有工作流和协议，skills 只能让 agent 更能做事，不能自动让协作变得可控。

## 第五部分：什么场景值得工程化，什么场景不值得

### 互动方式

用判断题，不要单向讲。

### 你可以直接抛的几个场景

1. 单文件小修，值不值得上完整工程化流程？
2. 跨 4 个模块改接口，值不值得？
3. 遗留系统改造，文档很少，值不值得？
4. 一次性 demo，值不值得？

### 你可以这样总结判断标准

只要方向错一次的成本很高，就值得做工程化管理。

### 值得工程化的场景

- 跨模块改动
- 接口或 Schema 变化
- 遗留系统改造
- 安全、性能、稳定性敏感路径
- 多人协作项目
- 需要复盘和审计的任务

### 不太值得工程化的场景

- 单文件小修
- 一次性脚本
- 原型验证
- 很明确的小 bugfix

### 结合 CCB 的落地说法

CCB 这类设计不是为了把所有任务都变重，而是为了把高风险任务先管住，把低风险任务保留快速通道。

### 你可以这样收尾

如果今天只留三句话，我希望大家记住的是：

1. Vibe Coding 解决的是产出速度。
2. 工程化管理解决的是稳定做对。
3. CCB 里 Claude + Codex 的价值，不是双模型，而是决策与执行的显式分工。

如果你们回去只做一件事，不是换模型，而是给 AI 协作加一个审批门、一个回抛点、一个回执标准，你们觉得项目稳定性会不会立刻提升？
