## 需求描述

## 原话（verbatim）

需求管理的tab改成 TasksBoardView 的效果。

## Claude 解读

把「需求管理」页面（`apps/ccb-console/web/src/pages/requirements/RequirementsPage.tsx`）从当前「SegmentedControl 4 tab + 纵向卡片堆叠」改造成与「任务看板」（`TasksBoardView`）一致的看板（kanban）视觉与交互。

**核心改造**
- 现有 4 个分类桶（待处理 / 推进中 / 已交付 / 已归档，见 `classifyRequirementTab`）转为看板 4 列并排展示。
- 每个需求渲染为看板卡片：标题 + 状态徽标 + 描述摘要 + 主操作入口；裁掉任务卡片特有的优先级条 / 进度条 / 节点徽标（需求无这些字段）。
- 复用任务看板的栅格 / 列容器 / 空列占位视觉（`TasksPage.module.css` 的 `.kanban` / `.column` 等），保证两页观感一致。
- 参照 `NODE_BOARD_COLUMNS` 模式，为需求生命周期新建列配置（建议 `REQUIREMENT_BOARD_COLUMNS`）+ projection 函数（建议 `createRequirementBoardProjection`），按 status 分组到列。

**边界**：仅前端 `apps/ccb-console/web/src/`；不动后端、需求 status 枚举、数据模型、CCB 内核 / 协议。需求 status 真相源仍是 requirement md，Console 仅投影。UI/UX 设计 / 交互 / 实施由 Claude 独自负责（用户明确授权）。

## 歧义点

均属 UI/UX 设计选择，用户已授权「由 claude 独自完成」，默认按"倾向"推进，技术设计阶段细化：

1. **替换 vs 切换**：「改成」按字面 = 替换当前 tab + 列表。`TasksBoardView` 本身有 board / list 双模式——需求页是否也保留 board/list toggle？倾向：默认看板，toggle 视实现成本可选。
2. **列方案**：用现有 4 桶（待处理 / 推进中 / 已交付 / 已归档）作 4 列最自然、零额外映射。是否把"推进中"拆成 planning / delivering 两列？倾向：先用 4 桶，与列表语义对齐。
3. **卡片信息密度**：verbatim 原话 / Claude 解读 / 歧义点等长内容，是仍在卡片内可展开，还是收进详情避免卡片过高？倾向：卡片只放摘要 + 操作，长内容进详情。
4. **复用粒度**：抽通用 `<KanbanBoard>`（可插拔卡片渲染）供任务 / 需求共用，还是各写一套？属实现细节，留技术设计决定。

## 保真差异

- 原话仅一句「需求管理的tab改成 TasksBoardView 的效果」，未指明列方案、是否保留切换、卡片字段裁剪——这些是补充的设计判断，非用户原文。
- 用户补充「可能原组件是 NODE_BOARD_COLUMNS？」：经核实 `NODE_BOARD_COLUMNS` 是任务看板的工作流列（dispatch / implementation / review / archive），是列配置的范式来源，但语义与需求生命周期不同，不能直接套用——需求看板须另建列配置。
- 「tab」理解为「需求管理这个页面 / 导航入口」，落地为把页面内 SegmentedControl 那组 tab 升级成看板列；若用户指的是别的 tab，需澄清。
- 「仅前端、仅 RequirementsPage 视图层」是分析推断的范围边界，原话未显式划定。
