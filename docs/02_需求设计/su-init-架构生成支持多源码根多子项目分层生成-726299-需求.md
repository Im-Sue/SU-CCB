## 需求描述

su-init 旧项目架构生成把「多源码根 → 跳过」策略升级为支持多源码根、多子项目、多技术框架的分层生成：根目录下的各子项目/源码根（前端、后端、后台管理端、子系统等）各生成一份独立架构文档，子系统之间的关联生成一份总架构文档；具体生成模式在技术设计阶段考虑。

## 原话（verbatim）

哦所以没有生成 我预想的架构文档是因为 我设置了规则“项目里有多个源码根 就不自动生成”？ 那这里要优化，要支持多源码根、多子项目、多技术框架的生成（本质上我们统一要求的是 根目录下的子项目或者多源码根/技术栈其实可能是一套系统存在前端、后端、后台管理端或者其他的更多的后端、子系统。这些都是独立的架构，他们直接有关联是总架构），只是生成模式要考虑设计一下。

## 二、背景与目标

1db492 需求（已交付）给 su-init 旧项目架构生成加了 eligibility gate：多源码根 → 跳过不生成（`su-ccb-claude-plugin/lib/su-init/index.mjs:234-288`，hasMonorepoSignal + markerDirs>=2 判定）。2026-06-06 用户的 Tauri 项目（根目录 Python 主工程 + src-tauri/Cargo.toml）被判 `multiple_source_roots` 跳过，用户不满意并推翻旧拍板。

**目标**：多源码根/多子项目/多技术框架的旧项目不再跳过，升级为分层生成——每个能独立运行的子系统一份架构文档 + 一份描述子系统间关联的总架构；生成模式细节留技术设计。

## 三、讨论与决策

经需求澄清 + 1 轮 Codex 协商（job_ce13ff724a0c / rep_380fecd83d94）+ 用户拍板 3 项：

- **切分规则**：用户拍板「看能否独立运行」——lib 按运行/部署强信号（独立服务入口、workspace member、独立部署配置等）枚举候选子系统并给置信度；框架壳目录（src-tauri、RN android/ 等）默认归并宿主工程，有独立运行证据才单独成篇。判定是启发式 + 证据 + 置信度，不宣称确定性识别部署边界真相（Codex 修正，接受）。
- **覆盖标记**：用户拍板「要标」——生成的架构文档 frontmatter 增加 scope 元数据（字段名留技术设计，候选如 architecture_scope / source_roots / subsystem_id），作为分层身份地基：防重复生成、支持精准补缺。契约/模板同步小改。
- **大仓库封顶**：用户拍板「超了只给总架构」——候选子系统数超上限（数值留技术设计）时，子架构一份不生成，只生成总架构 + 全部候选清单，回执提示用户点名补生成。
- **既有架构共存（Claude 决策，延续旧拍板语义）**：项目已有未标注 scope 的非模板架构 md → 整体跳过分层生成 + 回执候选清单（与原 gate「有架构就跳过」一致，不退化）；只有带 scope 标记的文档参与后续补缺判断。

**Claude 对 Codex 协商的 4 锚点反思**：

- **我同意**：lib 定位从「边界判定」降级为「候选发现 + 证据 + 置信度 + 建议模式」，接口从单 targetPath 升级为候选列表模型（不是改 reason 即可）；补缺第一版保守；「候选身份稳定性 + 部分失败恢复」（稳定 subsystem 命名、写入顺序、总架构不引用未生成成功的子架构）纳入设计输入。
- **我不同意**：indexer/Console 不消费 human_verified 是事实但属独立增强，不进本需求范围；Codex 备选「多根只生成清单总览」作为常规路径被否决——低于用户「都是独立架构」的原话期望（用户最终仅在超限场景接受清单式总架构作为降级形态）。
- **我的盲点**：原把壳归并想成固定名单规则（正确是看运行证据）；未考虑多次运行间 subsystem 身份漂移破坏 missing-only；未考虑多份 AI 架构在 Console 全显 parse success 的呈现问题。
- **接下来**：歧义收敛为用户 3 拍板 + 1 Claude 决策，固化需求后停在分析完成态，等用户意图进技术设计。

**sc 使用记录**：`/sc:analyze --focus requirement-clarity`（已跑，产出模糊词/隐含假设扫描与歧义分级）；`/sc:research` 以自有领域知识覆盖（C4 system landscape→container 分层、arc42、monorepo 文档惯例），无外部检索缺口；`/sc:business-panel` 以成本/风险/用户负担三维复核替代，结论已并入歧义 2/3 与拍板项。

## 四、功能 / 范围

1. **lib（确定性、无 LLM）**：子系统候选发现——枚举候选组件，输出证据（marker、入口、workspace 成员关系、部署配置）、置信度、建议模式（single / layered / overview_only）；接口模型从单 targetPath 升级为候选列表。
2. **框架壳归并**：src-tauri 等框架壳默认归并宿主；有独立运行证据才单独成候选。
3. **agent 层分层生成**：候选 1 个 → 现行单份路径；≥2 个且 ≤上限 → 每个子系统一份架构 + 一份总架构；超上限 → 只生成总架构 + 全部候选清单。
4. **scope 元数据**：生成文档 frontmatter 带覆盖范围标记；契约/模板/schema 同步（additive）。
5. **总架构证据底线**：子系统清单 + 各自职责 + 仅写有代码证据的关联；推不出的标「未推断」留人工，不编造。
6. **回执**：列出生成清单 / skipped 候选 / 跳过原因，沿用 AI 生成提醒与 review 建议。

## 五、业务规则

- 沿用 missing-only：绝不覆盖任何已存在文件；多文件生成用防覆盖写入（wx 语义）。
- lib 无 LLM；正文生成在 agent 层。
- 生成文档沿用 `generated_by: su-init-ai` + `human_verified: false` 信任标记，并新增 scope 元数据。
- 已有未标注 scope 的非模板架构 md → 整体跳过 + 候选清单。
- 总架构不得引用未生成成功的子架构（部分失败恢复）。
- subsystem 候选命名稳定：同一项目多次运行身份不漂移。

## 六、边界 / 不做项

- 仅 su-init 时机；不做增量补生成（后续新增子系统靠用户点名或另立需求）。
- 不做 indexer / Console 对 human_verified、scope 元数据的消费与展示（独立增强）。
- 不自动生成 02_需求 / 04_模块规格 等其他文档（沿用 1db492 边界）。
- lib 不引入 LLM 依赖。
- 不深入 git 子模块内部生成架构（子模块有自己的仓库；作为候选的呈现方式留技术设计）。

## 七、开放问题 / 假设

**假设**：
- 用户接受生成后自行 review，不阻塞初始化（沿用 1db492 拍板）。
- CLI 纯跑只得候选发现结果，不生成正文。

**留给 technical_design 的设计决策**：
- 候选发现的强信号集合与置信度算法（含 root aggregator、嵌套 package、examples/fixtures 的排除）。
- scope 元数据字段名与契约/schema/模板改法（additive）。
- 子系统数上限数值。
- 总架构文档的命名与模板章节结构。
- git 子模块作为候选的呈现。
- 多文件写入顺序与失败恢复细节。
- 接口签名（candidates 列表结构、建议模式枚举）。

## 八、拆分预览

粗分（最终切片在 task_breakdown 节点定）：
1. **lib**：候选发现接口升级（candidates + 证据 + 置信度 + 建议模式）+ 测试。
2. **契约/模板/schema**：scope 元数据字段 + 架构模板适配分层 + schema additive 校验。
3. **SKILL（agent 层）**：分层生成规程（单份/分层/仅总架构三模式）、总架构证据底线、回执格式。
4. **文档**：su-init SKILL 文档修订 + 对 1db492 交付链的修订说明。

## 九、数据(草案)

不涉及 DB schema 变更；candidates 为返回值/内存结构。架构文档 frontmatter 新增 scope 字段（additive，具体名留设计），旧文档不受影响。

## 十、接口(草案)

- `detectArchitectureCandidate` 升级或新增导出：返回形如 `{ mode: "single"|"layered"|"overview_only", candidates: [{ id, path, evidence, confidence, targetPath }], overviewTargetPath }`（签名留技术设计）。

## 十一、界面 / 页面布局

无 UI 改动；产出为 `docs/01_架构设计/` 多份文档 + su-init 回执文本。

## 十二、交互 / 流程

1. 用户跑 `/ccb:su-init`。
2. lib 三步脚手架（不变）→ 候选发现。
3. 0 个候选 → no_source 跳过；1 个 → 单份生成（现行路径）；2..上限 → 分层生成（子架构×N + 总架构）；超上限 → 只生成总架构 + 全部候选清单。
4. 已有未标注 scope 的架构 md → 整体跳过 + 候选清单。
5. 回执：生成清单 / skipped / 原因 / AI 生成与 review 提醒。
6. 用户事后 review；要补点名补生成（本需求内仅提示，不实现增量命令）。

## 十三、风险

- **启发式误判子系统边界**（root aggregator、嵌套 package、examples 被当候选）→ 缓解：强信号 + 置信度 + 框架壳默认归并 + 人工 review。
- **总架构编造子系统关联**（最大风险，1db492 同源）→ 缓解：证据底线，仅写可证实关联，未推断显式标注。
- **多文件部分失败**留下断链总架构 → 缓解：写入顺序约束 + 总架构不引用失败子架构 + 回执如实列出。
- **生成爆炸成本** → 缓解：上限封顶，超限只总架构。
- **scope 元数据契约改动影响 indexer 解析** → 缓解：字段 additive，旧文档零影响；不要求 indexer 消费。

## Claude 解读

「支持多源码根/多子项目/多技术框架的生成」= 把 1db492 的「多源码根→跳过」gate 升级为分层生成模式：lib 以「能否独立运行」为核心信号做确定性候选发现（框架壳目录默认归并宿主工程），agent 层按候选数选择单份 / 分层（N 子架构 + 1 总架构）/ 仅总架构三种模式；「他们之间有关联是总架构」= 总架构只写有代码证据的子系统关联，推不出的显式标注、不编造。生成模式细节（上限数值、字段名、模板章节）用户明确留给技术设计。仍受 1db492 未推翻拍板约束：missing-only 不覆盖、lib 无 LLM、generated_by/human_verified 信任标记、人工 review 即门。
## 歧义点

1. 切分维度（P0·已拍板）：「源码根/子项目/技术栈」是三种切分法，同一项目结果不同（触发本需求的 Tauri 项目按源码根=2、按业务端=1）→ 用户拍板「看能否独立运行」：框架壳（src-tauri、RN android/ 等）默认归并宿主，有独立运行证据才单独成篇。
2. 生成数量与成本（P0·已拍板）：大 monorepo 可能 10+ 候选，token 成本与幻觉风险线性放大 → 用户拍板「超上限只生成总架构 + 全部候选清单，用户点名再补」；上限数值留技术设计。
3. scope 身份（P1·已拍板）：防重复生成与精准补缺需要文档自声明覆盖范围 → 用户拍板「要标」：生成文档 frontmatter 增 scope 元数据，契约/模板同步小改。
4. 既有架构共存（P1·Claude 决策）：未标注 scope 的既有非模板架构 md 无法机器判定覆盖了哪个子系统 → 整体跳过分层生成 + 回执候选清单，延续原 gate 保守语义不退化；仅带 scope 标记的文档参与补缺判断。
5. 总架构证据底线（P1·协商收敛）：子系统间关联（API 调用/部署拓扑/数据流）正是代码最难推断的部分 → 总架构最低形态 = 子系统清单 + 各自职责 + 仅写可证实关联（import 方向/HTTP client 指向/配置引用），未推断显式标注。
6. 时机范围（P2·默认收窄）：仅 su-init 时机；增量补生成不做，超限/跳过候选靠用户点名补。
