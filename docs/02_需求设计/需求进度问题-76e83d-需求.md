## 原话（verbatim）

需求里 绑定了slot 或者执行了 需求分析，需求就应该直接变为推进中。而不是待处理/草稿。现在从需求管理里 查到的需求一直在 待处理

## 二、背景与目标

需求大状态(canonical `status`,真相在需求 md frontmatter,Console 仅投影)目前在「绑定 slot」「执行需求分析」两条路径上都不流转,导致已开始推进的需求在需求管理看板仍显示在「待处理」列,与用户对进度的直觉不符。

目标:让需求在任一推进事件(绑定 slot / 执行需求分析)发生后,canonical `status` 自动从 `drafting` 晋升为 `planning`(看板「推进中」),且晋升 forward-only、幂等、不越权写 canonical。

## 三、讨论与决策(Codex 协商纪要 + 用户决策)

- **A 写入边界**:采用「新增 capability-outcome policy(c)+ Console 绑定后触发 plugin-side outcome(a)」。Console 不直接写 canonical,由 plugin lib `applyCapabilityOutcome` 写 md。
- **B 目标态**:`planning`(不写 delivering,不复活已废弃的 analyzed)。
- **C 可复用框架**:无现成 `drafting→planning` 机制;复用 `applyCapabilityOutcome` + `applyRequirementMarkdownEffects`(capability-outcome/index.mjs:154、state-effects.mjs:90);Console 投影已能从 md explicit status 更新 DB(project-indexer.ts:1049);**不得**用 rollup(requirement-status-rollup.ts)改 canonical。
- **D forward-only 守卫**:drafting→planning;planning no-op;delivering/delivered/deferred/cancelled 拒绝不覆盖;必须以 md `expectedHash/base_hash` 做 CAS;不破坏 anchor.routes.ts:356 的 `["drafting","planning"]` planning anchor 启动门控。
- **用户决策(2026-06-01)**:目标态 planning;解绑 slot 不回退;批准进入技术设计。
- **深度提示**:Codex 建议 `sc-design`(跨层状态边界,confidence: medium)。

## 四、功能 / 范围

- 触发点 1:`applyRequirementAnalysis` 成功落盘后,触发 drafting→planning 晋升。
- 触发点 2:slot 绑定成功后,触发 plugin-side drafting→planning 晋升 outcome。
- 晋升统一经同一 capability-outcome policy + plugin lib 写 md,Console 投影同步。

## 五、业务规则

- 晋升 forward-only 幂等:仅 `drafting → planning` 生效;源态为其它状态时 no-op 或拒绝,不降级、不复活。
- 解绑 slot 不触发状态回退(用户决策)。
- canonical 真相唯一来源为需求 md frontmatter;所有写入走 plugin lib 的 safeWriteFile/CAS/EventJournal。
- Console 永不直接写 canonical status,只作为触发器调用 plugin-side outcome。

## 六、边界 / 不做项

- 不改 delivering 的产生方式(仍由子任务执行后 rollup/显式 outcome 决定)。
- 不复活旧的 `analyzed` 状态与 disabled 的 `requirement.analyze:analyzed` policy。
- 不处理解绑 slot 的状态回退(用户已决策不回退)。

## 七、开放问题 / 假设

- 歧义点 1、2 已用户拍板;3(分析触发时点)倾向 applyRequirementAnalysis 成功落盘,设计期最终确认。
- 假设 Console 投影对 md status 变更的延迟可接受,必要时触发 scan 或依赖 watcher。

## 八、拆分预览(供 task_breakdown 参考,非最终)

1. capability-outcome policy 新增/启用 `... → planning` + regenerate 两端 generated policy。
2. plugin lib:`applyCapabilityOutcome` 的 planning promotion executable guard(forward-only/幂等/CAS)。
3. 触发点 1:`requirement-analysis/index.mjs` 分析成功后触发同一 outcome。
4. 触发点 2:Console slot 绑定成功后触发 plugin-side outcome(避开吞错的 notifySlotBound,需可观测失败处理)。
5. 测试更新:修正 `anchor-requirement-dispatch.routes.spec.ts:139` 等锁定「绑定后仍 drafting」的断言;补 forward-only/幂等用例。

## 十三、风险

- `requirement.reanalyze` 允许 `delivering/deferred`,无 guard 会误降级/误复活——晋升守卫必须覆盖。
- Console `slot_bound` 是 DB EventJournal,plugin evidence registry 读 `docs/.ccb/events/journal.jsonl`,两者不可直接互当 evidence。
- md 写完到 Console 展示存在投影延迟,需触发 scan 或明确依赖 watcher。
- 修改受测试守护的现有契约,需同步更新断言,避免回归。

## Claude 解读

用户要求:需求在「绑定 slot」或「执行需求分析」任一事件发生后,大状态应自动从草稿/待处理晋升为「推进中」,而不再停留待处理。

根因(代码+测试双证据,经 Codex 协商确认):
- `applyRequirementAnalysis`(su-ccb-claude-plugin/lib/requirement-analysis/index.mjs:331)只写分析 section + `analysis_input_hash`/`analysis_applied_at`,不写 `status`。
- slot 绑定(slot-binding.service.ts:69)只写 `slotBinding` 与 `slot_bound` 事件;planning anchor 只写 `planningAnchorId/planningRuntimeState`;`anchor-requirement-dispatch.routes.spec.ts:139` 还**显式断言绑定后 status 仍为 drafting**。
- 看板按 canonical `classifyRequirementTab(requirement.status)` 分桶(ui-mapping.ts:279):`drafting→待处理`,`planning|delivering→推进中`。
- 故两条路径走完 canonical 仍是 drafting,需求恒留待处理列。属 canonical 未流转,非投影盖过 bug。

关键认知:当前「绑定后不晋升」是受测试守护的**显式契约**,本需求实质是「重新定义状态晋升契约」,而非单纯修 bug。

推荐方案(与 Codex 一致,已获用户批准):新增/启用一个 capability-outcome policy 实现 `drafting → planning` 晋升,由 plugin lib `applyCapabilityOutcome` + `applyRequirementMarkdownEffects` 写 md frontmatter(CAS/EventJournal);Console 端绑定 slot 成功后只**触发**该 plugin-side outcome,绝不直接写 canonical(遵守 ADR-0034/0037:Console 只投影)。目标态收敛为 `planning`。晋升 forward-only 幂等:仅 `drafting→planning`,`planning` no-op,`delivering/delivered/deferred/cancelled` 拒绝不覆盖。

用户决策(2026-06-01,step1_approval 门通过):①目标态=planning;②解绑 slot 不回退状态;③批准进入技术设计节点。
## 歧义点

1. **目标态 = planning**。✅【已定·用户批准】非 delivering / 已废弃的 analyzed;delivering 留给子任务执行后 rollup。
2. **解绑 slot 不回退 drafting**。✅【已定·用户批准】晋升为单向语义,解绑只释放资源,推进事实不撤销;无需降级守卫。
3. **「执行了需求分析」的精确触发点**:建议以 `applyRequirementAnalysis` 成功落盘为客观触发,不额外要求用户二次 approve。【倾向已定,设计期最终确认】
4. **两个触发本次都做**:原话为 OR,两者均纳入本需求范围。
5. **触发顺序/重复幂等**:先绑 slot 后分析(或反序),第二次晋升 no-op(已 planning),由 forward-only 守卫保证。
6. **Console→plugin 写入桥的可靠性**:slot 绑定成功后触发 plugin-side outcome 不能塞进当前吞错的 `notifySlotBound`(失败会静默);需可观测的失败处理。【实现期决策,Codex 已标注】
## 保真差异

- 原话只说「推进中」,未区分 planning/delivering——已收敛到 `planning`(用户批准)。
- 原话未提「解绑是否回退」与「分析触发的精确时点」——解绑不回退已经用户拍板;分析触发时点倾向 applyRequirementAnalysis 成功落盘,设计期最终确认。
- 原话未提实现落点——方案层(capability-outcome policy + plugin lib 写 + Console 仅触发)为遵守 ADR-0034/0037 的工程推导,最终落点待技术设计细化。
- 未引入原话之外的新功能诉求。
