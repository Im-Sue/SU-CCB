## 需求描述

需求详情里的子任务现在点击是跳转到任务详情，现在已经没有意义了，直接点击弹窗展示对应的任务文档内容即可

## 原话（verbatim）

需求详情里的子任务现在点击是跳转到任务详情，现在已经没有意义了，直接点击弹窗展示对应的任务文档内容即可

## 二、背景与目标

- 背景：commit 84e013f 给需求详情页加了 reader 弹窗（AI 解析 / 技术设计），但子任务行仍停留在「点击跳转任务详情页」的旧交互，与同页其它产物的「内联阅读」体验不一致。
- 目标：让需求详情页的子任务也能像 AI 解析 / 技术设计一样，就地内联阅读其开发任务文档，不被迫跳出当前阅读上下文。

## 三、讨论与决策

经一轮 Codex 协商（job_d21ff6bd9aeb，mode: consult）+ 4 锚点反思，确立以下决策：

- 决策 1：只改需求详情页子任务点击交互，保留任务详情页 / 路由（多入口在用，已核验）。
- 决策 2：点击子任务打开只读 reader 弹窗，内联展示该子任务 canonical dev_task markdown 正文，复用 Modal(size="reader") + MarkdownViewer。
- 决策 3：取数走后端 canonical 解析（依据已存在的 `Task.primaryDocumentId`，后备 `projectId + taskKey + kind=dev_task`），新增项目级只读端点；前端不做文档选择。
- 决策 4：默认不保留「打开完整任务详情」次要入口（忠实用户原话）；该项为可否决取舍。

## 四、功能 / 范围

范围内：
- 需求详情页子任务行点击 → 打开只读 reader 弹窗，内联渲染该子任务 dev_task 文档正文。
- 新增后端项目级只读端点，返回该 task 的 canonical dev_task md body。
- 弹窗加载 / 空 / 错误态（文档未物化 / 丢失）内联文案。

范围外（明确不做）：
- 不删除 / 不改造任务详情页 TaskDetailPage 及其它入口。
- 不在弹窗内提供编辑 / 流转 / 工作区操作。
- 不做通用相对资产（图片）路径解析（与现有 reader 同限制，留待单独需求）。
- 不改任务列表 API 负载（不往列表塞 linkedDocuments）。

## 五、业务规则

- 「对应的任务文档」= 该子任务（dev_task）的 canonical 开发任务 md，由 `Task.primaryDocumentId` 指向；后备用 `projectId + taskKey + kind=dev_task` 解析。
- 弹窗内容为 markdown body（去 frontmatter），与 AI 解析 / 技术设计 reader 一致。
- 同一 taskKey 出现多个 dev_task 视为投影 / 文档异常，reader 只读 primary，不暴露选择器。

## 六、边界 / 不做项

见「四、功能 / 范围」的范围外清单；核心边界：本需求是需求详情页一处交互的内联化改造 + 一个只读取数端点，不触及任务详情页、不改数据模型、不引依赖。

## 七、开放问题 / 假设

假设（默认采用，可否决）：
- 「任务文档内容」= dev_task md body，不含 frontmatter。
- reader 只读，不提供编辑 / 流转 / 工作区操作。
- 任务详情页继续从看板 / 迭代 / 我的工作 / 命令面板 / 活动流进入。

默认决定（可否决，非悬空 TBD）：
- 不保留「打开完整任务详情」链接。
- 文档缺失文案：「任务文档不存在或尚未进入索引」（弹窗内联，不 toast、不跳转）。
- 弹窗标题：「任务文档 · {task.title}」。

## 八、拆分预览

（仅预览，最终由 task_breakdown 决定）规模较小，倾向：
- 方案一：1 个全栈子任务（后端只读端点 + 前端弹窗接线）。
- 方案二：2 个子任务 ——（a）后端项目级只读端点 + canonical 解析；（b）前端 fetchTaskMarkdown + 子任务 reader state + 弹窗接线 + 空 / 错误态。

## 九、数据(草案)

- 无 schema 变更。复用已有 `Task.primaryDocumentId` 与 Document 投影。

## 十、接口(草案)

- 新增 `GET /api/projects/:projectId/tasks/:taskId/markdown`：
  - 200 → `{ path, content }`（content 为 dev_task md body，去 frontmatter）。
  - 404 → `{ message }`，当 task 不属于该 project 或 dev_task 文档缺失 / 未进入索引。
  - 形态镜像现有 requirement markdown 端点（`/api/projects/:projectId/requirements/:requirementId/markdown`）。

## 十一、界面 / 页面布局

- 复用 Modal `size="reader"` + MarkdownViewer，与 AI 解析 / 技术设计 弹窗同尺寸、同 a11y（focus trap / 初始聚焦 / 焦点恢复）。
- 弹窗标题：「任务文档 · {task.title}」。

## 十二、交互 / 流程

- 用户在需求详情页点击某子任务行 → 不再跳转 → 打开 reader 弹窗，加载该子任务 dev_task md body 并渲染 → 关闭弹窗回到需求详情页（焦点恢复到原子任务行）。
- 加载中：弹窗内 loading；加载失败 / 文档缺失：弹窗内联降级文案，不 toast。

## 十三、风险

- 子任务按钮可能承载其它语义（快捷键 / 焦点）——技术设计需确认仅导航语义后再替换。
- 文档投影过期 / 缺失需优雅降级（已定文案）。
- 去掉直达任务详情入口是有意取舍（用户明示），非回归，审查勿当 bug。
- 相对图片路径仍可能不渲染（与现有 reader 同限制），本需求不扩成通用资产解析。

## Claude 解读

把需求详情页里子任务行的点击行为，从「跳转到任务详情页 /tasks/:id」改为「打开一个只读 reader 弹窗，内联展示该子任务对应的开发任务文档(dev_task markdown)正文」，复用 84e013f 已落地的 Modal(size="reader") + MarkdownViewer 阅读基座，与同页「AI 解析」「技术设计」两个 reader 体验一致。

明确范围与口径：
- 只改需求详情页这一处的子任务点击交互；任务详情页 TaskDetailPage 及 /tasks/:id 路由保留（已核验它仍由看板 / 迭代 / 我的工作 / 命令面板 / 活动流多处进入，非孤立路由）。
- 「对应的任务文档」= 该 dev_task 的 canonical 开发任务 md，由后端依据已存在的 Task.primaryDocumentId 解析（后备 projectId + taskKey + kind=dev_task）；前端不做「挑哪份文档」。
- 弹窗只读，展示 markdown body（去 frontmatter），与现有 reader 一致；不提供编辑 / 流转 / 工作区操作。
- 默认不保留「打开完整任务详情」次要入口（忠实用户「跳转已经没有意义了」的原话）。

验收口径：在需求详情页点击任一子任务，不再发生路由跳转，而是就地弹出只读弹窗并渲染该子任务开发任务文档正文；文档尚未物化 / 缺失时弹窗内联给出降级文案，不 toast、不跳转；关闭弹窗焦点回到原子任务行。
## 歧义点

按节点要求至少列 3 项，逐条给出解决方式（已全部澄清，无悬空 TBD）：

1. 「任务文档内容」指哪份文档？ → 已澄清：指该子任务 dev_task 的 canonical 开发任务 md。Codex 核验 `GET /api/tasks/:taskId` 只查 `kind=dev_task`，且 `Task.primaryDocumentId` 已由 indexer 从 canonical dev_task 写入，故由后端按 primaryDocumentId 解析，无需在前端 / 弹窗暴露文档选择器。我原先「linkedDocuments 可能混入共享技术设计」的担心在当前 API 不成立，已撤销。

2. 「现在已经没有意义了」是否意味着要废弃任务详情页？ → 已用代码证据排除：`/tasks/:id` 仍被 TasksBoardView / SprintsPage / MyWorkPage / ActivityFeed / 命令面板多处使用，任务详情承载 node 流转 / 工作区 / 审查，远不止阅读。判定：本需求只改需求详情页的点击入口，不动任务详情页。该收窄基于证据，非旧 spec anchoring。

3. 是否只是「改一行 onClick」？ → 否。需求详情页子任务只持有 TaskView（无文档 id），需新增取数链路：后端项目级只读端点 + 前端 fetchTaskMarkdown + 弹窗 state。这是本需求的真实主体。

4. 子任务文档尚未物化 / 丢失时怎么办？ → 默认：弹窗内联显示「任务文档不存在或尚未进入索引」，不 toast、不跳转。

5. 是否保留通往完整任务详情的次要入口？ → 默认不保留（忠实用户原话）。属可否决的产品取舍，已在「开放问题 / 假设」显式标注。

## 保真差异

- 原话只说「弹窗展示对应的任务文档内容即可」，未指明文档粒度、取数方式与异常态。本解读在不超出用户意图的前提下补齐：文档 = canonical dev_task md body、后端按 primaryDocumentId 解析、缺失态降级文案、弹窗标题「任务文档 · {task.title}」。这些为默认决定、非用户明示，已在「开放问题 / 假设」标注为可否决。
- 原话「现在已经没有意义了」被解读为「从需求详情页跳走这件事没意义」，而非「任务详情页本身没意义」。该收窄基于代码证据（任务详情页多入口在用），非凭旧 spec anchoring。
- 为达成「内联展示」需新增一个后端只读端点，属实现细节：不改数据模型、不引依赖、不动任务列表 API 负载。用户原话未要求也未排除，作为必要手段纳入，差异在此记录。
