## 二、背景与目标

需求管理页(控制台「需求管理」看板)现在把所有需求按生命周期粗分成 4 个块。三个问题:

- **粒度不够**:「推进中」块其实混了「规划中」和「推进中」两种状态,看板上一眼分不清哪些还在设计、哪些已在交付;而卡片内的 badge 早已分别标「规划中/推进中」,只有列分组没跟上。
- **会越堆越长**:列内容区没有高度上限,需求一多块被无限撑高,整页越拖越长。
- **标题词不达意**:「已归档」块装的是「已暂缓 + 已取消」,这些其实是被搁下的需求,叫「已搁置」更贴切。

目标:让看板分组与卡片已有的细粒度状态对齐、给列限高可滚动、修正标题用词。纯前端展示优化,不动需求状态模型与任何数据。

> 术语:**planning=规划中**(需求已分析、在做技术设计/拆分);**delivering=推进中**(子任务已在交付);**块=看板列**。

## 三、讨论与决策

| 议题 | 决定 | 理由 |
|---|---|---|
| 推进中拆分 | 拆为「规划中」(planning) +「推进中」(delivering) | 与卡片 badge 已有粒度对齐;状态本就是两态 |
| 列顺序 | 待处理 → 规划中 → 推进中 → 已交付 → 已搁置 | 按生命周期,与 kernel status 枚举同序 |
| 已归档改名 | 标题改「已搁置」,桶成员不变,内部 key 保留 archived | 用户只要求改标题;全局替换 archived 会误伤任务看板/文档治理/归档节点 |
| cancelled 归属 | 仍归「已搁置」,不单列「已取消」 | 用户未要求新列;支持重新激活,作聚合桶可接受;badge 仍分别显示细态 |
| 滚动粒度 | 每列内容区独立限高 + 纵向滚动 | "组件长度无限"指各列;标准看板列内滚动 |
| 首页连带 | OverviewPage「活跃」聚合改为含 planning + delivering,文案改「待处理+规划中+推进中」 | 否则规划中需求从首页活跃计数静默丢失 |

## Claude 解读

本需求是对控制台「需求管理」页(su-oriel/web 看板视图 RequirementsPage)的纯展示层优化,共 3 项,均不触碰 kernel 需求状态枚举,也不动任何数据/接口/DB 投影:

1. **拆列**:把当前合并的「推进中」拆成「规划中」(planning)与「推进中」(delivering)两列,与「待处理」同级。现状 lib/ui-mapping.ts 的 classifyRequirementTab 把 planning 与 delivering 都归进 inProgress 桶;拆分即让 planning 单列,inProgress 桶只留 delivering。卡片内 badge 早已分别标「规划中/推进中」,本次让列分组跟上。
2. **改名**:把「已归档」列标题改为「已搁置」;底层桶成员(deferred 已暂缓 + cancelled 已取消)不变,仅可见标题变,内部 key 保留 archived。
3. **加滚动条**:列内容区(.columnBody)当前无 max-height/overflow,需求一多列被无限撑高;给每列内容区加限高 + 纵向滚动,卡片在列内滚动而非撑长整页。

目标态列序(按生命周期,与 status 枚举同序):待处理 → 规划中 → 推进中 → 已交付 → 已搁置。

**实现必做的连带项(非用户新增)**:首页 OverviewPage「活跃需求」聚合现用 tab==="inProgress" 判定,拆桶后必须同时纳入 planning 与 delivering,文案由「待处理+推进中」改为「待处理+规划中+推进中」,否则规划中需求会从首页活跃计数静默消失。

经核验,这套分桶 key/label 仅 web 前端使用(ui-mapping.ts 定义,消费者:requirement-board-config.ts、RequirementsPage.tsx、OverviewPage.tsx 及测试);server/DB/indexer/kernel 只认 canonical Requirement.status 六态,不依赖该 UI 分组,故本需求无后端/协议影响。
## 歧义点

1. **[已解决·默认]「已搁置」语义边界**:「搁置」贴合 deferred(已暂缓),但 cancelled(已取消)是放弃,归入「已搁置」略勉强。
   - 决定:保留原桶成员(deferred + cancelled 同列),仅改可见标题,不新增独立「已取消」列。依据:用户原话是"改标题"非改成员;系统支持 cancelled/deferred 重新激活(su-reactivate),作 UI 聚合桶可接受;卡片 badge 仍分别显示「已取消/已暂缓」,细粒度语义不丢失。

2. **[已解决·默认]「规划中」列位置**:用户只说"与待处理同级",未指定顺序。
   - 决定:按需求生命周期插在 待处理 与 推进中 之间,与 kernel status 枚举顺序一致。

3. **[已解决·默认] 第 3 项"加滚动条"的滚动粒度**:"那些组件的长度似乎是无限的"指向各列(块)。
   - 决定:每列内容区独立纵向滚动(限高 + overflow-y),即标准看板列内滚动,非整页/整板滚动;具体限高数值留技术设计。

**为何只有展示层歧义**:本需求是纯前端分组/标题/滚动调整,不涉及用户权利、数据模型、隐私、成本、合规或不可逆动作,故这些必问维度均不命中;不确定性集中在上述 3 个展示/语义细节,均可由合理默认解,无需打断用户。
## 保真差异

1. **[需用户知悉] 正文与原话条目数不一致**:文档「需求描述」区已有 3 项,但「原话(verbatim)」区仍只有 2 项(缺第 3 项"加滚动条")。两区在需求创建时内容一致,用户随后只在「需求描述」追加了第 3 项。
   - 处理:按节点「范围非锚定」原则,以变更后的完整用户原文(需求描述 3 项)为准,第 3 项纳入本轮;不以陈旧的 2 项 verbatim 收窄范围。
