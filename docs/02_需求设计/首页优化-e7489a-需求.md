## 原话（verbatim）

首页展示的东西太多了。目前整个console 主要关注 需求，task、文档是次要数据。优化一下界面上需要展示的内容和相关的数据。不需要特别复杂。 UI/UX相关的设计、交互、实施由claude独自完成即可

## Claude 解读

本需求经与用户两轮细化后锁定：首页**信息架构重心反转 + 改为"数据概览(数字盘)"形态**。

**现状**（根路由 `/` → `/overview`，见 `apps/ccb-console/web/src/App.tsx`）：`OverviewPage`（`apps/ccb-console/web/src/pages/overview/OverviewPage.tsx`）渲染 9 个区块（引导 banner / 扫描告警 / 4 指标卡 / 活动流 / 索引健康 / 任务节点分布 / 最近文档 + 最近任务 / 最近运行）。三类实体呈现与用户声明优先级**倒置**：需求最低（仅 `indexHealth.requirementCount`，`:170-171`），task 最高（4 区块），且 9 区块中 5 个属系统/运行时信息——这是"展示太多"的主因。

**最终方向（用户拍板）**：
1. 首页 = **数据为主的"数字盘"**，不堆"具体内容"；砍掉所有逐条列表/信息流（最近需求/任务/文档列表、最近运行表、活动流逐条、失败任务逐条详情→降级为"失败 N 次"计数+入口）。
2. **需求为主区，但粗粒度**——仅 3 个核心数字：`活跃` / `需关注`(阻塞+失败+分析过期合并) / `已交付`，复用现有 `MetricCard`；**不做多桶状态分布、不放需求列表**。
3. **次要 = 紧凑聚合计数**：Task（总 / blocked）、文档（总数）；想看明细进各自页面。
4. **健康 = 轻量**：解析失败 + State drift（保留 ADR-0012 active-drift 信号的计数 + 跳转入口）。
5. UI/UX 设计、交互、实施由 Claude 独立完成（用户授权），不派 Codex。

**数据可行性（已核）**：`RequirementView`（`web/src/types/requirement.ts`）提供 `status` / `currentPlanningNode` / `currentPlanningStep` / `planningRuntimeState`(blocked/failed) / `analysisStaleAt` / `rollupProgress`，足以算出活跃/需关注/已交付，**无需后端改动**。

**分级**：complexity = simple→moderate（纯前端聚合 + 页面重排，无后端/schema/API 变更）；impact = moderate-high（控制台主着陆页）；technical risk = low（UI 可逆、无数据风险）。
## 歧义点

原 3 个歧义点经与用户两轮细化已**全部收敛**（处置如下），无遗留高影响未决项：

1. **"次要数据"力度** → 定为：保留为**紧凑聚合计数**（Task 总/blocked、文档总数），非列表、非移除。
2. **"需求主区"形态** → 定为：**粗粒度 3 个数字**（活跃 / 需关注 / 已交付），不做多桶分布、不放需求列表。（用户："需求里也没必要拆那么细"）
3. **系统/运行时面板去留** → 定为：砍逐条列表/信息流；健康收敛为"解析失败 + State drift 入口"；**ADR-0012 active-drift 信号以"计数 + 跳转 `/tasks?projection=stale`"形式保留**，不丢治理信号。

剩余微决策（实现按既定默认）：`需关注` / `drift` 数字保留**可点击跳转**到对应过滤页（已采纳，介于纯数字与列表之间，不算堆内容）。
## 保真差异

- **扩展（已确认）**：原文字面"展示太多→优化"升级为"信息架构重心反转 + 数据概览化"，经用户两轮确认（"数据为主"、"需求不拆那么细"）。
- **撤回**：分析初版曾提"最近需求列表为主"，在用户"不堆具体内容 / 不拆那么细"后**撤回**，改为粗粒度数字盘——这正是与用户协商的产物，记录于此以示保真。
- **收窄**：范围限定根路由 `/overview`（`OverviewPage.tsx`），不动其他页面。侧边栏近期已临时隐藏 任务看板/Reconcile（commit `0a6854d`），方向一致但作为关联项不在本需求内改动。
- **执行边界**：依"UI/UX…由 claude 独自完成"，设计 + 交互 + 实施由 Claude 直接做、不派 Codex；主着陆页高影响已经用户审批门（本轮对话）确认方向后推进 → technical_design + implementation 合并执行。
