## 原话（verbatim）

我看到了顶部的 “通知” 有角标数量，
1. 但是点击之后并没有任何的消息列表、提示、内容
![](./assets/requirements/cmq7mj42v10e7ef5f4eebadfe/688965295ae035dbd0533ea7f9da22df0c15162c55c38aa413949fe4a5388b2e.png)
2. 是否页面上可以加一个 message 消息的提示，这样有新的消息可以更快速的感知到一点。

## 二、背景与目标

**目标对齐**：你在 Console 顶部看到「通知」带角标数字（截图里是 6），但点开只有「投递设置 + 项目暂停」，看不到这 6 条消息到底是什么——这是本次要修的 bug。要做两件事：① 点开「通知」能直接看到消息列表（标题、摘要、时间），点一条就跳到对应的事项去处理；弹窗改成**列表为主，设置收进底部次级入口**；② 页面里加新消息提示（Toast 浮层，= 页面角落自动弹出又自动消失的小提示条），不用授权浏览器通知也能更快感知新消息。另按你的拍板：「项目暂停」期间只**静默打扰、不隐藏消息**（点开仍能看列表）。

背景事实（已核验代码）：通知数据链路本来就是齐全的——后端 attention API（= 返回「需要人注意的消息」列表和数量的接口）已返回完整消息体，已读机制也存在；缺的只是前端没渲染列表。且浏览器通知权限是「未询问」状态时，用户没有任何能看到消息内容的通道，这解释了你的全部体感。

## 三、讨论与决策

| 决策点 | 结论 | 理由 / 否决了什么 |
|---|---|---|
| 弹窗信息架构 | 列表为主，设置降级为底部折叠入口（**用户拍板**） | 用户点击的是「通知」不是「通知设置」（现组件 aria-label 即「通知设置」）。否决「仅在设置上方追加列表」的最小改法 |
| DND 期间列表可见性 | 静默不隐藏：暂停投递但列表/角标仍可见（**用户拍板**） | 符合「免打扰 ≠ 隐藏」直觉；需后端把「投递」与「展示」拆成两层语义。否决「保持 visible=[] 全清空」 |
| 已读（ack）语义 | 打开弹窗不自动清零；单条点击跳转即 ack；提供「全部已读」按钮 | ack 代表「已处理/忽略」而非「看过」，打开即清零会掩盖仍需动作的事项（Claude+Codex 共识） |
| Toast 与浏览器通知关系 | 互斥：页面可见时只 Toast，页面隐藏时只浏览器通知 | 两通道同时打扰没有必要（采纳 Codex 质疑，修正了最初「并行双发」倾向） |
| Toast 触发阈值 | 默认沿用现有 severity=attention 候选集；列表则展示全部 severity | 与既有浏览器通知投递语义一致，技术设计阶段可复核 |
| 范围上限 | 否决「完整消息中心/历史分页」 | 超出本 bug 范围，历史可查 events journal |

## 四、功能 / 范围

使用者：Console 项目页用户。本次共三个工作面：

1. **通知弹窗消息列表**（前端）：渲染当前未读 attention items（标题、摘要、相对时间、severity 标识），单条点击按 cta（= 消息自带的跳转目标）导航并 ack，顶部提供未读数与「全部已读」，设置区折叠到底部。
2. **页面内新消息提示**（前端）：轮询发现新 attention 级 item 且页面可见时，弹 Toast；同批多条聚合为「N 条新通知」。
3. **DND 投递/展示拆层**（后端小改 + 前端文案）：项目暂停期间停止 Toast/浏览器通知/声音投递，但 attention 列表与角标仍可见。

## 五、业务规则

1. 打开弹窗不自动清零角标；ack 仅由「单条点击跳转」或「全部已读」触发。
2. 列表只展示当前未 ack 的 items（全部 severity），不做历史分页。
3. Toast 仅针对 severity=attention 的新 item（与现有浏览器通知候选一致）；同一轮询批次多条时聚合为一条。
4. 投递互斥：页面可见 → Toast；页面隐藏 → 浏览器通知；不双发。声音跟随实际投递通道的开关设置。
5. DND 期间：三种投递（浏览器通知/Toast/声音）全部暂停；列表与角标照常展示并更新。
6. ack 请求失败时不得乐观清除可见项，沿用现有失败提示/重试方式。
7. 不改 7s 轮询间隔（感知瓶颈在呈现通道，不在频率）。

## 六、边界 / 不做项

- 不做历史消息分页、独立消息中心页面。
- 不做跨项目全局通知（仍限定当前选中项目）。
- 不改通知的产生与聚合逻辑（review/consult/devtask/journal/slot 等来源不动），唯一后端改动是 DND 拆层。
- 不引入新依赖、新通知来源、新存储实体。

## 七、开放问题 / 假设

开放问题已全部闭环（见「三、讨论与决策」两项用户拍板）。移交技术设计的技术项：

1. DND 拆层的具体 API 形态（响应里如何同时表达「可展示列表」与「暂停投递」状态）。
2. 「全部已读」走批量 ack 端点还是逐条循环（取决于现有 ack 接口形态）。
3. Toast severity 阈值若在设计阶段发现 warning 类误报率低，可复核是否扩大。

假设：现有 AttentionItem 字段（title/summary/severity/createdAt/cta）足以渲染列表，无需后端补字段。

## 十二、交互 / 流程

```
每 7s 轮询 /attention ──► 角标/列表数据更新（DND 也照常更新展示）
        │
        ▼ 发现新 item（severity=attention）
   DND 暂停中？ ──是──► 不投递（仅展示层更新）
        │否
        ▼
   页面可见？ ──是──► 页面 Toast「N 条新通知」（+声音，若开）
        │否
        └────────► 浏览器通知（+声音，若开）

点击列表单条 ──► 按 cta 跳转 + ack ──► 角标 -1（失败则保留并提示）
点「全部已读」──► 批量 ack ──► 角标清零（失败则保留并提示）
```

端到端示例：你停留在看板页，Codex 完成一个任务产生 attention 消息 → 7 秒内轮询拿到 → 页面可见，右下角 Toast「1 条新通知：xxx 待审查」→ 你点顶部「通知 7」→ 弹窗首条即该消息 → 点击跳转到对应任务详情，该条自动 ack → 角标变 6。

## 十三、风险

| 风险 | 处理 |
|---|---|
| DND 拆层是唯一后端改动，会改变现有「visible=[] 即 count=0」契约 | 技术设计时同步更新依赖该行为的现有测试与角标语义说明 |

## Claude 解读

这是一个「bug 修复 + 轻量增强」复合需求。bug：Console 顶部「通知」按钮有未读角标，但点开的弹窗只有投递设置和项目暂停，没有任何消息内容——已核验为前端 NotificationBell 未渲染消息列表（NotificationBell.tsx:113-167），而后端 attention API（items+count）与 ack 已读机制完整存在，数据链路齐全、纯前端展示缺失。叠加因素：浏览器通知权限「未询问」时，用户没有任何能看到消息内容的通道。
增强：用户希望页面内有 message 提示以更快感知新消息——解读为复用现有全局 Toast 组件，在轮询发现新 attention 级消息且页面可见时弹出，不依赖浏览器通知权限。
经用户拍板的范围：弹窗重排为「消息列表为主、设置降级为底部次级入口」；「项目暂停」(DND) 改为「静默不隐藏」——暂停投递但列表/角标仍可见，需后端把 attention 的「投递」与「展示」拆成两层语义（本次唯一后端改动）。
交付判定：① 点开「通知」能看到未读消息列表并可单条跳转 ack、全部已读；② 页面可见时新消息以 Toast 提示（与浏览器通知互斥）；③ DND 期间不投递但可见。
## 歧义点

全部歧义已闭环（2 项用户拍板 + 4 项共识默认值 + 3 项移交技术设计的技术项）：
1.【用户拍板】DND 期间消息列表可见性——答复：静默不隐藏（暂停投递但列表/角标仍可见，接受后端小改）。理由：免打扰不等于隐藏，符合用户「更快感知」的总目标；2026-06-10 终端答复。
2.【用户拍板】弹窗信息架构——答复：列表为主、设置降级为底部折叠入口。理由：用户点击的是「通知」而非「通知设置」，列表才是主内容；2026-06-10 终端答复。
3.【共识默认】已读语义：打开弹窗不自动清零，单条点击跳转即 ack，提供「全部已读」。理由：ack 是「已处理」而非「看过」，打开即清零会掩盖待办（Claude+Codex 一致）。
4.【共识默认】Toast 噪音控制：同批聚合为「N 条新通知」，DND 期间不弹。
5.【共识默认】范围：仅展示当前未 ack items，不做历史分页/消息中心（否决 O3）。
6.【共识默认】不改 7s 轮询间隔：瓶颈在呈现通道而非频率。
7.【移交技术设计】DND 拆层的具体 API 形态（响应如何同时表达可展示列表与暂停状态）。
8.【移交技术设计】「全部已读」走批量 ack 还是逐条循环。
9.【移交技术设计】Toast severity 阈值默认沿用 attention 候选集（与现有浏览器通知一致），设计阶段可复核是否纳入 warning。
## 保真差异

1. 用户原话第 2 点是问句「是否页面上可以加一个 message 消息的提示」——解读为确认纳入范围的请求（用户自己提出的方案），并具体化为「复用现有 Toast 组件、页面可见时投递、与浏览器通知互斥」；互斥规则是分析补充，非用户原话。
2. 用户原话只说「点击之后没有消息列表」——「弹窗重排为列表为主、设置降级」超出字面最小修复，已经用户显式拍板确认，不算擅自扩大。
3. 用户原话未提及 DND、已读语义、severity、轮询频率——这些是代码核验后补充的关联决策点，其中 DND 已用户拍板，其余为与既有系统语义一致的默认值。
4. 「更快速的感知」未被解读为提高轮询频率（保持 7s），而是解读为打开页面内呈现通道；依据是现有 7s 轮询延迟远小于人工感知阈值，真正缺口是权限未授权时零通道。
