## Claude 解读

**方向（用户已拍板 2026-05-29）**：把「文档中心」从"按 doc_type 扁平筛选的工件列表"重构为 **文档治理中心**——以 docs 目录/档位为导航主线，一眼呈现**文档健康度与覆盖度**，而非再造需求/任务生命周期看板。

**核心价值主张**：

- **导航**：按 docs 目录结构（`00`–`06` + `99_归档`）与档位（生效中/历史/归档）组织文档，取代当前纯 doc_type 筛选 chips。
- **覆盖度**：以需求为锚点显示文档覆盖缺口——哪些需求缺技术设计、哪些需求缺任务文档（消费 indexer 已派生的 `requirementId` / `entityStatus`）。
- **健康度/异常**：突出 parse 异常（`parseStatus=error`）、陈旧文档（stale）、解析失败项。
- **阅读**：保留右侧 Markdown 阅读器（现有 `MarkdownViewer`），阅读体验不退化。
- **边界**：需求 → 设计 → 任务链路只做**摘要入口/跳转**，不复制 Requirements/Tasks 看板的状态流转（避免功能重叠）。

**范围边界（用户已拍板）**：

- 允许 **API 轻扩展**：把 indexer 在 `project-indexer.ts` 构建 document-map 时已派生的 `requirementId / entityStatus / tier / task_id / 覆盖度` 通过文档 API 暴露（或新增 doc-map 端点）。
- **不做 DB migration**：数据已在 indexer 派生，无需新增 Document↔Requirement 关系表或字段迁移。
- **不引入新重依赖**：优先复用现有 UI 组件（`Card` / `Badge` / `EmptyState` / `MarkdownViewer`）；若确需新组件库，属新一轮必问。
- 受众默认**人类 operator**；agent 导航应消费 doc-map/API 而非本 UI。

**验收口径（草案，因用户授权 Claude 主导设计/交互）**：

1. 能按 docs 目录/档位浏览全部文档（替代扁平 chips）。
2. 能看到每个需求的文档覆盖：有无技术设计、有无任务文档；缺失项显式高亮。
3. 突出 parse 异常 / 陈旧文档。
4. 保留 Markdown 阅读器，能从任意文档/缺口跳转阅读。
5. 不与 Requirements/Tasks 看板功能重叠（链路只做摘要）。
6. 仅 API 轻扩展实现，无 DB migration、无新增重依赖。

**执行边界**：用户要求"由 claude 独自完成设计、交互、实施"——解读为 **执行者=Claude**（不派 Codex 实施），但仍走完整节点链（技术设计 → 拆分 → 实施 → 审查 → 归档），且 **🔴 review 门不自审自批**（由 Codex 或用户复核），符合 CCB 硬规则。

**关键技术依据（已读码核验）**：

- 当前列表 API `GET /api/projects/:projectId/documents`（`document.routes.ts:20-32`）只返回扁平字段，**不含 `requirement_id`/frontmatter**；故"按需求分组/覆盖度"必须经 API 轻扩展。
- indexer `buildDocumentMapEntries`（`project-indexer.ts:1531-1551`）**已派生** `path/docType/title/task_id/tier/requirementId/entityStatus/parseStatus`，并写入 document-map 缓存 + `dev_task_paths_by_task_id` 索引——治理中心所需数据基本就绪，缺的是 API 投影与前端消费。

**协商与反思**：经 1 轮 Codex（`slot1_codex`）consult 协商（job_500de1be1b78）。Codex 质疑我原"requirement 树"framing 有 anchoring 现有看板、与 Requirements/Tasks 重叠之嫌，并核验列表 API 不带 `requirement_id`、indexer 已派生关系数据。我据此修正方向并升级用户拍板。4 锚点反思：①同意——列表 API 不带关系字段、document-map 已派生关系数据（均亲自复核）；②保留——方向收敛属用户权利，不由 Codex 代定，故升级为选项；③盲点——原 framing anchoring 现有看板、第一轮高估 migration 风险；④下一步——用户已拍板治理中心 + API 轻扩展，进入技术设计。
