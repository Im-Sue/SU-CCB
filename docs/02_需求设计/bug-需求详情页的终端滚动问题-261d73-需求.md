## 二、背景与目标

需求详情页右栏内嵌的 slot 实时终端（`SlotTerminalPanel → SlotTerminalSurface`，xterm.js）存在滚动缺陷，影响用户观察 agent 实时输出。

- 现象 1：向上翻出现持续重复内容，翻不到正常历史。
- 现象 2：终端存在多层滚动条相互打架，有时只剩当前屏、无法上翻。

目标：让用户能在内嵌终端里干净地看实时输出，并能向上翻读**最近一段真实历史**。

## 三、讨论与决策

- 渲染模型（已核验）：`SlotTerminalFrameRenderer.ts:64` 每帧 `\x1b[H\x1b[2J` 清屏 + 整屏快照重绘；后端 `slot-terminal.frame-stream.ts:63` `tmux capture-pane -p -e` 只抓可见屏、不带历史；`useXtermTerminal.ts` 无 FitAddon。
- Codex 协商（job_5116286d7b21，已独立校验）：重复行的精确触发是"resize 到 pane 行数后写入带尾随换行的满高快照 → xterm baseY 增长 → 重复堆叠"（探针证实去掉尾随 LF 即不堆）；后端确无历史（探针：默认 5 行 vs `-S -` 21 行）；双滚动条独立成立；FitAddon 受"tmux pane 尺寸为真相"约束。
- **方向决策**：用户在 A（完整历史 + 流式，重）/ B（仅干净当前屏）/ B+（干净 + 最近一段真实历史）三档中**选定 B+**，并表示"先看效果再决定是否要更多"。

## 四、功能 / 范围

B+ 范围：

1. **后端**：`capture-pane` 增加历史范围（默认最近 ~2000 行，对齐原生 anchor 终端），让前端拿得到最近历史。
2. **前端渲染**：消除"重复行"根因（满高快照尾随换行 / 重复重绘污染 scrollback），改为能承载历史快照、又不每帧把用户滚动位置拽回的渲染方式。
3. **前端布局**：把三层滚动容器（`.terminalColumn` / `.terminalHost` / `.xterm-viewport`）收敛成单一滚动，消除双 / 多滚动条。

**验收口径（"修好"的定义）**：

- 向上翻不再出现重复行 / 重复内容；
- 终端只有一层滚动条；
- 向上翻能读到最近一段真实历史（目标 ~2000 行）；
- 不再出现"只显示当前屏、无法上翻"。

## 五、业务规则

- 终端 WS 协议：可接收输入 / 粘贴，但**拒绝前端驱动 resize**（`slot-terminal.ws.ts:252` 返回 READ_ONLY）；终端尺寸以 tmux pane 为真相。任何"前端自适应容器尺寸"做法须与该约束兼容。
- 历史窗口默认 ~2000 行；live 刷新维持现有 active 150ms / idle 1s 节奏。

## 六、边界 / 不做项

- 不做 A：不重构为增量 / 流式协议，不重设计 WS 帧模型。
- 深度 / 完整会话历史仍由 ccb 原生 sidebar 兜底（fallback 文案已指向 sidebar）。
- 不改终端可输入语义，不动 paste / 复制逻辑。
- "修复前先 commit 当前版本"为**实施阶段硬约束**，不属本分析范围的代码改动。

## 七、开放问题 / 假设

- 假设：历史窗口 ~2000 行足够（对齐 anchor 终端）；若实测体验不足，由用户看效果后再提升或转 A。
- 待设计期确认（非需求级未决）：历史快照变大后 150ms 推帧的体积 / 性能、上翻时的滚动位置保持策略 —— 属技术实现取舍，在拆分 / 实现时定。
- 用户已明确：B+ 先落地，"之后看效果"再决定是否需要 A —— A 若需要将作为独立后续需求。

## 八、拆分预览

预计为 **1 个聚焦子任务**（跨前端渲染 / CSS + 后端 capture 一处参数），含安全检查点 commit；是否补一页薄技术设计在拆分时定。核心实现取舍（如何承载历史快照又不污染 scrollback / 不拽回滚动位置）由 Claude 在子任务 spec 写清，Codex 实施。

## 十三、风险

- 历史快照变大 → 帧体积与 150ms 推帧性能（Codex 提示）。
- 上翻时若仍周期性重绘，可能把用户滚动位置拽回底部 —— 渲染方式须规避。
- FitAddon / 前端自适应与"tmux pane 尺寸为真相"冲突，须取舍兼容。
- B+ 落地后用户仍可能要完整历史（转 A）—— 已通过"先看效果"对齐预期。

## Claude 解读

这是需求详情页内嵌 slot 终端（xterm.js + 快照重绘模型）的滚动缺陷修复需求。两个症状同源：①重复行 —— 满高快照带尾随换行致 xterm baseY 持续增长（Codex 探针证实，去掉尾随 LF 即不堆）；②多滚动条 / 只显当前屏 —— 三层滚动容器（`.terminalColumn` / `.terminalHost` / `.xterm-viewport`）+ 无 FitAddon + 后端只传可见屏无历史。

经一轮 Codex 协商（job_5116286d7b21，已独立校验并修正我的初始假设）与用户拍板，方向定为 **B+**：修掉重复行与多滚动条，并让后端带最近 ~2000 行历史，使用户能上翻读到最近真实历史；不重构终端协议（A 暂缓）。
