## 需求描述

新建项目后需要“一键初始化知识库”，这个按钮点击后是一个确认弹窗，点击确定之后应该主动弹出 main终端的那个弹窗，这样方便用户更直观的看到当前正在被初始化和初始化的情况

## 原话（verbatim）

新建项目后需要“一键初始化知识库”，这个按钮点击后是一个确认弹窗，点击确定之后应该主动弹出 main终端的那个弹窗，这样方便用户更直观的看到当前正在被初始化和初始化的情况

## 二、背景与目标

- **背景**：项目接入 Console 后，`knowledge-missing` 态 banner 提供「一键初始化知识库」。当前确认提交后仅有 banner 文字「初始化中（jobId）」+ 3s 轮询（60s 超时）+ toast 反馈；`/ccb:su-init` 在 main claude pane 的真实执行过程对用户完全不可见，等待体验是黑盒。
- **目标**：确认提交成功后，Console 主动打开既有「main agent 组终端」弹窗（M 入口的 web Modal），让用户实时、直观地看到初始化正在进行及其过程输出。
- **成功标准（验收口径）**：确认初始化成功提交（拿到 jobId）后，无需任何额外点击，「main agent 组终端」弹窗自动出现且默认显示 main claude pane 实时输出；提交失败时不弹窗、仅既有错误 toast；终端 descriptor 未就绪时用户有可自行恢复的路径而非死端 fallback。

## 三、讨论与决策

**Codex 协商（1 轮，job_89079568659c，consult 只读）关键结论**：

| 议题 | 结论 |
| --- | --- |
| 「那个弹窗」指代 | 双方一致：`MainTerminalLauncher` 的 web Modal；排除实体终端（`spawnMainTerminal`）与 AI CLI 面板（`AiCliPanel`/`EmbeddedTerminal`，不消费 main ccbd claude pane） |
| 弹出时机 | 拿到 jobId 后弹；API 失败只 toast 不弹 |
| ready 态「重新初始化」 | 同 handler 同行为，不必升级用户 |
| 完成后行为 | 弹窗不自动关闭；「知识库已就绪」toast 保留为完成信号 |
| toast CTA 替代方案 | 不作为默认——违背原话「主动弹出」；仅当未来想降级打扰度时才需用户拍板 |
| descriptor race | server `init-knowledge-base`（ccbd submit）与 main web terminal descriptor（tmux/runtime 解析）是两条契约，存在 submit 成功但 descriptor 暂 404 的边界；自动打开使其成为主路径，静态 fallback 不够，需求层定义「必须有恢复路径」，机制留技术设计 |

## 四、功能 / 范围

1. 「一键初始化知识库」与 ready 态「重新初始化知识库」（同一确认 handler）在 `initProjectKnowledgeBase` 成功返回 jobId 后：关闭确认弹窗 → 自动打开「main agent 组终端」弹窗。
2. 自动打开的弹窗与 M 入口手动打开的是**同一弹窗实例**（复用 `MainTerminalLauncher` 的 Modal，open 状态提升；不在 banner 再造第二个终端弹窗）。
3. 默认显示 claude pane（`/ccb:su-init` 的投递目标），与现有默认一致。
4. API 失败：不弹窗，仅既有错误 toast，确认弹窗维持现状行为。
5. 自动打开后若 main 终端 descriptor 未就绪：用户必须有可自行恢复的路径（自动轻量重试或显式重试入口，机制由技术设计决定），不得停留在静态 fallback 文案。
6. banner 既有「初始化中（jobId）」轮询、60s 超时与成功/失败 toast 全部保留不变。

## 六、边界 / 不做项

- 不动 server API、不改 `/ccb:su-init` 投递链路、不引入新依赖。
- 不自动打开实体终端——「🖥 打开实体终端」（`spawnMainTerminal`）保持为独立显式按钮。
- 不在终端弹窗内叠加初始化进度条/状态 UI（终端实时输出本身即进度）。
- 知识库就绪后不自动关闭弹窗。
- 不改动 AI CLI 面板（`AiCliPanel`/`EmbeddedTerminal`）。

## 七、开放问题 / 假设

- 假设：banner 仅在 overview 当前项目上下文使用，`projectId === selectedProjectId` 当前恒等（已核验 OverviewPage 直传）；实现仍应在打开请求中携带 projectId，launcher 仅在其等于当前 `selectedProjectId` 时响应，防御未来 banner 复用到其它上下文。
- 无遗留用户决策。唯一已识别的潜在用户拍板项（把「自动弹出」降级为「toast + 查看终端按钮」）因遵守原话行为而不激活；若未来要改，须先问用户。

## 八、拆分预览

- 预计单个 web 前端子任务：Modal open 状态提升（专用 `mainTerminalOpen` / `requestOpenMainTerminal(projectId)` 类状态，不混入既有 `modalOpen/modalType` 互斥机制）+ banner 确认成功触发 + descriptor 未就绪恢复路径 + 组件测试更新。
- 实施归属：web UI/UX 由 Claude 亲自实施（用户既有决定），不派 Codex。

## 十一、界面 / 页面布局

- 无新页面、无新组件；复用既有 banner、确认 Modal 与「main agent 组终端」Modal。可见变化仅两处：确认成功后弹窗自动出现；终端未就绪时 fallback 区域出现恢复路径（形态由技术设计定）。

## 十二、交互 / 流程

```text
点击「一键初始化知识库 / 重新初始化知识库」
  → 确认 Modal（既有文案与风险提示保留）
    → 点「确认初始化」 → initProjectKnowledgeBase(projectId)
        ├─ 失败：error toast；不弹终端弹窗（现状不变）
        └─ 成功（jobId）：
            1. 关闭确认 Modal（避免双 focus trap）
            2. 自动打开「main agent 组终端」弹窗（默认 claude pane；不自动 focus xterm）
                ├─ descriptor 就绪 → 实时显示 su-init 执行输出
                └─ 未就绪 → fallback + 可自行恢复路径（机制由技术设计定）
            3. banner 进入「初始化中（jobId）」轮询（既有逻辑）
                ├─ 知识库就绪 → success toast「知识库已就绪」（弹窗不自动关）
                └─ 失败/超时 → 既有 error toast
```

## 十三、风险

| 风险 | 说明 | 应对方向 |
| --- | --- | --- |
| Modal 叠层 / 双 focus trap | 确认 Modal 与 main 终端 Modal 同 z-index（100） | 先关确认弹窗再开终端弹窗（时序约束） |
| 误输入进 main claude pane | main 终端默认可写；自动打开时用户手仍在键盘上 | 自动打开不自动 focus xterm |
| submit 成功但 descriptor 未就绪 | ccbd submit 与 tmux/runtime descriptor 是两条契约 | 见「四、功能/范围」第 5 条恢复路径验收口径 |
| projectId 恒等假设未来失效 | banner 若复用到项目卡片等上下文 | 打开请求携带 projectId，launcher 校验后响应 |

## Claude 解读

本需求是把「确认初始化 → 黑盒等待」改为「确认初始化 → 立即可视」的纯前端交互增强。具体：在 ProjectOnboardingBanner 的确认弹窗点「确认初始化」且 initProjectKnowledgeBase 成功返回 jobId 后，Console 自动打开既有的「main agent 组终端」web 弹窗（即 M 入口的 MainTerminalLauncher xl Modal，复用同一实例，默认 claude pane），用户无需自行寻找 M 入口即可实时看到 /ccb:su-init 在 main claude pane 的执行过程。ready 态「重新初始化知识库」走同一 handler，行为一致。提交失败不弹窗、仅既有 error toast；banner 既有轮询/超时/toast 机制全部保留。自动打开后若 main 终端 descriptor 未就绪，必须给用户可自行恢复的路径（机制留技术设计）。范围硬边界：不动 server API、不改 su-init 投递链路、不加依赖、不自动打开实体终端、不叠加进度 UI、完成后不自动关弹窗。
## 歧义点

- **A.「main终端的那个弹窗」指代** —— 已消解：指 MainTerminalLauncher 的「main agent 组终端」web Modal。依据：「弹窗」语义 + 「更直观看到正在被初始化」均指向 web 内嵌实时终端；排除「🖥 打开实体终端」（spawnMainTerminal，native 窗口，非弹窗）与 AiCliPanel/EmbeddedTerminal（AI CLI PTY，不消费 main ccbd claude pane）。Codex 协商一致；仅当用户明确说「实体终端窗口/系统终端」才改判。
- **B. 弹出时机与失败分支** —— 原话未定义；按唯一合理默认消解：成功拿到 jobId 后弹；API 失败只 toast 不弹（失败时命令未投递，弹出终端无意义）。
- **C. ready 态「重新初始化知识库」是否同行为** —— 原话只说「新建项目后」；按同一确认 handler 的代码事实消解：行为一致，同样自动弹出。
- **D. 完成后弹窗关闭行为** —— 原话未定义；默认不自动关闭（用户可能仍在查看输出），既有「知识库已就绪」toast 保留为完成信号。
- **E.（协商新增）自动打开遇 main 终端 descriptor 未就绪** —— 原话未提；由「更直观看到初始化情况」目标推出验收口径：必须有用户可自行恢复的路径，不得停留在静态 fallback；具体机制（自动轻量重试 vs 显式重试按钮）留技术设计。

以上 B/C/D/E 均为有唯一合理默认的交互细节，未达到需用户前置拍板的产品分叉级别（必问 7/8 类不命中理由见「三、讨论与决策」必问扫描）。
## 保真差异

1. 原话「新建项目后需要『一键初始化知识库』」——该按钮与确认弹窗已存在（knowledge-missing 态 banner），本需求不新增按钮；实际增量仅为「确认成功后主动弹出 main 终端弹窗」。
2. 原话未提 ready 态「重新初始化知识库」；因同一确认 handler，解读将其纳入同行为（Codex 同意，无需用户拍板）。
3. 「main终端的那个弹窗」为指代表述，解读消解为 web「main agent 组终端」Modal（MainTerminalLauncher），非实体终端窗口。
4. 原话未提失败分支与终端未就绪 race；解读按目标语义补充：失败不弹、未就绪须有恢复路径——这是对原话意图的保真延伸而非范围扩张。
5. 原话「方便用户更直观的看到当前正在被初始化和初始化的情况」是目的性表述，解读将其落为验收口径（自动弹出 + 默认 claude pane 实时输出 + 未就绪可恢复），未引入原话之外的新功能面。
