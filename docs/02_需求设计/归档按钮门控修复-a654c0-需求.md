## 需求描述

需求详情页的"归档"按钮当前可点击门控为 `status === "delivered"`，与 kernel 两段式 archive 设计方向相反：`delivered` 是归档 finalize 的结果而非点击前置，导致需求处于推进中（status=delivering、worktree runtime=merged）时按钮一直灰、不可点。

目标：
1. 需求处于推进中时，让需求详情页的"归档"按钮可点击。
2. 按钮触发的归档，与"批量推进 N 子任务"(su-batch) 跑完所有子任务后的收尾/finalize 走同一条 requirement-level archive/finalize 路径，保证两条完成路径行为一致。

触发场景：用户报告需求详情页归档按钮一直不可点；经排查（含 Codex 核对 rep_8a57bf5c8b9e）确认为前端门控漂移。

## 原话（verbatim）

需求处于推进中，就让"归档"按钮可以点击，然后确保按钮的归档和执行完"批量推进N子任务"所有子任务后的归档逻辑是一致的。

（触发上下文）现在需求详情页有一个"归档"按钮，但是一直状态是不可点击。

## 二、背景与目标

> 📌 目标对齐：把详情页那个一直点不动的"归档"按钮修好——需求做到"推进中"时能点，且点下去做的归档，跟你用"批量推进 N 子任务"跑完后系统该做的收尾，是同一件事的延续。

**两段式归档（关键背景，术语白话）**

- **子任务归档**：su-batch 把每个子任务跑完 implementation→review→archive，只写该子任务终态，不动需求大状态。
- **merge-only 收尾**：所有子任务归档后，su-batch 只把各实施空间合并（mergeRequirementWorktree），整体进入 `merged`（预览暂停态），需求仍是 `delivering`，**不** finalize。
- **需求级 finalize**：把需求真正收成 `delivered`（cleanup + requirement.finalize）。这正是详情页"归档"按钮该干的第二段。

```
当前（错）                              目标（对）
su-batch ─► merged / delivering        su-batch ─► merged / delivering
归档按钮 gate: status==delivered         归档按钮 gate: delivering 且 runtime∈{merged,archived}
   └─ 永远点不到（死循环）                  └─ 此时可点 ─► finalize ─► delivered
```

**为什么现在做**：按钮门控 `status==="delivered"` 与 kernel 方向反了——delivered 是 finalize 的产物，却被当成点击前置，形成死循环，导致 su-batch 跑完后按钮一直灰，用户只能走命令行 finalize。

## 三、讨论与决策

**Codex 协商摘要（rep_8a57bf5c8b9e，对本需求 framing 的质疑/确认）**

- confirm：前端门 `status==="delivered"` 与 kernel 两段式 archive 反向；delivered 只由 requirement.finalize 产生，正常按钮流死循环。
- 修正：planningRuntimeState 可被 anchor 置为 running，但 indexer/reindex 会写回 idle，且从不读 worktree json 的 aggregate_status，所以 merged/archived 确实没进 DB。
- 推荐门控 a+：`status==="delivering"` 且 worktree runtime∈{merged,archived}；archived 是 cleanup 成功但 finalize 失败的 recovery 入口。反对只用 delivering。
- 一致性 confirm：按钮派的 /ccb:su-archive 已带 subject=requirement，命中需求级 finalize 路径。
- guard 清单（finalize 前置必须挡）：非 delivering / worktree 文件缺失或 id 不匹配或 aggregate 非 merged|archived / 非 cancelled dev_task 未全终态 / 有未决 must_ask 或未物化 follow-up / cleanup 已 escalated / md hash 不匹配。

**4 锚点反思**

- 我同意：a+ 是正确口径——只在 finalize 真正安全（merged/archived）时让按钮亮，避免污染队列。
- 我不同意：不能只改前端门就算完；merged/archived 未投影进 DB，纯前端无门控对象，必须含后端投影地基。
- 我的盲点：起初以为是纯前端一行翻转；实际跨前后端（投影 + 门控 + dispatch guard）。
- 接下来：进入技术设计，确定 worktree aggregate_status → DB 投影的最小改动链与 dispatch guard 落点。

## 四、功能 / 范围

**范围内**

1. 后端：indexer + reindex 读 `docs/.ccb/worktrees/<id>.json` 的 aggregate_status，投影 merged/archived 到 Requirement 运行态字段（复用 planningRuntimeState 或新增字段，技术设计定）。
2. 后端：requirement 详情接口透出该运行态。
3. 前端：归档按钮门控由 `status==="delivered"` 改为 `status==="delivering"` 且 runtime∈{merged,archived}。
4. 后端：su-archive(requirement) dispatch 增加 canonical guard（读 worktree runtime + finalize 前置校验），防 stale UI 提前派发。
5. 配套测试（indexer/reindex 投影、dispatch guard、前端门控）。

**范围外（明确不做）**

- 不改 kernel 两段式 archive 流程本身（su-batch merge-only、archive.node.md finalize 路径保持）。
- 不为已 delivered 的历史需求回溯改状态。
- 不做 su-batch 自动 finalize（用户要的是按钮可手动 finalize，不是自动化）。

## 五、验收标准

1. 需求 status=delivering 且 worktree runtime=merged 时，详情页"归档"按钮可点击；非该态禁用并给准确 tooltip。
2. 点击归档派出 /ccb:su-archive(subject=requirement)，命中 archive.node.md 需求级 finalize 路径（cleanup + requirement.finalize → delivered）。
3. 后端 dispatch 在 finalize 前置不满足时（worktree 非 merged/archived、子任务未全终态等）拒绝并返回明确原因，不入脏队列。
4. merged/archived 运行态经 scan/reindex 正确投影进 Console，且不被局部 reindex 重置为 idle。
5. "经 su-batch 收尾(merged) → 点按钮归档 → delivered"与"命令行 /ccb:su-archive requirement_id"产生一致终态与证据。

## 十三、风险

- **投影滞后（WSL2 watcher 漏事件）**：merged 投影若没及时进 DB，按钮可能仍灰。缓解：scan/reindex 收敛 + 后端 dispatch guard 以 canonical worktree 为准（不只信投影）。
- **门控与 guard 双源漂移**：前端门与后端 guard 口径须同源（均以 worktree aggregate_status + finalize 前置为准），避免再次"门"与"真相"打架。
- **字段选择**：复用 planningRuntimeState 可能与现有 idle/running 语义混淆；技术设计需决定复用还是新增字段。

## Claude 解读

详情页"归档"按钮 = kernel 两段式 archive 的第二段：需求级手动归档（/ccb:su-archive，subject=requirement）做 cleanup + requirement.finalize，把需求收成 delivered。su-batch 跑完所有子任务后的尾部是第一段：逐子任务 archive + merge-only（mergeRequirementWorktree → runtime merged，需求保持 delivering），不 finalize。

用户要的是：①需求"推进中"（精确口径：status=delivering 且 worktree runtime∈{merged,archived}，即 finalize-ready）时按钮可点；②按钮就走这同一条 requirement-level finalize 路径，使"su-batch 收尾 → 点按钮"成为一条连贯的两段式完成流，而非两套互不相干的归档。

当前门控 status==="delivered" 把 finalize 的产物当成点击前置，形成死循环，导致 su-batch 跑完后按钮一直灰。修复 = a+ 门控 + 把 merged/archived 运行态投影进 Console + 后端 dispatch guard。
## 歧义点

1. 门控精度：用户说"推进中"，kernel 里对应 status=delivering，但 finalize 安全前置更窄——需 worktree runtime=merged（子任务已全 merge、就差 finalize）或 archived（cleanup 成功但 finalize 失败的 recovery 入口）。采用 a+（delivering 且 runtime∈{merged,archived}）。已由 Codex 核对推荐，反对只用 delivering（否则按钮可点但排队后失败、污染队列/事件）。
2. 后端投影是否属本需求范围：a+ 依赖 merged/archived 运行态，但它当前没投影进 DB（planningRuntimeState 恒被 indexer/reindex 写回 idle、不读 worktree json 的 aggregate_status）。结论：投影地基属本需求必含；具体拆几步是 task_breakdown 决策。
3. 防御层次：是否前端门 + 后端 dispatch guard 双层（后端以 canonical worktree runtime 为准，拦 stale UI 的提前派发）。Codex 建议双层。
4. "一致"的验收口径：界定为按钮派的 /ccb:su-archive(subject=requirement) 与 su-batch 收尾后续走同一 archive.node.md requirement-finalize 路径，并共享同一组 finalize 前置 guard（子任务全终态、有 merged worktree、expectedHash 匹配等）。
## 保真差异

- 用户原话"推进中"是宽泛说法；据 kernel 收窄为 status=delivering 且 runtime∈{merged,archived}（finalize-ready）。若用户本意是"只要 delivering 就可点"，与 a+ 有差异；因 Codex 已论证宽口径的 UX/数据风险，采纳 a+。
- 用户原话"一致"具体化为"同一 requirement-finalize 路径 + 同一组 guard"。用户未必意识到 su-batch 尾部是 merge-only（不 finalize），即两者本是同一两段式流程的前后两段、不是同一个动作；已在背景讲清。
- 用户未提"后端投影 / 后端 guard"；这是达成目标①②的必要技术前置，由我据代码现状补入范围，非扩大需求。
