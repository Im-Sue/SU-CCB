## 需求描述

需求详情页的"归档"按钮当前可点击门控为 `status === "delivered"`，与 kernel 两段式 archive 设计方向相反：`delivered` 是归档 finalize 的结果而非点击前置，导致需求处于推进中（status=delivering、worktree runtime=merged）时按钮一直灰、不可点。

目标：
1. 需求处于推进中时，让需求详情页的"归档"按钮可点击。
2. 按钮触发的归档，与"批量推进 N 子任务"(su-batch) 跑完所有子任务后的收尾/finalize 走同一条 requirement-level archive/finalize 路径，保证两条完成路径行为一致。

触发场景：用户报告需求详情页归档按钮一直不可点；经排查（含 Codex 核对 rep_8a57bf5c8b9e）确认为前端门控漂移。

## 原话（verbatim）

需求处于推进中，就让"归档"按钮可以点击，然后确保按钮的归档和执行完"批量推进N子任务"所有子任务后的归档逻辑是一致的。

（触发上下文）现在需求详情页有一个"归档"按钮，但是一直状态是不可点击。

## 二、背景与目标

> 📌 目标对齐：把详情页那个一直点不动的"归档"按钮修好——需求只要处于"推进中"(delivering) 就能点；点下去做的归档，跟你用"批量推进 N 子任务"跑完后系统该做的收尾，是同一件事的延续。至于"现在到底能不能真的归档成功"，交给 plugin 那道关卡判断，前端不预先卡死。

**两段式归档（关键背景，术语白话）**

- **子任务归档**：su-batch 把每个子任务跑完 implementation→review→archive，只写该子任务终态，不动需求大状态。
- **merge-only 收尾**：所有子任务归档后，su-batch 只把各实施空间合并（mergeRequirementWorktree），整体进入 `merged`（预览暂停态），需求仍是 `delivering`，**不** finalize。
- **需求级 finalize**：把需求真正收成 `delivered`（cleanup + requirement.finalize）。这正是详情页"归档"按钮该干的第二段。

```
当前（错）                          目标（对·简版）
gate: status == "delivered"         gate: status == "delivering"
   └─ 永远点不到（死循环）               └─ 推进中即可点 ─► 派 /ccb:su-archive
                                          └─ 能否真正 finalize 由 plugin 关卡判定
```

**为什么现在做**：按钮门控 `status==="delivered"` 与 kernel 方向反了——delivered 是 finalize 的产物，却被当成点击前置，形成死循环，导致 su-batch 跑完后按钮一直灰，用户只能走命令行 finalize。

## 三、讨论与决策

**门控口径（用户拍板·简版）**

最初技术设计倾向窄口径 a+（delivering 且 worktree 运行态∈{merged,archived}），并为此把 worktree 运行态投影进 DB + 加后端 dispatch 守卫。经与用户确认，改为**宽口径简版**：

- 前端：`status==="delivering"` 即可点击，**不**依赖 worktree 运行态投影。
- 后端：**不**加细粒度归档守卫，"推进中"即放行派发。
- 真正的 finalize 关卡**只在 plugin 一层**（唯一真相源，避免多处门控各说各话、相互漂移）。

**plugin 关卡已验证有硬代码兜底（非仅文档约定）**

- 合并/清理关卡：cleanup 代码要求 worktree 必须 merged/archived，否则拒绝（`lib/worktree` cleanupGateAllows）。
- finalize 关卡：`requirement.finalize` 策略要求全部非 cancelled 子任务终态 + md hash CAS 校验，否则拒绝并保持原状（`lib/capability-outcome` generated-policy + guard）。

故"推进中早点"最坏只是被 plugin 拒绝并提示，不会误标 delivered、不会破坏数据。

**4 锚点反思**

- 我同意：plugin 已是唯一权威关卡；前端只需解死循环，无须复制一套门控。
- 我不同意：不再坚持"投影 + 后端 guard"复杂方案——它与 plugin 关卡重复，反而增加双源漂移面。
- 我的盲点：起初把"按钮精确只在可 finalize 时才亮"当成硬需求；实际它只是 UX 增强，用户更要的是简单与单一真相源。
- 接下来：前端改门 + 点击确认加提示 + 测试；需求与技术设计文档同步为简版。

## 四、功能 / 范围

**范围内**

1. 前端：归档按钮门控由 `status==="delivered"` 改为 `status==="delivering"`；非 delivering 态禁用并给准确 tooltip。
2. 前端：复用已有"确认归档需求"弹窗，加一句提示（如"请确认子任务已全部完成并合并后再归档"），防手滑提前点。
3. 配套测试：前端门控三态（delivering 亮 / 非 delivering 灰 / delivered 灰）+ 确认弹窗提示文案。

**范围外（明确不做）**

- 不做 worktree 运行态（merged/archived）投影进 DB。
- 不加后端 dispatch 细粒度归档守卫（"推进中"即放行，finalize 关卡交给 plugin）。
- 不改 kernel 两段式 archive 流程本身（su-batch merge-only、archive.node.md finalize 路径保持）。
- 不为已 delivered 的历史需求回溯改状态；不做 su-batch 自动 finalize。

## 五、验收标准

1. 需求 status=delivering 时，详情页"归档"按钮可点击；planning / delivered / cancelled / deferred 等非 delivering 态禁用并给准确 tooltip。
2. 点击归档先弹"确认归档需求"，弹窗含"先确认子任务已完成合并"的提示；确认后派出 /ccb:su-archive(subject=requirement)。
3. 派出的归档命中 archive.node.md 需求级 finalize 路径（cleanup + requirement.finalize → delivered），与命令行 /ccb:su-archive requirement_id 产生一致终态与证据。
4. 子任务未全部完成/未合并时点击归档，由 plugin 关卡拒绝并给出明确原因，需求保持 delivering、不被误标 delivered。
5. "经 su-batch 收尾(merged) → 点按钮归档 → delivered"与"命令行 finalize"是同一条 requirement-level 路径。

## 十三、风险

- **推进中早点被拒的体验**：按钮在整个 delivering 期可点，子任务没跑完时点会被 plugin 拒绝。缓解：确认弹窗加提示 + 拒绝信息清晰可见。属可接受的"能点但被拒"体验，非数据风险。
- **单一关卡依赖**：finalize 安全完全依赖 plugin 关卡。缓解：已验证 cleanup / finalize 两道关卡均有硬代码 + 测试兜底；本需求不削弱它们。

## Claude 解读

详情页"归档"按钮 = kernel 两段式 archive 的第二段（需求级手动 finalize）。**简版决策**：前端门只看 `status==="delivering"`（推进中即可点），不投影 worktree 运行态、不加后端守卫；能否真正 finalize 由 plugin 唯一关卡判定——cleanup 要求 worktree merged/archived、requirement.finalize 要求子任务全终态 + md hash CAS，二者均有硬代码 + 测试兜底。当前门控 `status==="delivered"` 把 finalize 的产物当成点击前置，形成死循环；修复 = 把门改成 delivering + 确认弹窗加提示。子任务未完成时点击，由 plugin 拒绝并提示，不会误标 delivered。按钮与命令行 finalize 同路径，使"su-batch 收尾 → 点按钮"成为连贯的两段式完成流。
## 歧义点

1. 门控口径（已闭合）：用户原话"推进中"，最初技术设计倾向窄口径 a+（delivering 且 worktree∈{merged,archived}）。经用户拍板，确认采**宽口径**：`status==="delivering"` 即可点，finalize 安全交给 plugin 关卡。
2. 早点体验：delivering 期按钮恒可点，子任务未完成时点击会被 plugin 拒绝。已决定以"确认弹窗提示 + 清晰拒绝信息"承接，不做前端硬门预防。
3. 一致性口径：按钮派的 /ccb:su-archive(subject=requirement) 与 su-batch 收尾后续走同一 archive.node.md requirement-finalize 路径，共享同一组 plugin finalize 前置 guard（子任务全终态、merged worktree、expectedHash 匹配）。
## 保真差异

- 用户原话"推进中"按字面采纳为 `status==="delivering"`（宽口径）。早前技术设计曾据 kernel 收窄为 a+（merged/archived）并加投影 + 后端 guard；经用户明确，回到宽口径简版，去掉投影与后端守卫。这是用户对范围的主动收敛，非扩大需求。
- 用户原话"一致"具体化为"同一 requirement-finalize 路径 + 同一组 plugin guard"。用户未必意识到 su-batch 尾部是 merge-only（不 finalize）——它与按钮归档是同一两段式流程的前后两段。
- 后端投影 / 后端 guard 不再属本需求范围（简版移除）；finalize 安全由 plugin 唯一关卡保证。
