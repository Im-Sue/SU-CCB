## 需求描述

所以是否可以强制如果没有做完这些，左边“项目” 的导航点击的时候禁止点击或者点击之后提示需要去初始化，但是不要打开对应页面，就是强制初始化

## 原话（verbatim）

开发人员第一次使用SU-Oriel，不知道要 初始化ccb、也不知道要去初始知识库。
所以是否可以强制如果没有做完这些，左边“项目” 的导航点击的时候禁止点击或者点击之后提示需要去初始化，但是不要打开对应页面，就是强制初始化

## 二、背景与目标

**要解决什么(白话)**:开发者第一次打开 SU-Oriel,看到左侧"项目"组(需求管理、文档中心、Slot 拓扑、运行记录)就点进去;但此时项目往往还没"初始化 CCB"(术语:在项目目录跑 `ccb`,生成 `.ccb/ccb.config` 并拉起守护进程,下称 **CCB 运行时就绪**),也没"初始化知识库"(术语:提交 `/ccb:su-init` 生成 docs 索引,下称 **知识库就绪**)。结果进了页面是空的/不可用,用户不知道"应该先去初始化"。

**给谁**:首次接入某个项目的开发者。

**为什么现在做**:接入引导组件(`ProjectOnboardingBanner`)其实已存在,但只是概览页顶部一条紧凑横幅;左侧导航与概览卡片都能直接进项目业务页,没有任何阻挡 —— 首次用户极易跳过初始化、撞进空页面,也"不知道要初始化"。

**目标**:① 未就绪前把"项目"组业务页入口"锁住",点击主动弹引导;② 把概览页在未就绪时整页变成**分步初始化引导**(而不是堆一条横幅 + 一堆空指标卡),直接消除"不知道要初始化"的根因;就绪后概览回到正常数据盘。

**模拟示例(首次使用)**:
1. 新开发者选中/新建项目 X(尚未跑过 `ccb`)。进概览 → 整页是"两步初始化引导":第①步初始化 CCB 运行时、第②步初始化知识库。
2. 点左侧"需求管理" → 不打开页面,弹引导:"项目 X 还没初始化,请先初始化",CTA「去概览初始化」把人带到上面的引导页。
3. 用户照引导做完两步,CCB 运行时就绪 + 知识库就绪后,"项目"组解锁、概览回到正常数据盘。

## 三、讨论与决策

经与 slot3_codex 协商 + 用户拍板,确定以下取舍:

| 议题 | 决定 | 理由 |
|---|---|---|
| 拦截广度 | **应用内点击拦截**(Sidebar 项目组 + 引导态联动);命令面板、直达/深链 URL **不拦** | 用户取最小可用范围;命令面板/直达 URL 明确豁免 |
| 概览未初始化态 | **整页分步初始化引导**(隐藏指标卡);就绪后回到现概览(banner + 卡片) | 比"横幅 + 空卡"更直观,直击"不知道要初始化"的根因;modal CTA 落点也更有用 |
| 交互呈现 | **锁定态 + 点击弹 modal 引导 + 去初始化 CTA**;不打开目标页 | 比纯禁用置灰可发现性好、主动引导 |
| 解锁条件 | CCB 运行时就绪 **且** 知识库就绪(两步全绿) | 用户"这些"指两步;业务页多依赖知识库,半解锁规则难解释 |
| 门控边界 | 仅"项目"组业务页 + 概览未初始化态;概览(就绪)/我的工作/项目设置/AI CLI 可达 | 概览是初始化落脚页必须可达;其余非"项目"组不在诉求内 |
| 被拦行为 | 停在当前页弹引导,不导航、不强制重定向 | 忠于"不要打开对应页面" |

**否决**:① 路由级全入口 guard(覆盖命令面板/直达 URL)——用户判断不必要;② 纯禁用置灰——可发现性差;③ 业务页统一占位页——URL 实际进入了,违背"不要打开对应页面";④ **(已废)概览顶部横幅 + 指标卡点击拦截**——被"概览整页引导"取代:未就绪时指标卡根本不渲染,无需再拦卡片。

## 四、功能 / 范围

1. **共享门控状态**:把项目"接入状态"(CCB 运行时就绪 / 知识库就绪)从 `ProjectOnboardingBanner` 的组件私有缓存,提升为可复用的共享读取(store),供导航、概览、引导页共用同一份数据。
2. **Sidebar 锁定态**:未解锁时,"项目"组导航项显示锁定态(锁图标/弱化样式),点击不跳转、改弹引导 modal。
3. **概览整页引导**:`!ready` 时概览**隐藏指标卡**,整页呈现分步初始化引导(步骤① CCB 运行时、步骤② 知识库),复用既有 banner 的检测/动作(复制命令 / 一键 init / 打开终端 / 轮询);`ready` 后回到现概览(banner 就绪态 + 指标卡)。
4. **引导 modal**:说明"需先初始化",提供「去概览初始化」CTA → 跳概览引导页 + 关闭 modal(复用概览引导,不另造初始化 UI)。
5. **解锁联动**:两步全绿后,Sidebar 锁定解除、概览回到数据盘。

## 五、业务规则

1. 解锁条件:`ccbRuntimeReady === true && knowledgeBaseReady === true`;任一为 false 即"未初始化",门控生效、概览走引导态。
2. 状态来源:`GET /api/projects/{id}/onboarding-status`;同一项目的导航、概览、引导页必须读同一份状态。
3. loading 策略:接入状态未加载完成时,"项目"组按"未解锁"处理并提示"正在检测接入状态",不放行;概览未加载时不抢先渲染引导/数据盘。
4. 失败策略:接入状态接口失败时,提示"无法确认接入状态",按未解锁处理(**不 fail-open**)。
5. 切项目:切换项目后按新项目的接入状态重新判定门控与概览呈现。
6. 概览指标卡仅在 `ready` 时渲染;`!ready` 时不渲染(由引导页占位),因此无需对概览卡片单独做点击拦截。
7. 仅前端 UX 引导:门控不作为权限/安全控制,后端 API 不依赖本门控做鉴权。

## 六、边界 / 不做项

1. **不拦**命令面板、直达/深链 URL(用户明确豁免);承认这是已知绕过口。
2. 不做无差别强制重定向(被拦时停在当前页)。
3. 不改后端:仅消费既有 onboarding-status 接口,不新增/改接口、不改 schema、不加鉴权。
4. 不做业务页占位页方案;不门控非"项目"组区域(概览就绪态/我的工作/项目设置/AI CLI)。
5. 不改初始化本身流程(`ccb` 运行时、`/ccb:su-init` 知识库初始化逻辑不动),只改"未完成时"的引导呈现与导航门控。
6. 概览引导页**复用** banner 既有检测/动作,不新造一套初始化逻辑;不做花哨教学动画。

## 七、开放问题 / 假设

无未决问题(拦截广度/交互/边界/概览呈现已由用户拍板,见歧义点闭环记录)。

**假设**:
- "强制初始化"是 UX 体验约束,非安全/合规边界(用户与 Codex 一致)。
- 概览页始终可达,作为初始化引导落脚页(未就绪=引导态,就绪=数据盘)。
- 命令面板/直达 URL 绕过率低且用户可接受。

**移交技术设计的技术项**:
- 共享状态落点(project-store slice)与缓存/失效;onboarding 动作+轮询的复用单元(hook)落点。
- 锁定态视觉与无障碍;`ProjectSetupGuide` 两步呈现与现 banner 逻辑的拆分方式。

## 八、拆分预览

- 任务 B:Sidebar"项目"组锁定态 + 点击拦截弹引导 modal + onboarding-required modal。
- 任务 C:概览未初始化态整页引导(`ProjectSetupGuide` + banner 逻辑复用/拆分;`ready` 切回数据盘)。

## 十、接口(草案)

## 十一、界面 / 页面布局

未初始化时:左侧导航锁定 + 概览整页引导(ASCII 草图):

```
左侧栏                     概览页(未就绪 = 整页引导)
┌──────────────┐          ┌──────────────────────────────────┐
│ 工作          │          │   欢迎接入项目 X · 完成两步即可开始  │
│  ◎ 概览       │ ← 可达   │                                    │
│  👤 我的工作   │ ← 可达   │   ① 初始化 CCB 运行时   [复制命令]   │
│ 项目          │          │      在项目目录执行 ccb …            │
│  🔒 需求管理   │ ← 锁定   │                                    │
│  🔒 文档中心   │ ← 锁定   │   ② 初始化知识库   [一键初始化][终端]│
│  🔒 Slot拓扑   │ ← 锁定   │      生成 docs 索引(/ccb:su-init)  │
│  🔒 运行记录   │ ← 锁定   │                                    │
│ 工具          │          │   (隐藏指标卡;就绪后自动切回数据盘) │
│  ⚙ 项目设置   │ ← 可达   └──────────────────────────────────┘
│  ▶ AI CLI    │ ← 可达
└──────────────┘   点锁定项 → 弹 modal:"需先初始化"[去概览初始化]
```

就绪后:概览回到现状(顶部"项目接入已就绪" banner + 需求/任务文档/系统健康指标卡)。

## 十二、交互 / 流程

```
进入概览 / 点"项目"组入口
        │
        ▼
读接入状态(共享 store ← onboarding-status)
        │
   ┌────┴───────────┬──────────────────┐
   │未加载/失败       │ready=false        │ready=true
   ▼                ▼                   ▼
 锁定+提示          概览=整页两步引导     概览=数据盘
 (不放行)          Sidebar 项目组锁定    Sidebar 项目组正常
                    │
         点锁定项/被拦 → modal[去概览初始化]→跳概览引导
                    │
         照引导做完两步 → 状态写回 store → ready=true → 自动解锁+切数据盘
```

## 十三、风险

| 风险 | 处理 |
|---|---|
| 状态不一致/重复请求(banner 私有缓存 vs 导航各自拉取) | 接入状态集中为共享 store 源,单一来源 |
| fail-open(接口失败时误放行) | 失败/未加载一律按未解锁处理 |
| 绕过口(命令面板/直达 URL) | 用户已接受为已知边界;非安全控制 |
| 引导页 init 轮询与"就绪后切走"的卸载时序 | 轮询结果写 store(非局部 state)、轮询owner 落在稳定层;卸载只清 interval,不丢更新 |
| 锁定态可发现性/无障碍 | 锁图标 + 点击解释 + CTA;技术设计细化 aria 语义 |

## Claude 解读

**核心意图**:未完成"初始化 CCB(运行时就绪)+ 初始化知识库(知识库就绪)"前,① 在前端把"项目"组业务页入口锁住、点击弹引导;② 把概览页在未就绪时整页变成分步初始化引导,直击"不知道要初始化"的根因。就绪后概览回到正常数据盘。这是 UX 引导门控,不是权限控制。

**现状核查(读码)**:`su-oriel/web`(React19 + React Router7 + Zustand);左侧 `Sidebar.tsx` 三组导航;接入状态由 `GET /api/projects/{id}/onboarding-status` 暴露(`ccbRuntimeReady` / `knowledgeBaseReady`);引导组件 `ProjectOnboardingBanner` 已实现三态 + 动作(复制命令/一键 init/终端)+ init-job 轮询,但只是概览顶部一条横幅;导航层无 guard;概览卡片用 `navigate()` 直跳业务页。

**协商(slot3_codex)**:需求轮 `job_f4f1a3deafed` 质疑"只拦 Sidebar 会被命令面板/概览卡片/直达 URL 绕过";设计轮 `job_e6fbf23418f3` 就"单一 store 源 + 派生 hook + modal + banner 重构"达成共识。

**用户拍板收敛(两轮)**:

## 歧义点

以下拍板项均已在终端向用户问到答案并闭环:

1. **拦截广度** — 答:**应用内点击拦截**(Sidebar 项目组 + 引导态联动);命令面板、直达/深链 URL **不拦**(明确豁免、已知绕过口)。
2. **交互呈现** — 答:**锁定态 + 点击弹 modal 引导 + 去初始化 CTA**,不打开目标页。
3. **门控边界** — 答:**仅门控"项目"组业务页**(+ 概览未初始化态);概览(就绪)、我的工作、项目设置、AI CLI 保持可达。
4. **解锁粒度** — 判定:两步全绿(`ccbRuntimeReady && knowledgeBaseReady`)才解锁。依据用户"这些"=两步,半解锁规则难解释(Codex 一致)。
5. **是否强制重定向** — 判定:否。被拦时停在当前页弹引导,忠于"不要打开对应页面"。
6. **概览未初始化态呈现** — 答(本轮新增):**未就绪时概览整页变成分步初始化引导(隐藏指标卡),就绪后回到现概览(banner + 指标卡)**。理由:用户认为更直观,且直击"不知道要初始化"的根因;连带使原"概览卡片点击拦截"作废(卡片不渲染、无需拦),modal CTA 落点更有用。

**移交技术设计的技术项(非用户拍板)**:共享接入状态落点(project-store slice)与缓存/失效;`useProjectOnboardingActions` 复用单元 owner 落点(取稳定的 OverviewPage 层 + 写回 store,规避卸载时序竞态);`ProjectSetupGuide` 与现 banner 逻辑的拆分;锁定态视觉与无障碍语义。

**未命中必问类(说明理由)**:隐私/数据(不新增数据,仅读既有状态)、成本(纯前端、无外部调用)、合规(无)、不可逆动作(无)、安全/权限(明确为 UX 门控,非鉴权)、依赖/schema/API/migration(纯前端重构,无新增/变更)——均不命中;Codex 亦确认。
## 保真差异

1. **"左边'项目'的导航点击" → 含义聚焦在 Sidebar 项目组**:命令面板/直达 URL 按用户原意不纳入(已知绕过口)。
2. **"禁止点击或者点击之后提示" → 取"提示(锁定 + modal + CTA)"**:用户给两个并列候选,拍板取"锁定态 + 点击弹引导 + CTA",非"纯禁止点击置灰"。
3. **"就是强制初始化"的强度被有意收窄**:字面指向"无法绕过",但用户豁免命令面板/直达 URL,故实际为"覆盖主要误点入口的强引导",存在已知绕过口(用户接受)。已在「六、边界」「十三、风险」显式标注。
4. **"不要打开对应页面"严格遵守**:被拦时不导航、不重定向。
5. **超出原话的范围扩展(用户主动追加并确认)**:用户原话只提"导航锁定/提示";本轮用户主动提出并拍板"未初始化时概览整页做初始化教程"。这是对"初始化落脚页"的增强(超出字面"导航门控"),已在用户确认下纳入范围;连带使原"概览卡片点击拦截"方案作废(改为未就绪不渲染卡片)。属用户驱动的范围正向扩展,非 AI 擅自放大。
