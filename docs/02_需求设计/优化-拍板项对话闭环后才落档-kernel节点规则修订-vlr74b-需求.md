# 优化：拍板项对话闭环后才落档——kernel 节点规则修订 需求设计

> 一句话：命中用户拍板项时，必须在当前终端问到答案才允许落正式档，消灭「拍板问题躺在文档里被疏忽」 ｜ 最后更新: 2026-06-07

---

## 需求描述

需求分析 / 技术设计节点产出的文档中，仍可能包含「待用户拍板」的未决问题（实例：23ee06 的修复档位三选一、c5c98f 开放问题表 4 项「待谁定：用户」）。用户很难在文档中发现这些待决项，导致疏忽的决策被固化进设计与代码。本需求修订 plugin kernel 节点规则与模板：把「已问用户」收紧为「已获用户答复」，命中拍板项攒齐一次性问、在当前终端等到答案才落正式档（Console 派发的 anchor 会话同样挂着等，不区分执行上下文）；模板删除未决项的「合法床位」；dispatch 闸门从「挡 TBD 字样」扩为「挡任何待用户拍板项」。纯 plugin 文档修订，零代码。

> 立项

## 二、背景与目标

**目标对齐**：现在 AI 做需求分析/技术设计时，发现「需要你拍板的问题」有时会直接写进文档（比如「待用户拍板：修复档位选哪个」），而不是当场问你。你不会逐篇细读文档，这些问题就被漏掉，错误方向被固化进设计和代码。本需求把规则收紧为：**命中拍板项 → 攒齐一次性问 → 在当前终端等到你的答案 → 才允许落正式档**。Console 派发的 anchor 会话同样挂着等（anchor 是交互式 Claude TUI，不是后台进程），不区分执行上下文。

**模拟示例**（拿真实案例 23ee06 端到端走一遍）：

| | 旧流程（实际发生过） | 新规则下 |
|---|---|---|
| 分析中发现「修复档位三选一」「多 tab 并发是否必须支持」两个拍板项 | 写进文档「七、开放问题」标【待用户拍板】，节点宣告完成、文档落档 | 分析主体先不落档；终端里一次性抛出 2 个拍板问题，会话挂起等答案 |
| 用户侧 | 必须自己打开文档才能发现有问题等着拍；漏看 = 疏忽决策 | 流程阻塞，需求不动了；用户迟早看终端，问题就在眼前 |
| 拿到答案后 | —— | 答案写入「三、讨论与决策」，文档落档时已是闭环形态 |

- **背景**：实际产物抽查证实三层漏洞——规则措辞（「已问用户」≠已获答复）、模板床位（「待谁定：用户」列）、执行上下文误解（把 Console 派发的交互式 anchor 当成无对话通道的后台任务）。
- **目标**：
  1. 任何节点产出的正式档中不再出现「待用户拍板/待确认」的未决项。
  2. 拍板动作发生在终端对话里，阻塞式等待，不靠用户翻文档发现。
  3. 模板与 SKILL 文案不再提供未决项的「合法床位」，防旧行为复活。

## 三、讨论与决策

> 本需求的全部歧义在立项对话中已拍板闭环；以下为决策记录（用户原话见「原话（verbatim）」section）。

### 决策 1：统一规则，不区分执行上下文
- 讨论：Claude 最初提出按链路拆两套方案（交互链路先问后落档；异步链路落档+pending_decisions 状态机+Console UI 高亮+通知联动）。用户质疑：「异步链本质上应该还是触发su-flow？所以我理解是同一个问题？」——核验证实 Console 按钮派发的就是 su-flow（`RequirementDetailPage.tsx:1117`），且 anchor 是 tmux pane 里的交互式 Claude TUI（`claude-pane-readiness.ts`），具备问答能力。
- **最终决策 + 理由**：一条统一规则「命中拍板项 → 攒齐一次性问 → 终端等到答案 → 才落正式档」。不做状态机、不做 UI 高亮、不做通知联动——因为阻塞式等待让未决项根本进不了正式档，「落档后被忽视」的防护对象不存在。

### 决策 2：终端挂着等，不做超时防护
- 讨论：Claude 提出需实测 ccbd job 执行时长上限。用户否决：「只是一个问答而已……能持久挂着就一直挂着呗，反正我在工作期间肯定能看到，就算我看不到最终也肯定会在某个时间去看。」
- **最终决策 + 理由**：不做实测、不做超时防护。job 状态在 CCB 架构里只是投影件（ADR-0030：文件是真相源，Console 只投影）；落档走 lib + file watcher，不依赖 job 状态。最坏情况是 Console 显示瑕疵，遇到再说。

### 决策 3：等待期产物不做草稿机制，丢了就重做
- 讨论：Codex 协商（job_d2ca685dfc77）指出「先落 drafting 态分析稿」与现有 lib 不兼容（`applyRequirementAnalysis` 三字段非空校验 + 写入后自动 promote planning，`lib/requirement-analysis/index.mjs:315-323,453-460`），零代码前提下只能「不调用 apply、等待期只保留对话上下文」。
- **最终决策 + 理由**：用户拍板「丢了就重做」。正常情况会话挂着等、上下文都在；只有崩溃/compact 偶发场景丢分析，重做一次协商+分析的代价可接受，比新增一套非 canonical scratch 机制更符合最简取向。

### 决策 4：Console seed 占位文案不动，保住零代码
- 讨论：Codex 发现第三套模板源——server 内置 renderer 写「尚未确认的问题」（`su-oriel/server/src/indexer/project-indexer.ts:1076-1077`）。改它破「零代码」边界。
- **最终决策 + 理由**：用户拍板「不动它」。占位文案会被 AI 正式分析落档整段覆盖，不影响任何行为；下次动 su-oriel 时顺手改。

### 决策 5（Claude 自治界定，不命中必问）：「正式档」的边界
- 「正式档」= 分析结论主体（`applyRequirementAnalysis` 写入的解读/主体章节/歧义点）及技术设计文档正文。**不含**立项动作（原话存档 + `promoteRequirementToPlanning`）——原话是用户自己的话，无 AI 代答内容；`planning` 状态只表示「在规划中」，不表示「分析已定稿」。因此 su-flow 的 entry promotion 契约不动，保持零代码成立。

## 六、边界 / 不做项

- 不做 pending_decisions 状态机、Console UI 高亮、通知联动（决策 1；通知由已立项的 52441f 另行覆盖）
- 不做 ccbd job 超时实测与防护（决策 2）
- 不做等待期草稿/scratch 机制（决策 3）
- 不改 su-oriel 任何代码，含 `project-indexer.ts` 占位文案（决策 4）
- 不改 `applyRequirementAnalysis` / `promoteRequirementToPlanning` lib 与 schema（`ambiguities` 字段保持必填非空，写闭环记录即满足；Codex 已核验 server 解析侧只抽 section 文本，无投影冲突）
- 不回溯迁移存量需求文档（23ee06、c5c98f 等留档形态不动；新规则只约束此后产出）
- 交付节奏：单批文档修订，不拆多任务

## 七、开放问题 / 假设

无遗留待用户拍板项（6 项决策全部闭环，见「三、讨论与决策」）。

**假设**（移交技术设计核验）：
- 9 处文件的精确 diff 在技术设计阶段产出；若发现 manifest 间交叉引用导致额外联动文件，属同性质工作自然扩展，不构成范围变更。

## Claude 解读

用户痛点：AI 解析产物中混着待用户拍板的问题，用户难以在文档中发现，导致疏忽决策。诊断为三层漏洞：①kernel 节点措辞「已问用户」不要求已获答复（requirement_analysis.node.md 完成条件3/硬约束5）；②模板「待谁定：用户」列与「仍需用户确认」注释提供未决项合法床位；③把 Console 派发的交互式 anchor（tmux 内 Claude TUI）误当无对话通道的后台任务。修订方案：一条统一规则「命中拍板项攒齐一次性问、终端阻塞等待、答案到手才落正式档」，覆盖 kernel 节点 manifest、两份需求模板、3 个 SKILL 文案与下游闸门措辞，纯文档零代码。实证锚点：23ee06/c5c98f（未决项落档）vs fqn0ds/9201ab（闭环后落档）。

## 歧义点

立项对话识别 6 项歧义/拍板项，全部闭环，无遗留待用户项：
1. 方案形态（按链路拆两套 vs 统一规则）——用户拍板：统一规则，命中拍板项→攒齐一次问→终端等到答案→才落正式档；Console 派发不区分（anchor 为交互式 Claude TUI，经 claude-pane-readiness.ts 核验）。
2. 等待超时防护——用户拍板：不做实测不做防护，挂多久都行；job 状态仅投影件，落档链路（lib+watcher）不依赖。
3. 等待期产物丢失（Codex 协商 job_d2ca685dfc77 提出 drafting 草稿与 lib 不兼容）——用户拍板：不做 scratch，丢了重做。
4. 第三套模板源 project-indexer.ts:1076 占位文案（Codex 发现）——用户拍板：不动，保住零代码；正式分析落档整段覆盖。
5. 范围完整性——Codex 指出仅改 3 处会被 SKILL 输出文案复活旧行为，采纳扩为四组 9 处；同方向完整化，零代码性质不变。
6. 「正式档」边界（Codex open question：是否含 promote planning）——Claude 自治界定：仅指分析结论主体，立项动作（原话存档+promote）不含，entry promotion 契约不动；理由：原话无 AI 代答内容，planning 仅表示在规划中。

## 保真差异

用户原话为对话式提问与质疑，整理为规则修订需求时做了归纳：①「对话先和我聊清楚再去落档」具体化为「攒齐一次性问+终端阻塞等待+正式档边界界定」；②「就让它放着呗」归纳为「不做超时防护、接受任意时长等待」；③四组 9 处文件修订范围是 Claude 扫描+Codex 协商的分析推导，非用户原话内容；④「丢了就重做」「不动 renderer」来自立项后 AskUserQuestion 的明确选择。方向与边界与原话一致，无扩大解释。
