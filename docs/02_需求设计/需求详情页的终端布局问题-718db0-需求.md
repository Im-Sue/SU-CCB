## 需求描述

我把ccb的slot的agent布局变成左右布局后UI界面上的终端看不到最底部，缺了整个终端外部组件的滚动条？还是什么问题？

## 原话（verbatim）

我把ccb的slot的agent布局变成左右布局后UI界面上的终端看不到最底部，缺了整个终端外部组件的滚动条？还是什么问题？

## 二、背景与目标

- 背景：su-oriel 需求详情页右栏「Slot 运行位置」用共享组件 `SlotTerminalSurface` 只读镜像 slot 终端（claude / codex pane）；该 surface 同时被 main agent 组终端 `MainTerminalPanel` 复用。
- 目标：当被镜像的 tmux pane 高于 web 固定容器（典型触发：slot 两 agent 改左右 split 使 pane 变高）时，用户仍能看到 / 滚动到 pane 底部（最新输出与 prompt），不再被裁剪且滚动条可达。

## 三、讨论与决策

**Codex 协商（slot3_codex · mode:consult · job_53c7b8643bbc）摘要**：确认根因大方向与「左右布局＝运行时 tmux 布局」判断；纠正「无 scrollback」一环（initial / live 分支）；以代码证据排除 B（放不下）、C（WS `READ_ONLY` 拒 resize）；新增 D（裁底）；建议选 A 且在共享 surface 层修；将 A 与既有「单一滚动层」目标的取舍标记为 human-decision。

## 四、功能 / 范围

- 在共享 `SlotTerminalSurface`（`.terminalHost` / `.xterm` / `.xterm-viewport` 的渲染与滚动策略）解决「pane 行数 > host 可视高度」时的纵向可达性。
- 提供外层纵向滚动以看全 / 滚到 pane 底部；新增「在底部才自动贴底」跟随。
- 连带覆盖 `MainTerminalPanel`（同一 surface）；需求详情页 `.terminalColumn` 仅做必要父容器配合。

## 六、边界 / 不做项

- 不改后端：不让前端反向 resize tmux pane（WS `READ_ONLY`，且与 ccb 原生 sidebar 共享会冲突）。
- 不动 ai-cli `EmbeddedTerminal`（FitAddon 自适应，不在本 bug）。
- 不引入新依赖、不改 WS 协议 / schema。
- 不改横向滚动（现有 `overflow-x:auto` 为既定设计，保留）。

## 七、开放问题 / 假设

- 假设：用户要「看全当前 tmux pane（含底部）」，而非「持续滚动完整实时历史」（Codex 同此假设）。
- 唯一待用户确认点：A（外滚看全，推荐）vs D（裁底、单一干净视口但丢顶部）。默认按 A 推进，用户可在技术设计前改选 D。

## 十一、界面 / 表现

- 终端 host 在当前屏 rows 高于可视高度时出现纵向滚动条；滚动条与现有深色终端皮肤一致；避免与 xterm 内置 scrollback 滚动条形成双条。

## 十二、交互 / 流程

- 用户可纵向滚动查看 pane 任意位置；滚动停在底部时新帧自动贴底跟随；滚动离开底部时不被新帧强制拉回。
- 横向：保留既有 `overflow-x:auto`（pane 列宽于右栏时横向滚动）。

## 十三、风险

- 双滚动条 / 滚动层冲突：需以外层单滚动条 + 抑制 xterm 冗余滚动条收敛。
- `height:auto` 处理不当可能复发「xterm 内容反向撑大父容器」；需配合 `min-height:0` / `flex` 约束。
- auto-follow 回归：须保证「仅在底部跟随」，避免每帧弹走用户。
- 回归面：同一 surface 被 `MainTerminalPanel` 复用，改动需双场景验证。

## Claude 解读

**一句话**：su-oriel web 终端镜像（共享组件 `SlotTerminalSurface`）的「pane 行数 ≠ 固定容器像素高度」错配 bug。slot 两 agent 由上下 split 改左右 split 后，被镜像的 tmux pane 变高、行数增多，而 web 右栏容器高度固定，超出部分被裁掉且无可达的外层纵向滚动条，于是「看不到最底部」。

**触发链（已核验代码）**：
- `SlotTerminalSurface` 只读镜像 tmux pane；后端用 `display-message -p "#{pane_width} #{pane_height}"` 取 pane 实际宽高（`su-oriel/server/src/modules/slot-terminal/slot-terminal.frame-stream.ts` `getPaneDimensions`）。
- 前端 `SlotTerminalFrameRenderer.flush()` 用 `terminal.resize(frame.cols, frame.rows)` 把 xterm 行数强制对齐 pane 行数。
- 左右 split → 每 pane 占满整列高 → `frame.rows` 约翻倍。

**裁剪点（已核验代码）**：
- `SlotTerminalSurface.module.css`：`.terminalHost{overflow-y:hidden}` + `.xterm{height:100%}`。当前屏高 = rows×cellHeight 超过 host 时，超出部分被 `overflow-y:hidden` 裁掉。
- xterm 内置 `.xterm-viewport` 是绝对定位的 scrollback 滚动层，只滚「历史缓冲」，不承担「当前屏溢出 host」的外层滚动 → 没有可达底部的滚动条。
- 外层 `RequirementDetailPage.module.css` `.terminalColumn{height:calc(100vh-64px);overflow:hidden}` 固定高度并隐藏溢出，进一步锁死。

**决策方向 A（web-UI 职责内拍板，可回退 D）**：在共享 `SlotTerminalSurface` 层引入「pane 行数 > host 可视高度时的外层纵向滚动」，让用户滚动看全/滚到 pane 底部；以外层 host 滚动作为唯一可见纵向滚动条 + 「在底部才自动贴底」跟随，避免双滚动条与 `height:auto` 反向撑大。需求详情页仅父容器配合。

**范围**：修共享 surface（连带修好 MainTerminalPanel / main launcher）；ai-cli 的 `EmbeddedTerminal` 走 FitAddon 按像素自适应行数，不在本 bug 范围。
## 歧义点

1. [高·已定] 「左右布局」指什么 → 判定为**运行时 tmux 把 slot 两 pane 改左右 split**（`.ccb/ccb.config [windows]` 只声明 window 成员、不含 split 方向；Codex 协商同此判断）。非 su-oriel 内的布局开关。

2. [高·已决策 A，可回退 D] 超高 pane 的期望显示行为：
   - **A 外层纵向滚动条、看全 pane（采纳）**：最贴合用户「缺外部滚动条」原话，且唯一能满足「看到底部 / 看全」。
   - B fit-to-height（缩放 / 重算行数）：**排除** —— 不缩字体则物理放不下，且与「前端不 resize 后端」契约冲突。
   - C 前端请求后端 resize pane：**排除** —— WS 明确 `READ_ONLY` 拒绝 `resize`（`slot-terminal.ws.ts`），且会干扰 ccb 原生 pane / sidebar。
   - D 裁到底部、只看当前底部视口：**备选** —— 能露出最新输出，但裁掉 pane 上半、未满足「看全」。
   → 决策 A；若用户实际更想要 D（接受裁顶换单一干净视口），技术设计前可回退。

3. [中·已决策] 修复范围：在**共享 `SlotTerminalSurface` 层**解决 rows / 像素错配（Codex 同），需求详情页仅父容器配合；连带修好 `MainTerminalPanel`。非仅改需求详情页（仅改 `.terminalColumn`，因 `.xterm{height:100%}` 不一定生效）。

4. [中·设计约束] 纵向修复须避免破坏既有「单一纵向滚动条」体验：实现为外层单一滚动条、抑制 xterm 内置冗余滚动条。

5. [低·设计约束] auto-follow：现 `scrollToBottom` 只跟随 xterm buffer；改外层滚动后须新增 host 级「在底部才跟随」，否则每帧把用户弹离查看位置。
