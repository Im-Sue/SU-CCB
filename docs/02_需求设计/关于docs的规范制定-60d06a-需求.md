## Claude 解读

本需求是一次**白纸重做的、根本性 docs 文档治理设计**,不是给现状打补丁,也不是字面"把多个文件合并成一个"。

**要解决的真问题(用户视角)**

1. **命名/格式没标准** —— docs 下文档怎么取名、怎么写、用什么格式,缺统一规范,内容随意。
2. **历史包袱重** —— 大量旧的、过程性的、孤儿文档堆积,该清理,只留 v1.0-plugin-sovereignty 主线相关。
3. **体验割裂 + 结构过碎** —— 在 Console 建一个需求,背后散落成一堆"机器文件"(需求、设计、拆分、子任务、状态分布在多个子目录),既看不到一份连贯可读的"需求包",目录层级也过深过碎。

**用户本轮三条拍板(已收敛的方向)**

- **不被历史决策捆绑**:此前 ADR-0033 + 三份治理 spec(cleanup / governance-impl / housekeeping)从此只作"参考证据",不作"既定结论";完全按本需求重新讨论、重新设计。
- **从根本治理**:排除"只在上层加一个好看的聚合视图、底层结构不动"的轻量方案;接受触及底层结构。
- **方法 = 参考项目深度提炼 + 与本项目融合**:吃透 `prediction_market/docs` 的 (a) 管理结构(按开发生命周期分 `00_~99_` 目录、README 标注"当前主流程"、旧文档保留并标注"已被替代"而非乱删、单文档持续更新)、(b) 模板系统(每个目录一份 `_模板_*.md`)、(c) 内容风格(业务/模块角度、少术语),提炼成原则后再与 SU-CCB 现实结合 —— 不是照抄。

**SU-CCB 的现实(设计必须正视的硬约束,不用于否决需求)**

SU-CCB 是元工具,`docs/.ccb/` 不是给人读的文档,而是 Console / indexer 当数据库读的机器文件:建需求、拆分、子任务、状态、归档各有**写死的固定落点**,且"文件放在哪个目录"本身就编码了它的状态(位置即语义)。因此"根本治理"**不等于**把机器文件也粗暴拍平;设计要给出一个**既让人看到参考项目那样清爽结构、又不破坏程序读写**的方案 —— 这是本需求要在技术设计阶段啃的核心。

**交付判定(什么算这个需求做完)**

产出一份"根本性 docs 治理设计",至少回答:
1. 人类知识库(`00_~99_`)按什么生命周期结构 + 命名规则 + 模板组织(是否去掉 console/plan/plugin 的 subject 分层)。
2. 在 Console 建需求后,人能否、以及如何看到一份连贯"需求包"。
3. 机器文件(`.ccb`)是原样保留、还是有节制演进;演进到哪、怎么不破坏 Console/indexer。
4. 旧文档(含 ADR-0033 + 3 份 spec 自身)如何清理 / 备份 / 标注 / 归档。
5. 内容质量标准(业务角度 + ASCII 图)怎么落进模板,成为明确要求而非可选润色。

设计须回来过审,不在分析阶段替用户定终稿。
