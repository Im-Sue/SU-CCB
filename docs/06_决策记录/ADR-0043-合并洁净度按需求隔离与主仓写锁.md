# ADR-0043: 合并洁净度按需求隔离 + 有界容忍 + 主仓写锁

> 一句话：把合并前的洁净度检查从「整个 docs/ 必须干净」改成「只管本需求自己的东西、对别的需求在途档有界容忍」，并补 technical_design 进收拾名单、加一把主仓写锁防并发踩踏。本 ADR **amends** ADR-0036 的 clean-gate 安全模型，ADR-0036 其余结论不变。

## 状态

active。修订（amend）[[ADR-0036]] 决策 1 中「合并前主仓必须全局干净」这一条 clean-gate 取舍；不取代 ADR-0036，其 per-需求 worktree 归属、字段驱动隔离等结论原文保留。

## 背景

ADR-0036 落地 per-需求实施 worktree 时，合并预览（`mergeRequirementWorktree → canonicalSyncCommit`）采用**全局严格** clean-gate：要求整个主仓 `docs/` 除当前需求白名单外一律干净，否则 escalate `canonical_dirty_outside_allowlist`。

req vlr74b（2026-06-08）autonomous-batch 实证暴露问题：实施/审查/归档全过，唯独合并被**别的 6 个需求的未提交文档 + 1 个经验沉淀档 + 本需求自己的技术设计稿**挡死。两层根因：

1. **没有「节点产物即时落 commit」环节**——需求档/设计稿/任务档从生成起长期未提交，要拖到各自需求走到合并那步才被顺带提交；而合并前 gate 是全局严格的，却只自动提交当前需求那一小片 → 多需求并行必然互挡。
2. **canonical-sync allowlist 漏 `technical_design`**——只收了需求档(02)、dev_task(03)、文档地图、journal、worktree 台账、拆分草案；设计稿不提交就被当「不明脏文件」挡合并。

## 决策

clean-gate 从「全局严格」改为「**按需求隔离 + 有界容忍 + 主仓写锁**」，落在 `su-ccb-claude-plugin/lib/worktree/`：

1. **三态 classifier**（`classifyDirtyEntry`，OWN/TOLERATE/FOREIGN）：
   - **OWN**（∈ 本需求 allowlist）→ 收拢提交；
   - **TOLERATE**（可识别的受管在途档：别需求的 02 requirement-bound / 03 technical_design·dev_task-bound、`.ccb/worktrees|drafts` 协调件、04–06 **已被 git 跟踪**的常青档）→ 放行、**绝不代提交**、记 `tolerated_paths`；
   - **FOREIGN**（源码、未跟踪无绑定常青档、frontmatter 读不出、删除/rename 读不到归属）→ 照旧 escalate。
   - 行内多路径：每条都 OWN/TOLERATE 才放行，任一 FOREIGN 即整行 FOREIGN。
2. **allowlist 补 technical_design**（`doc_type==="technical_design" && requirement_id===reqId`），与 dev_task 谓词并列。
3. **主仓写锁 `withCanonicalRepoLock(projectRoot)`**：锁目标 `.ccb/locks/canonical-repo`（**严禁 docs/ 树内**——否则 gate 会看见自己的锁文件而自爆）；锁序固定 `requirement → canonical`（防死锁）；覆盖三条写主仓路径——单空间 merge、多空间 `mergeOneSpace(root)`、association `toSpace=root` 的 gitlink commit；`LockTimeoutError` 转结构化 escalation `canonical_repo_lock_timeout`，不裸抛。
4. **`tolerated_paths` 诊断**（去重+排序，落 escalation/event payload）：放宽全局严格后错误类型会从 `canonical_dirty` 漂成 `merge_conflict`，留痕便于归因。

**F1（节点产物即时落 commit）拆独立后续需求**（用户 2026-06-09 拍板「先做 1」）：本需求只做 F3+F2+锁；F1 要改提交模型，回归面独立，且有本修订后紧迫性大降。

## 否决的备选

- **blanket 容忍整个 `docs/`**：太松，等于废掉脏树保护，源码改动和不明文件会被一起吞进合并。改取三态严格判定。
- **连新建未跟踪的无绑定常青档也默认容忍**：它们无 `requirement_id`、无提交归属，容忍后会把「挡合并」变成「产物永远没人提交」（Codex 指出我方案最可能错的一处）。安全默认：新建无绑定常青档**仍 escalate**。
- **ADR 按 `related_doc` 反查绑回需求纳 allowlist**：现有简易 frontmatter parser 解析不了多行 YAML 数组（`related_doc` 是多行列表），可靠反查需健壮解析 → 列为 F1/后续，本需求取安全默认（ADR 不进 allowlist）。
- **本需求一并做 F1**：见上，拆后续。

## 后果

**正面**：多需求并行时本需求合并不再被别需求在途产物阻塞；本需求 canonical 产物（含设计稿）能正确收拢提交；放宽全局严格附带解除的「隐性写主仓串行化」由仓库锁显式补回。

**安全保持**：源码改动、未跟踪无绑定脏档、读不出归属的删除/rename 仍 FOREIGN escalate——「拒绝合并进脏树」的安全意图不丢。

**已知残留**（设计已拍板接受，用户 2026-06-09 确认）：新建无绑定常青档（新经验沉淀 / 新 ADR——**包括本 ADR-0043 自己**）合并时仍被 FOREIGN 拦，需人工提交过渡，由 F1 根治。
