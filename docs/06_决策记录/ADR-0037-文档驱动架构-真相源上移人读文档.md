# ADR-0037: docs 文档驱动架构 —— 真相源从 `.ccb` 上移到人读文档

> 一个决策一篇,记下"为什么这么定",防止以后反复扯 / 误改
>
> **编号**: ADR-0037
> **状态**: 已采纳(范式方向已定;实施迁移与旧 ADR 标记由后续 spec / 动作执行)
> **日期**: 2026-05-27
> **拍板人**: 用户
> **取代 / 修订**: 取代 ADR-0023(核心主张)、ADR-0033 + 其 3 份 spec;修订 ADR-0024、ADR-0012 的真相源定位

---

## 一、背景

- **原范式**(ADR-0023 plugin sovereignty):`docs/.ccb/` 文件是 canonical 真相,Console 的 SQLite DB 只是投影。
- **现状问题**(本需求 cmpmv55uy7d2673077860d06a 触发):在 Console 建一个需求,背后散成一堆机器文件(需求 / 设计 / 拆分 / 子任务 / 状态 分布在 `.ccb` 多个子目录),看不到一份连贯可读的东西、目录过深过碎;`.ccb` 机器层与人读知识库搅在同一棵树。
- **用户定调**(2026-05-27):不被历史决策捆绑、完全重做;从根本治理(非临时);吃透参考项目 `prediction_market` 的结构 / 模板 / 内容再融合(非字面合并)。
- **核验事实**:现状其实已是"markdown 文件 = 真相、DB = 可重建投影"(代码注释"md 是真源,DB 失败 next scan 自愈");真相 md 却埋在 `.ccb/`,系统也不读人读知识库;plugin 无声明式目录契约,全靠硬编码路径。

---

## 二、决策

1. **真相上移**:需求 / 设计 / 进度 / 状态统一维护在人读 `docs/01-99` 文档;真相 markdown 不再藏在 `.ccb/`。
2. **`.ccb/` 降级为「索引 + 协调层」**:只存目录契约、索引缓存、EventJournal、写锁、(可能)拆分草稿 JSON 等机器机制件;**不再存需求 / 任务本体**。
3. **DB 仍是可重建投影**:Console 读 DB,文档改 → 缓存更新 → 界面变(不变)。
4. **规则只写一份**:plugin 声明一份"目录契约"(目录 / 类型 / 每命令产物落点 / 索引入口),plugin 与 Console indexer 共读,取代硬编码路径。
5. **写作工作流是核心**:人读文档里可手改字段(描述性)与必须走命令的字段(关键状态)分开;手改坏关键状态 → 界面显式标"无效 / 部分",**不静默修正**。
6. **docs 结构**:`00-99` 生命周期目录 + 一份《文档地图》(状态三档:生效中 / 历史 / 归档)+ 每目录一份模板;参考视图(架构、模块规格)按部分(前端 / 后端 / 后台管理)拆、命名部分在前,集成视图(需求 / 技术设计 / 开发任务)整份不拆;ADR 入 `06_决策记录/`。

---

## 三、否决的方案

| 方案 | 为什么没选 |
|------|------------|
| 磁盘真扁平(每需求一份文件当唯一真相) | 会砸 createRequirement / breakdown CAS / subtask materialize / path-as-semantics 一整套,等于重写地基;用户也明确"不是单纯合并成一个文件" |
| 纯换门面(只加聚合视图、底层不动) | 用户要"从根本治理",不够彻底 |
| 维持现状(ADR-0023 不动) | 没解决"建需求看到一堆产物 + 目录过碎"的体验与治理问题 |

---

## 四、影响

- **好处**:人读文档当家、一处可读;目录清爽;规则集中、可维护;贴合"文档驱动"。
- **代价 / 风险**:
  - 中大型横切迁移(见受影响)。
  - **双真相**风险 → 对策:冲突以人读文档为准,其余重建;不一致显式标"无效 / 部分",不静默修。
  - 人手改坏关键状态 → 对策:只有过校验的内容进 accepted 投影,坏的界面标红。
- **受影响**:Console server(indexer / document-parser / 需求创建编辑 / 投影)、Claude plugin(lib / skills / 路径 helper)、`su-flow/su-init/su-plan/su-materialize/su-resume/su-status/su-reconcile`、kernel schema / manifest、docs 全树。
