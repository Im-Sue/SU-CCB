## 二、任务分解

### 一、任务概述

把控制台「需求管理」看板从 4 列改成 5 列、给每列加滚动条、并修正一处首页统计口径——纯前端展示优化,一行后端 / DB / 接口 / kernel 都不碰。

一句话:现在「推进中」这一列其实混着两种需求(还在规划的 + 已在交付的),卡片上早就分别标了「规划中 / 推进中」,只有列没拆;这次把列也拆开,顺手把「已归档」标题改成更贴切的「已搁置」,并给越堆越长的列加上限高滚动。

**真相源**:技术设计 `docs/03_开发计划/优化-需求管理页面的优化-d91293-技术设计.md`(下称技术设计),其「八、文件结构 / 变更清单」为逐文件改动清单;需求 `docs/02_需求设计/优化-需求管理页面的优化-d91293-需求.md`。

**术语(白话)**:planning = 规划中(需求已分析、在做技术设计 / 拆分);delivering = 推进中(子任务已在交付);桶 / 列 = 看板的一栏。

**为什么是一个子任务**:全部改动围绕同一个分桶真相源 `ui-mapping.ts`,且组件测试与 e12 快照同时断言列结构和滚动属性,拆开会让多个子任务改同一测试文件、制造二次回归窗口;CSS 仅数行无独立交付价值。经 slot3_codex 执行视角协商确认合并不拆。

### 二、任务分解(改 4 源文件 + 3 测试,以技术设计 §八 为准)

目标态列序(按生命周期,与 status 枚举同序):

```
待处理 → 规划中 → 推进中 → 已交付 → 已搁置(灰)
drafting   planning   delivering   delivered   deferred+cancelled
```

1. **分桶真相源 `web/src/lib/ui-mapping.ts`**
   - `RequirementTabKey`:由 `"pending" | "inProgress" | "delivered" | "archived"` → 加 `planning`、`inProgress` 重命名 `delivering`,即 `"pending" | "planning" | "delivering" | "delivered" | "archived"`。
   - `REQUIREMENT_TAB_LABELS`:加「规划中」、`delivering`=「推进中」、`archived` label 改「已搁置」。
   - `classifyRequirementTab`:`planning→planning`(单独桶)、`delivering→delivering`;其余不变(drafting→pending、delivered→delivered、deferred|cancelled→archived、未知→archived 兜底)。
   - `getRequirementAction(deferred/cancelled)`:旧返回 label「已归档」→改「已搁置」(消除残留旧词,Codex 点出)。
   - 新增 `isActiveRequirementTab(tab)` 谓词:`pending|planning|delivering`=true,`delivered|archived`=false。

2. **列定义 `web/src/lib/requirement-board-config.ts`**
   - `REQUIREMENT_BOARD_COLUMNS`:4 列 → 5 列,按上面列序;`archived` 列保留 `muted`;`buckets` Record 同步加 planning。
   - JSDoc「4 个分类桶 / 4 列」→「5」。

3. **首页连带 `web/src/pages/overview/OverviewPage.tsx`**
   - 活跃口径由内联 `tab === "inProgress"` 改用 `isActiveRequirementTab`(含 planning + delivering);行内文案「待处理 + 推进中」→「待处理 + 规划中 + 推进中」。
   - (本任务头号回归风险:不改则规划中需求从首页「活跃」静默漏算。)

4. **列内滚动 `web/src/pages/requirements/RequirementsPage.module.css`**
   - `.columnBody` 加 `max-height: calc(100vh - ~220px); overflow-y: auto; scrollbar-gutter: stable; overscroll-behavior: contain;`(具体 px 实现时按视觉调,沿用仓内 `DocumentsPage.module.css` 同款 calc idiom)。
   - **窄屏口径(Codex 收口)**:窄屏 media query **解除**列限高,回归整页滚动(列内滚动只在桌面)。验收按此口径,勿在窄屏强制列内滚动。

5. **测试同步**
   - `ui-mapping.spec.ts`:`classifyRequirementTab` 六状态 → 五桶;新增 `isActiveRequirementTab` 真值表。
   - `RequirementsPage.spec.tsx`:渲染 5 列、标题(含「已搁置」)、archived muted、列内容区滚动属性存在。
   - `e12-acceptance-snapshots.spec.tsx` + `__snapshots__`:overview + requirements baseline 重生成;**tasks board「已归档」快照不应变**。

**更新顺序(防快照掩盖回归,Codex 确认)**:改实现 + 非快照断言 → 跑相关 vitest 确认只剩 E12 快照差 → 人工看 diff 确认只变 overview / requirements、tasks board 不变 → 定向 `pnpm vitest -u e12-acceptance-snapshots.spec.tsx`(勿全量 `-u`)→ 最后 `pnpm build` / `pnpm test`。**不要先 `-u`**。
