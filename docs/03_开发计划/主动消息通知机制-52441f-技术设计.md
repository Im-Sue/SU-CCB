# 主动消息通知机制 技术设计

> 一句话：建一个**项目级「该找你的事」单一 derive 源（attention-inbox）**，让 Console 浏览器通知与 tmux sidebar 都只做它的投影；ack/DND 用一张**极小状态表**持久化（承认是小 migration），内容不物化。
>
> **无独立 status** —— 跟随 `requirement_id`（cmpzoupdv…52441f）。需求来源：`docs/02_需求设计/整套oriel和plugin缺少主动的消息通知机制-52441f-需求.md`。

> 经 technical_design 节点产出：双路源码核验（web hook 点 + server schema/通道）+ slot1_codex 设计层协商（job_a8ffa861fee7）+ 用户方向拍板（local-first）。**一处用户拍板待定**：是否接受小 migration（见第十章）。

## 一、设计概述

| 项 | 说明 |
|----|------|
| 名称 | 主动消息通知机制（attention notification） |
| 核心职责 | 当出现「需要你介入 / 你的活已完成」事件时主动触达用户，并支持一键切换进入对应需求 / slot |
| 设计原则 | **单一 attention 源、多端只投影、内容 derive 不物化、仅持久化极小用户状态（ack/DND）、best-effort 非阻塞、正确性优先于时效** |
| 需求来源 | `docs/02_需求设计/整套oriel和plugin缺少主动的消息通知机制-52441f-需求.md` |
| 覆盖范围（本期 A+B） | 项目级 attention derive 源 + ack/DND 极小持久化 + Console 浏览器通知投影 + tmux sidebar attention 投影 |
| 不覆盖（非目标） | C 硬触达（OS/邮件/IM，仅留 adapter 接口延后）；改 CCB runtime 本体；per-user ack/DND（本地单用户）；attention 内容物化表 |

## 二、方案与架构

```
                 ┌──────────── 单一 attention 源（Console server）────────────┐
 源表(只读) ───► │  attention-inbox.service.computeAttention(projectId)        │
 ReviewIntent    │   批量 derive：ReviewIntent / ConsultRequest / dev_task     │
 ConsultRequest  │   approval+decision / EventJournal(receipt|rejected) /      │
 dev_task doc    │   SlotBinding(unhealthy|recovering)                          │
 EventJournal    │   ── left-anti-join AttentionAck(已读) ── 过滤 DND ──►       │
 SlotBinding     │   AttentionItem[]（含 source-native ref + severity）         │
                 └───────┬───────────────────────────────────┬────────────────┘
                         │ (投影 A)                            │ (投影 B，server 内)
                         ▼                                    ▼
   GET /api/projects/:id/attention (+ack/settings)     computeSlotTipsProjection 扩展
   [P1] /attention/stream (SSE, 并存新通道)             （join 绑定需求的 attention，
                         │                                debounce+content-hash 后写 tips）
                         ▼                                    ▼
   web ConsoleLayout 挂 NotificationManager 单例        managed-config 原子写 [ui.sidebar.view].tips
   poll 5-10s diff → 新高 severity →                    `slot-2: ⚠️待你决策 需求名`
   Notification + 声音 + favicon/title badge            CCB sidebar 自轮询 ~1-2s 显示
   点击 → navigate(/requirements|/tasks) + POST ack
```

| 关键原则 | 说明 |
|----------|------|
| 单一源 | A、B 都调 `attention-inbox.service` 同一 derive，规则只算一遍，绝不在两端各算 |
| 内容不物化 | attention 条目实时 derive（真相留源表），只持久化 ack/DND 极小用户状态 → 避开 projection-freeze 类陈旧 bug |
| 并存通道 | **不改** task-keyed `publishProjectionSignal` / `/api/tasks/:taskId/events`；新增 project attention 通道与之并存 |
| best-effort | 通知 / sidebar 写失败绝不影响主业务；通知必达靠 poll 兜底，不依赖 fail-open hook |
| 时效让位正确性 | poll MVP 5-10s；SSE 作 P1 上限 ~1-2s |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `pending-interactions.service.ts` | 抽取其聚合逻辑到 project 级**批量** derive（新 `attention-inbox.service.ts`），复用映射但改批量查询 | task 级 endpoint 与语义不动 |
| `event-stream.service.ts` | （仅 P1）参照 streamTaskEvents 新增 project attention stream | task-keyed `projectionListeners` / `publishProjectionSignal` / task 订阅者不动 |
| `slot-tips-projection.service.ts` | `computeSlotTipsProjection` join attention，加 ⚠️ 标记；写前加 debounce + content-hash guard | 全量重算 + 原子写 + 项目锁 + 自轮询语义不变 |
| Prisma schema | **新增极小 `AttentionAck` 表 + DND 设置**（小 migration，待用户拍板） | 不新建 attention 内容表；不动 EventJournal / ProjectSettings 现有列语义 |
| web `ui-store.ts` / `ConsoleLayout` | 加 notificationSettings + unread slice；挂 `NotificationManager` 单例 | 现有 toast / 路由 / store 结构不动 |

## 三、关键决策与取舍

- **Fork-1 内容 derive + 极小状态表（选） vs 完整 AttentionItem 内容表（拒）**：选 derive 内容、只持久化 ack/DND。拒内容表——物化 derive 会重蹈 projection-freeze 陈旧坑，MVP 不需要。
- **ack/DND 落点（修正）**：原设想「ack 作 EventJournal 事件 + DND 进 ProjectSettings JSON」——经 Codex 源码核验**不成立**：`EventJournal` subject_type 严格(requirement|subtask)、strict discriminated union、无 `attention_acked`，且是 plugin 主权边界（Console「已读」不该混入）；`ProjectSettings` 仅 3 个 strict JSON 列、payload schema strict。**改为**：新增极小 `AttentionAck(projectId, ref, ackedAt)` 表 + DND 设置（`ProjectSettings.attentionSettingsJson` 新列或 `ProjectAttentionSettings` 表）。**承认这是小 migration / 公共 schema 变更 → 升级用户拍板（第十章）。**
- **Fork-2 poll MVP（选）vs SSE 即做**：选 poll 5-10s 落 MVP（简单、绕开 ADR-0027「SSE push 前端 v1.x 非目标」）；SSE 作 P1，若做则**补 ADR addendum** 并新增并存 project stream，不改坏 task stream。
- **信号通道：并存（选）vs 提升 task 通道（拒）**：拒「把 `publishProjectionSignal` 提到 project 级」——会破坏现有 task 订阅语义；新增并存 project attention 通道。
- **stable ref：source-native（选）vs 粗 subjectType+subjectId+kind（拒）**：用源生 id —— `review_intent:<id>` / `consult_request:<id>` / `event_journal:<eventId>` / `dev_task_approval:<taskKey>/<ref>` / `slot_binding:<slotId>/<state>`，避免重启 / 重扫重复弹或误 ack。
- **拒绝**：改 CCB runtime 本体加通知（runtime 不懂 requirement attention）；启用 CCB 邮件通道做主路径（本机 `lib/mail` 缺失 / 退役）。

## 四、核心流程 / 逻辑

```
computeAttention(projectId):                        # 单一源
  并行批量查 ReviewIntent[pending,projectId] /
    ConsultRequest[pending, taskId∈project 的 task]（ConsultRequest 无 projectId，需 join）/
    Document[kind=dev_task,projectId].approval_records+pending_user_decision /
    EventJournal[projectId, type∈{codex_receipt_ready,codex_rejected}, 近窗口] /
    SlotBinding[projectId, state∈{unhealthy,recovering}]
  → 映射为 AttentionItem{ ref(source-native), kind, severity, subjectType, subjectId, title, cta }
  → left-anti-join AttentionAck[projectId]（按 ref 过滤已读）
  → 过滤 DND（settings.dndUntil）→ 排序（severity desc, time）→ 返回

ack(projectId, ref):  upsert AttentionAck(projectId, ref, ackedAt=now)   # 点击/进入即调

NotificationManager（web，ConsoleLayout 单例）:
  每 5-10s（复用现有 ticker）拉 computeAttention → 与上次 by-ref diff
  新增高 severity item 且非 DND：
    若未授权 → Notification.requestPermission()（一次）
    new Notification(title) + 可选 <audio> 声音 + favicon/title badge
    onclick → navigate(/requirements/:id 或 /tasks/:id?node=) → POST ack
  多标签去重：leader 选举（BroadcastChannel/localStorage）只让一个标签弹

syncSlotTips（扩展，server）:
  attention = computeAttention(projectId)           # 同一源
  rows = activeSlotBindings join requirement
  tips = rows.map(r => hasHighAttention(attention, r.requirementId)
                        ? `${slot}: ⚠️待你决策 ${title}` : `${slot}: ${title}`)
  if contentHash(tips) 变化（debounce 窗口内）→ ensureManagedCcbConfig 原子写   # 防高频狂写
```

## 五、测试策略

- [ ] 单元：`computeAttention` 批量查询无 N+1；ConsultRequest 经 task join 正确纳入
- [ ] 单元：ack 后该 ref 从 derive 结果消失；未 ack 的保留；DND 窗口内全部抑制
- [ ] 单元：source-native ref 稳定——同一源对象跨两次 derive ref 不变（不重复弹）
- [ ] 单元：severity 分级（仅 attention 级进主动弹候选；warning/info 不弹）
- [ ] 单元：sidebar content-hash guard —— attention 高频变动但 tips 内容不变时**不写**配置
- [ ] 集成：slot-binding 绑定的需求出现 pending review → tips 出现 ⚠️ 标记；ack 后消失
- [ ] 集成：ack endpoint upsert 幂等（重复 ack 同 ref 不报错）
- [ ] 集成：DND 设置读写
- [ ] 组件：NotificationManager 多标签 leader 选举只弹一次；onclick 导航 + ack；权限拒绝时降级为 title/favicon badge
- [ ] 回归：task 级 `/api/tasks/:taskId/events` SSE 与订阅者不受影响

## 六、数据设计

**新增极小用户状态持久化（小 migration，待用户拍板）**：

```prisma
model AttentionAck {            // 仅用户「已读」状态，不物化 attention 内容
  id         String   @id @default(cuid())
  projectId  String
  ref        String   // source-native: review_intent:<id> | consult_request:<id> | event_journal:<eventId> | dev_task_approval:<taskKey>/<ref> | slot_binding:<slotId>/<state>
  ackedAt    DateTime @default(now())
  project    Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  @@unique([projectId, ref])
  @@index([projectId, ackedAt])
}
```

**DND 设置**（二选一，均为小 schema 变更）：① `ProjectSettings` 加 `attentionSettingsJson` 列 + 扩 payload schema；② 新 `ProjectAttentionSettings(projectId, dndUntil, ...)` 表。倾向 ①（贴现有 project 设置）。

**只读源表**（不改）：ReviewIntent / ConsultRequest / Document(dev_task) / EventJournal / SlotBinding / Requirement。

## 十、迁移影响与风险 + 用户拍板项

**【用户拍板 · 命中 schema/migration 必审门】**

- **是否接受一个小 migration**（新增 `AttentionAck` 极小状态表 + DND 设置列/表）？
  - **默认（推荐）：接受**。这是经核验后正确且最省的持久化路径。
  - **fallback（不改库）**：ack/DND 退化为浏览器 `localStorage` —— 则 **sidebar(B) 无法共享已读状态**、清缓存即丢、跨设备不一致。功能明显受损。
- **DND 作用域**：默认 **project 全局**（本地单用户工具，无 User 模型）。
- **交付时效**：默认 **poll MVP（5-10s）先落 → SSE 作 P1**（届时补 ADR-0027 addendum）。

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| project 级 derive N+1 / 性能 | 中 | 慢 | 批量查询 + 小窗口 + ack 表过滤；规模本地中小 |
| sidebar 高频写配置 | 中 | 抖动 / 狂写 | debounce + content-hash guard，内容不变不写 |
| stable ref 不稳 → 重复弹 / 误 ack | 中 | 体验 | source-native id（见第三章） |
| 浏览器通知 UX 竞态（权限 / 多标签 / 导航失败） | 中 | 重复弹 / 漏弹 | 多标签 leader 选举 + 权限降级 + 测试覆盖 |
| ADR-0027 漂移（SSE） | 低 | 治理质疑 | poll MVP 绕开；SSE 时补 addendum |
| 改坏 task event stream | 低 | 回归 | 并存新通道，不动 task-keyed 通道 + 回归测试 |

## 协商与反思留痕

- **协商**：slot1_codex 设计层 1 轮（job_a8ffa861fee7）。Codex 抓出我设计的实质错误：① Fork-1A「零 migration」不成立（EventJournal/ProjectSettings 均 strict + 主权边界，ack/DND 无法白塞）；② pending-interactions 不能裸 lift（task 级 / N+1，ConsultRequest 无 projectId）；③ 不应提升 task-keyed 信号通道，应并存；④ sidebar 高频写需 debounce/content-hash；⑤ stable ref 必须 source-native。我全部采纳。
- **sc 替代说明**：未单独跑 `/sc:design`、`/sc:analyze`；其视角由**双路源码核验**（web hook 点 + server schema/通道，= /sc:analyze 现有代码影响）+ **我自拟架构草案**（= /sc:design）+ **Codex 设计质疑**覆盖。Codex `analysis_depth_hint=sc-design(high)` 已反映本设计跨 schema/通道的复杂度，相应风险已落第十章。
- **4 锚点反思**：① 同意——ack/DND 须小 migration、ack 不入 EventJournal 主权边界、并存通道、sidebar debounce、source-native ref、批量 derive；② 修正自己——我为躲 schema 必审门做了个站不住的「零 migration」设计，承认小 migration 才是正解；与 Codex 在「内容不物化 + 极小状态」其实收敛同一点，无实质分歧；③ 盲点——把 EventJournal/ProjectSettings 当松 schema、漏 N+1/ConsultRequest 无 projectId、漏 sidebar 高频写、漏 source-native ref、漏浏览器通知竞态测试面；④ 下一步——落本文档 + 升级用户拍板小 migration（带默认）；拍板后再进 task_breakdown，不擅自拆。
- **用户拍板**：方向 local-first（已采纳）；**小 migration 待用户确认**（第十章）。
