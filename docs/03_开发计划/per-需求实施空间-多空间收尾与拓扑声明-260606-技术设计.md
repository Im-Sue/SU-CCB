## 二、方案与架构

```
docs/.ccb/config/implementation-topology.yaml   ← 项目层机器声明(版本化)
        │ ensure 时展开恰好一次(content_hash 入档)
        ▼
docs/.ccb/worktrees/<req>.json  requirement-worktree-v0.2
  aggregate_status + spaces[](per-space status) + associations[](pending/synced)
        ▲ 单 requirement 锁下增量写,每次写盘重算 aggregate
        │
lib/worktree（三层）
  ├─ space 级原语: 今天 5 个 helper 体,git cwd 参数化为 join(projectRoot, space.repo)
  ├─ requirement 级 orchestrator: ensure/merge/cleanup/reopen/discard(公开 API 名称签名不变)
  │    merge = preflight(全空间只读) → root → 子空间(声明序) → associations(sync+verify)
  └─ association executor registry: kind → {sync, verify}(首个 kind: git_submodule_gitlink)
```

| 关键原则 | 说明 |
|----------|------|
| 展开恰好一次 | 首次 ensure 把全量空间实例(status=pending)+topology content_hash 写入 runtime；此后 merge/cleanup/discard 只信 runtime 空间集，topology 改版不追溯已展开需求 |
| root-first 序 | 子仓先合会推进子仓 canonical HEAD，root canonical 立现 gitlink dirty 撞 `canonical_dirty_outside_allowlist`；root→子仓→association 才能收敛 clean |
| preflight-then-destructive | destructive 动作前对全部空间跑只读 preflight（worktree/分支/clean/target 在位且各 repo 在各自 target 上），把部分失败面压到最小（consult C1/C7） |
| association 必 verify | sync 后强制 verify（gitlink sha==子空间 target tip、to_space 在 target 上、pathspec 外无脏、commit 可追溯）才标 synced；重入时 verify 可独立运行，杜绝假 synced（consult C6） |
| kernel 零拓扑词汇 | submodule/gitlink/superproject 只活在 docs/.ccb/config + lib executor + 本项目文档；kernel 只写量词与不变量，下游零声明自然退化单空间 |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `lib/worktree/index.mjs` | 5 helper 重构为 orchestrator+space 原语 | 公开 API 名称/签名、escalation 风格、req 级锁、260606 force-retry、canonicalSyncCommit(仍只在 root) |
| `docs/.ccb/worktrees/<req>.json` | v0.1→v0.2(spaces[]+associations[]+aggregate) | v0.1 读时内存 lift，不批量迁移 |
| `lib/subtask`(materialize) | 零改动 | dev_task `code_workspace` 单字段、subtask-spec validator |
| kernel manifests(dispatch/archive/implementation) | 量词措辞 amend | 6 章节结构、节点链 |
| ADR-0036 | amend(不变量推广+拓扑契约决策) | 既有决策 1-8 主体 |
| `docs/.ccb/docs-structure-contract.yaml` | machine_layer.holds 补 `config/` 条目(additive) | 其余全部 |

## 三、关键决策与取舍（协商结论）

- **D1 拓扑声明落点**（需求开放问题#1）：`docs/.ccb/config/implementation-topology.yaml`，`schema_version: implementation-topology-v0.1`，与 capabilities.project.yaml 并列；校验在 `lib/worktree/topology.mjs`，**不进 references/kernel**（kernel 零拓扑）。内容=spaces[]（space_id/kind/repo/worktree_path 模板/branch 模板，占位符 `<requirementId>`）+ associations[]（kind/from_space/to_space/submodule_path）。零声明默认=单 root space（由 dev_task code_workspace 派生），行为与今天逐字节一致。否决放 kernel schemas（污染 kernel 词汇表）；否决 prose 声明（需求阶段已否决，缺口形成机制本身）。
- **D2 orchestrator/executor 切分**（需求开放问题#2）：三层——space 原语（cwd 参数化）/requirement orchestrator（循环+增量写盘+事件）/association executor registry（kind→`{sync, verify}`，未知 kind→escalated）。executor 框架在子任务④，gitlink executor 本体在⑤。**verify 独立于 sync**（Codex C6 驳回我原 sync-only 方案，采纳：无 verify 时 noop/重入/人工修复后无法审计真假 synced）。
- **D3 空间来源 S1**：每空间=该 repo canonical checkout 的独立 worktree（`git -C <repo> worktree add <path> -b ccb/req-<id>`）。否决 S2"root worktree 内嵌 submodule checkout 作为空间"：嵌入式 gitdir 在 `.git/worktrees/<id>/modules/` 下与子仓 canonical ref 隔离、merge 需 fetch 管道、生命周期被 root worktree 绑架、su-oriel（pinned 本地 only）无法 init——9dd2ae 实战已验证 S1。
- **D4 merge 执行序**：preflight(全空间只读)→root→子空间(声明序)→associations，stop-on-first-failure。否决"继续合并独立空间"：会扩大已推进 target 的面（Codex C7 同意）。已知残留风险：root merge 成功后子空间失败时，root target 已前进（可能含实施期手工 gitlink bump，指向未完成关联）——接受但 escalation payload 必须明示 `preview_consistency: "incomplete"`，恢复=重跑 merge 收敛（Codex C1）。
- **D5 aggregate 规则**（Codex C2 修正后）：`computeAggregateStatus` 优先序=all discarded→discarded; all archived→archived; all merged**且**all associations synced→merged; all ready→ready; 仅 pending+ready 混合→**pending**(ensure 进行中/未完，非失败态); 其余混合→escalated。ready 阶段 association pending 不影响 aggregate；崩溃中断自然呈现 escalated；每次写盘重算并持久化（单写者+锁下安全；orchestrator 运行中存在短暂 escalated 窗口，诚实可接受）。
- **D6 展开恰好一次 + pending 空间态**（Codex C3 同意）：首次 ensure 锁内写全量实例(pending)，物理 ensure 逐个翻 ready；ensure 失败记 `last_error{op:"ensure",space_id}` 后保持 throw 语义（不转 escalation，与现状一致）；dispatch 门=aggregate ready。
- **D7 dev_task schema 不动**（Codex C5 同意）：materialize 继续盖单 `code_workspace`（root 空间；ensure 与 topology root 展开交叉校验，漂移 ConflictError）。否决 `code_workspaces[]` 盖章：base_sha/target_branch 在 materialize 时不可知，必然产生双真相漂移。执行者的全空间路径表由 **dispatch brief 从 runtime 生成**；配套 implementation.node.md 与 ccb-execute 措辞同步（"主空间为 codeRoot；声明多空间时按 brief 空间表各自为 codeRoot"），否则执行者继续单空间实施（Codex 风险点）。
- **D8 target 捕获污染守卫**（本轮活体证据新增）：ensure 捕获的 target_branch 若命中本拓扑 branch 模板前缀（如 `ccb/req-`）→ escalate `implementation_branch_as_target`。实证：当下 `git submodule status` 显示 plugin canonical 停在 `ccb/req-cmpwtcleanupsubmodule260606`（260606 的子仓收尾残债=第 6 例正在发生），无此守卫则下个需求会把残留需求分支记成合并 target。确定性零配置（前缀由 branch 模板派生）。
- **D9 reopen/discard/cleanup 语义**：reopen 仅 aggregate merged、逐空间校验全过才整体翻 ready、associations 重置 pending（all-or-nothing，partial escalated 走恢复重跑不走 reopen）；discard 仅全空间 ready（root-first 保证 partial 必有 root merged→必被拦截）；cleanup 仅 aggregate merged、逐空间复用 260606 force-retry 删除语义、partial 失败保留成功空间 archived 事实。
- **D10 archiveRequirementWorktree**（Codex open question）：保留为 legacy 兼容 wrapper（multi-space 下=merge→cleanup 链式），**不属于任何节点契约**——su-quick-archive 已明文禁用，无 skill 消费方；不删除（测试与潜在外部调用兼容），标注 legacy。
- **D11 事件 additive**：既有 `requirement_worktree_*` 保持 aggregate 级，payload 增 spaces 摘要+失败 space_id；新增 `requirement_worktree_association_synced`。已核 finalize evidence 链不依赖 worktree 事件，additive 安全。
- **D12 SU-CCB 拓扑实例**：4 空间（root + su-ccb-claude-plugin + su-oriel + su-ccb-codex-skills，三 submodule 均已 init 且会承接实施提交）+ 3 条 gitlink association。未触碰空间全生命周期 no-op（branch tip==base→merge 跳过补 merged），换取规则全覆盖（第 5/6 例的形成机制=有仓游离在规则外）。否决按需子集声明：增加每需求决策面，premature。

**AI 自决实现细节**：模块文件切分（state.mjs/topology.mjs/associations.mjs）、escalation reason 枚举命名、fixture builder 组织、pending 写盘批次粒度。**用户授权事项**：无新增——零新依赖、无 DB migration、无外发动作；runtime schema 演进与新 yaml 均在需求立项已授权方向内；两项开放问题属需求§七显式移交本设计的决策。

## 四、核心流程 / 逻辑

```
mergeRequirementWorktree(req):
  锁(req) → 各 repo worktree prune → 读 runtime(v0.1 则内存 lift)
  ├─ Phase 0 preflight(全空间只读,零副作用):
  │    每空间: worktree 在位+branch 匹配+porcelain clean
  │           +target_branch 存在+该 repo 当前在 target 上
  │    任一不过 → escalated(space_id, reason) 全程未动任何 repo
  ├─ Phase 1 root merge: canonicalSyncCommit(allowlist 不变,仍仅 root)
  │    → merge --no-edit → 写 space[root]=merged(增量写盘,aggregate 重算)
  ├─ Phase 2 子空间按声明序: 各自 repo 内 merge 回各自记录的 target
  │    失败 → space.last_error + aggregate=escalated + 事件 + 停(不继续后续空间)
  │    (此时 root target 已前进 → payload 带 preview_consistency:"incomplete")
  ├─ Phase 3 associations 逐条: executor.sync(scoped pathspec add+commit,
  │    porcelain 含声明 path 外脏件→escalated association_dirty_outside_path)
  │    → executor.verify(gitlink sha==from_space target tip / to_space 在 target
  │    / commit 在 target 上) → 过则 synced+synced_commit_sha 入档
  │    FF 已对齐 → noop:true+现 gitlink sha 入档(仍跑 verify)
  └─ 全过 → aggregate=merged(预览暂停态) + requirement_worktree_merged 事件
  恢复重入: 按 per-space status 跳过 merged 只重试未成;全 merged+assoc pending
            自然走 association-only;不重放已 merged 空间
```

| 处理规则 | 说明 |
|----------|------|
| 幂等/恢复 | 每空间增量写盘；重跑 orchestrator 按 runtime 状态续作；ensure 重入 existing→noop；v0.1 lift 后旧需求行为不变（associations 空集 vacuous） |
| 失败处理 | 沿用 escalation 结构化返回+journal 事件双通道；escalated 永不删现场；ensure 保持 throw 语义 |
| 安全边界 | 每空间 merge target 只用该空间 runtime 记录值，绝不 fallback/继承；D8 守卫拦污染 target；association 只 commit 声明 pathspec；discard 任一 merged 即拒 |
| 可观测 | per-space last_error、aggregate last_error{op,space_id}、association synced_commit_sha/noop、escalation payload 带空间序号与 preview_consistency |

## 六、数据设计

**runtime `docs/.ccb/worktrees/<req>.json` → `requirement-worktree-v0.2`**：

| 实体 | 关键字段 | 说明 |
|------|----------|------|
| 顶层 | requirement_id, aggregate_status, spaces[], associations[], topology_source{path,schema_version,content_hash}, last_error{op,space_id?,reason,at}, created_at/updated_at | aggregate 每次写盘重算持久化 |
| space | space_id, kind("git_worktree"), repo(相对仓根,root="."), path, branch, target_branch(各 repo ensure 捕获), base_sha, status, merged_branch_sha, target_sha_after_merge, merged_at, merge{}, archive{}, last_error | 字段语义沿 v0.1 平移 |
| association | association_id, kind("git_submodule_gitlink"), from_space, to_space, submodule_path, status, synced_commit_sha, noop, synced_at, last_error | synced 仅在 verify 通过后 |

**状态/枚举**：

| 字段 | 值 | 说明 |
|------|----|------|
| space.status | pending → ready → merged → archived；ready→discarded | pending=已展开未物理 ensure（新增）；escalation 不是存储态 |
| aggregate_status | pending / ready / merged / archived / discarded / escalated | 见 D5 优先序；merged 仍=预览暂停非交付终态（260604 语义） |
| association.status | pending / synced | reopen 时重置 pending |

**topology yaml（v0.1，SU-CCB 实例骨架）**：spaces: root(".", `../SU-CCB-req-<requirementId>`) + su-ccb-claude-plugin + su-oriel + su-ccb-codex-skills(各 `../<name>-req-<requirementId>`，branch 统一 `ccb/req-<requirementId>`)；associations: 3 条 git_submodule_gitlink(from=各子仓, to=root)。展开期校验：worktree path 不得嵌于 projectRoot/任一 repo/彼此（C4）。

## 十、迁移影响与风险

- **受影响**：lib/worktree 全部消费方（su-dispatch/su-batch/su-archive/su-quick-archive/requirement-cancel）——API 签名不变仅语义扩展；runtime 文件 lazy 升版。
- **打法**：5 子任务按依赖序串行（①契约→②schema→③ensure→④orchestrator→⑤落地 E2E），每步独立可验。
- **落地前置**：先人工收掉 260606 的 plugin 子仓残债（`ccb/req-cmpwtcleanupsubmodule260606` 分支 merge 回 plugin main+gitlink 对齐），否则 D8 守卫会拦住首个新需求 ensure（守卫按设计工作，但残债须清）。
- **回滚 / 恢复**：lib 纯增量重构，git revert 子任务粒度 commit；v0.2 文件向后不兼容旧 lib——回滚须连 runtime 文件一起 revert（escalated 现场保留原则保证无隐性丢失）。
