## 一、设计概述

**目标对齐**:开发者第一次进某项目、还没"初始化 CCB(跑 `ccb` 起运行时)+ 初始化知识库(`/ccb:su-init` 建索引)"时,做两件事让他不迷路:① 左侧"项目"组(需求/文档/Slot/运行)**锁起来**——点了不跳转、弹个"请先去初始化"的框,框里按钮把人带到概览;② **概览页整页变成"两步初始化引导"**(不再是一条横幅 + 一堆空指标卡),直接告诉他"先做这两步"。两步都就绪后,导航解锁、概览自动切回正常数据盘。命令面板和直接敲 URL 不拦(用户已确认豁免)。

| 项 | 说明 |
|----|------|
| 名称 | SU-Oriel 首次使用·项目导航门控 + 概览初始化引导 |
| 核心职责 | 未就绪前锁定"项目"组应用内入口;未就绪时概览整页引导初始化;就绪后回到数据盘 |
| 设计原则 | 单一状态源 / 不 fail-open / 复用既有初始化逻辑 / 纯前端 UX 门控(非鉴权) |
| 需求来源 | `docs/02_需求设计/优化-第一次使用引导问题-b9d989-需求.md` |
| 覆盖范围 | Sidebar"项目"组锁定;onboarding-required modal;概览未就绪整页引导;onboarding 状态单源化 |
| 不覆盖 | 命令面板 / 直达 URL / 旧链接重定向;后端鉴权;初始化流程本身;非"项目"组区域;~~概览卡片点击拦截(已废,卡片未就绪不渲染)~~ |

## 二、方案与架构

> 术语:**onboarding 状态**(= 项目接入就绪信息 `{ccbRuntimeReady, knowledgeBaseReady, manualCommand, ...}`,来自 `GET /api/projects/{id}/onboarding-status`);**ready**(= `ccbRuntimeReady && knowledgeBaseReady` 全绿);**gate / 门控**(= 就绪与否的锁定判断)。

```
                     ┌──────────────────────────────────────────┐
                     │  project-store (Zustand) · 单一状态源      │
                     │  onboardingByProject: Record<pid,{        │
                     │    value, fetchedAt, loading, error }>    │
                     │  ensureOnboarding(pid,{force})            │
                     │  (in-flight 去重: 模块级 Map,不进 state) │
                     └───────────────┬──────────────────────────┘
                       consume (订阅同一份)
   ┌──────────────────────┬──────────┴───────────┬────────────────────┐
   ▼                      ▼                       ▼                    ▼
 useProjectOnboarding   useProjectOnboarding   Sidebar"项目"组       onboarding-required
 Gate(pid)              Actions(pid)           gated 项:            modal (App.tsx 渲染)
 →{ready,loading,       检测+动作(复制/一键     !ready→锁定态         读 store + 触发时 pid
   error,requireInit()} init/终端)+init-job     点击 requireInit()    CTA→navigate(/overview)
   (Sidebar/modal 消费)  轮询;结果写回 store           
                              │ owner = OverviewPage(稳定层,跨切换不卸载)
                              ▼
                     OverviewPage:
                       ready ? <现概览: ReadyBanner + 指标卡>
                             : <ProjectSetupGuide 整页两步引导>(隐藏指标卡)
```

| 关键原则 | 说明 |
|----------|------|
| 单一状态源 | onboarding 状态只存 project-store 一份;导航/概览/引导/modal 读同一份,杜绝重复请求与竞态 |
| in-flight 去重在模块级 | 并发请求用模块级 `Map<pid,Promise>` 合并;promise 不进 Zustand state(state 只存可序列化数据) |
| 轮询写 store + owner 稳定 | init-job 轮询 owner 落 OverviewPage(跨"引导↔数据盘"切换不卸载),结果写 store;规避"就绪后引导卸载丢更新/unmounted setState" |
| 不 fail-open | `ready` 仅两步全绿为真;loading / error / 未就绪一律锁定 |
| 复用而非复刻 | 引导页与就绪 banner 复用同一套检测/动作 hook;不另造初始化逻辑 |

**与现有系统的关系 / 边界**:

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `stores/project-store.ts` | 新增 `onboardingByProject` slice + `ensureOnboarding` | 现有项目加载/选择逻辑不动 |
| `components/projects/ProjectOnboardingBanner.tsx` | 拆分:检测/动作/init-job 轮询移入 actions hook;runtime/knowledge-missing UI 迁入引导页;ready 态展示保留 | 三态文案、初始化动作语义、确认 modal 全保留(只是搬位置) |
| `pages/overview/OverviewPage.tsx` | 持有 actions hook(稳定 owner);`ready ? 现概览 : <ProjectSetupGuide/>`;移除原卡片 gate 计划 | `!selectedProjectId`/`loadingData` 分支、指标计算不动 |
| `components/layout/Sidebar.tsx` | "项目"组标 `gated`;`!ready` 锁定渲染 + 点击 `requireInit()` | 其它两组、collapsed、badge 不动 |
| `stores/ui-store.ts` | `modalType` 增 `"onboarding-required"` | 现有 modal 类型/机制不动 |
| `App.tsx` | 渲染 `onboarding-required` modal 分支 | 其它 modal 渲染不动 |

## 三、关键决策与取舍

- **状态落点**:选 **project-store 加 onboarding slice + 两个派生消费 hook**(`useProjectOnboardingGate` 给导航/modal;`useProjectOnboardingActions` 给概览引导/banner);否决"纯 hook + 模块缓存"(多处同步弱、测试分散),否决"独立 onboarding-store"(为小状态多建 store)。
- **概览未就绪态 = 整页引导**:`OverviewPage` 按 `ready` 切 `<ProjectSetupGuide>` / 现概览;否决原"横幅 + 指标卡 + 卡片点击拦截"——未就绪时空指标卡无意义、拦截多余,整页引导更直观且直击"不知道要初始化"。
- **banner 逻辑复用方式 = 抽共享 hook(方案 a)**:把检测 + 动作 + init-job 轮询抽到 `useProjectOnboardingActions`,引导页与就绪 banner 都消费;否决(b)给 banner 加 `full-page variant`(组件职责膨胀)、(c)逻辑下沉 guide(就绪 banner 又得重复)。
- **init-job 轮询归属 + 写 store(规避卸载竞态)**:轮询 owner 放**稳定的 OverviewPage 层**(它跨"引导↔数据盘"切换不卸载),结果**写 store 而非局部 state**;就绪翻转 → 引导卸载时只清 interval,末次更新已落 store,无丢失、无 unmounted setState。
- **"去初始化" CTA**:只 `navigate(/overview)` + 关闭 modal,**不在 modal 复刻初始化**;已在概览时关闭 modal 让引导可见即可。
- **触发时 projectId 锚定**:`requireInit()` / CTA 记录**触发当时的 projectId**;防 modal 打开期间切项目跳错。

## 四、核心流程 / 逻辑

```
进入概览 / 点"项目"组入口
        │
        ▼
读接入状态(useProjectOnboardingGate/Actions ← store.onboardingByProject[pid];
未拉过则 ensureOnboarding(pid),模块级 Map 去重)
        │
   ┌────┴────────────┬───────────────────┬────────────────────┐
   │未加载/loading    │error              │ready=false          │ready=true
   ▼                 ▼                   ▼                    ▼
 锁定·"检测中"      锁定·"无法确认"      Sidebar 项目组锁定     Sidebar 正常
 概览不抢渲染        (不 fail-open)      概览=ProjectSetupGuide 概览=数据盘(banner+卡片)
   │                 │                   (整页两步引导)
   └───────┬─────────┴───────────────────┘
           ▼
   点锁定项/被拦 → requireInit() 记录触发 pid → openModal("onboarding-required")
           ▼
   modal: 说明未就绪 + [去概览初始化](→navigate(pid,/overview)+closeModal) / [关闭]
           ▼
   概览引导页:步骤① 复制 ccb 命令执行 → 步骤② 一键 su-init(轮询)
           ▼
   轮询/重检结果写 store → ready=true → Sidebar 解锁 + 概览自动切回数据盘
```

**模拟示例(端到端)**:
1. 新开发者选中项目 X(没跑过 `ccb`)。进概览:`useProjectOnboardingActions("X")` 触发 `ensureOnboarding("X")`,返回 `{ccbRuntimeReady:false, knowledgeBaseReady:false}` → `ready=false`。概览渲染 `<ProjectSetupGuide>`(隐藏指标卡):步骤① 高亮"初始化 CCB 运行时"。同时 Sidebar"项目"组四项显示锁定态。
2. 用户没注意概览,先点了左侧"需求管理" → 不导航,`requireInit()` 记录 pid="X" → 弹 modal。点 [去概览初始化] → `navigate("/projects/X/overview")` + 关闭 modal,落到引导页。
3. 步骤①:点 [复制命令],在项目目录粘贴执行 `ccb` → `ccbRuntimeReady:true`;引导页步骤② 点亮"初始化知识库"。
4. 步骤②:点 [一键初始化知识库] → 投递 `/ccb:su-init`;OverviewPage 持有的轮询(3s/60s)拉到 `knowledgeBaseReady:true`,写回 store。
5. store 中 X 的 `ready` 翻 true → 概览从 `<ProjectSetupGuide>` 切回 `<现概览>`(此时引导卸载,interval 清理,末次状态已在 store);Sidebar 四项解锁。

| 处理规则 | 说明 |
|----------|------|
| 就绪判定 | `ready = value?.ccbRuntimeReady === true && value?.knowledgeBaseReady === true` |
| 未加载 | `value` 空或 `loading` → 锁定 + "检测中";概览不抢先渲染引导/数据盘;不放行、不延迟判定 |
| 失败 | `error` 存在 → 锁定 + "无法确认接入状态,请重新检测";不 fail-open |
| 并发去重 | `ensureOnboarding` 模块级 `Map<pid,Promise>` 合并;沿用既有 30s TTL |
| 轮询卸载安全 | 轮询写 store(非局部 state),owner 在稳定的 OverviewPage;切回数据盘时引导卸载只清 interval |
| 切项目 | modal/CTA 用触发时锚定的 pid |
| 解锁联动 | 轮询/重检写回 store → 订阅组件(Sidebar / 概览)自动响应 |

## 五、测试策略

## 十、迁移影响与风险

- **受影响**:Sidebar、OverviewPage、ProjectOnboardingBanner(拆分)、ui-store / App modal、project-store。
- **打法**:① store onboarding slice + 两 hook(地基)→ ② Sidebar 锁定 + onboarding-required modal → ③ banner 拆分 + ProjectSetupGuide + OverviewPage 切换。banner 拆分与引导页可单独一个提交,降低回归面。
- **回滚 / 恢复**:纯前端,`git revert` 即可;无数据 / schema 变更。

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| banner 拆分回归(初始化动作/轮询被破坏) | 中 | 中 | 逻辑整体搬进 actions hook,语义不变;集成测试覆盖 su-init→就绪→切换 |
| 引导卸载时序(就绪后 guide 卸载) | 中 | 中 | 轮询 owner 在稳定 OverviewPage、写 store 非局部 state;卸载只清 interval |
| 首次加载锁态/引导态闪烁 | 中 | 低 | "检测中"文案 + 概览未加载不抢渲染;沿用 TTL 缓存;不延迟判定 |
| modal 期间切项目跳错 | 低 | 中 | 触发时锚定 pid |
| 命令面板 / 直达 URL 绕过被误当 bug | 低 | 低 | 设计与测试显式声明为产品接受的豁免 |
| 锁定项 a11y 退化 | 低 | 中 | button 非 disabled + 焦点保留 + aria-disabled + 点击解释 |

## 协商与反思

- **协商(slot3_codex)**:需求轮 `job_f4f1a3deafed` 质疑"只拦 Sidebar 会被绕过",促成边界澄清;设计轮 `job_e6fbf23418f3` 就"单一 store 源 + 派生 hook + modal + banner 重构"达成共识并补强(in-flight Map 不进 state、locked button 非 disabled、CTA 只跳概览、触发锚定 pid、测试覆盖豁免)。
- **v1.1 修订说明(用户驱动)**:用户拍板"未初始化时概览整页做初始化引导"。本轮为已完成设计的用户驱动修订,沿用前两轮共识架构;因处自由对话轮(无 active CCB job)、`ccb ask` 无同步模式且 `--callback` 需 active job,未发起第三轮 Codex 协商。修订属前端 UX 范畴(由 Claude 自决),最棘手的"引导卸载时序"已显式推理处理(写 store + 稳定 owner)。建议 task_breakdown / review 阶段补一轮 Codex 复核。
- **反思(4 锚点)**:
  - 同意:单一 store 源;不 fail-open;复用 banner 不复刻;整页引导比"横幅 + 空卡"更贴合根因。
  - 调整:原"概览卡片点击拦截"作废(未就绪不渲染卡片);banner 逻辑抽 hook、轮询 owner 上移到稳定层。
  - 盲点:概览未就绪态的"空指标卡"本就别扭,用户的整页引导提案修正了这点;轮询从 banner 迁出后必须处理卸载时序(写 store 解决)。
  - 接下来:落档后判断进入 task_breakdown(按八的 A/B/C 切块)。
- **用户授权事项**:无命中(无依赖/schema/API/migration/成本/许可证;纯前端重构,均 AI 自决实现细节);产品方向(含本次整页引导)已在需求侧闭环。
