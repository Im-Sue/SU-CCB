# docs 治理与文档驱动架构 — 技术设计

> **版本** v0.2 ｜ **状态** 草稿(待过审) ｜ **最后更新** 2026-05-27
> **对应需求** 关于docs的规范制定(cmpmv55uy7d2673077860d06a)

一句话:把"人能读的文档"变成当家做主的真相源,机器层 `.ccb/` 退到后台只做索引和打杂。

---

## 一、要解决什么(大白话)

1. docs 文档命名/格式没标准、内容散乱。
2. 旧的、过程性的、孤儿文档堆积,该清,只留 v1.0 主线相关。
3. 在 Console 建一个需求,背后散成一堆"机器文件"(需求、设计、拆分、子任务、状态分布在好几个子目录),既看不到一份连贯可读的东西,目录也过深过碎。

根子:**机器要读的东西** 和 **人要读的文档** 搅在一棵树里,而且缺一个给人看的入口。

---

## 二、核心决策:文档驱动翻转

现状已是"文件是真相、数据库(DB)是缓存"。本设计不推翻这点,而是:

- **真相上移**:需求、设计、进度、状态,统一维护在 `docs/01-99` 人读文档里。
- **`.ccb/` 降级**:不再存需求/任务本体,只做索引 + 协调 + 打杂(登记本、门锁、机器草稿、规则)。
- **DB 仍是缓存**:从文档重建,Console 网页读 DB,文档改 → 缓存更新 → 网页变。
- **规则只写一份**:目录有哪些、各放什么、每个命令产物落哪,全声明在 plugin 的一张"目录契约"里,plugin 和 Console 都读它,取代到处硬编码路径。

被拒方案:① 磁盘真扁平(每需求一份文件当唯一真相)—— 会砸建需求/拆分/状态一整套,且用户明确"不是单纯合并成一个文件";② 纯换门面(底层不动)—— 用户要"从根本治理"。

---

## 三、`docs/` 目录结构(人读层 · 当家)

```
docs/
├── 00_项目总览.md      先读这个:项目是啥、当前版本/方向、从哪看起
├── 00_文档地图.md      总览表:每个目录放什么、当前在用的主文档、哪些是旧的/归档的
│                        (这张表就是"孤儿文档"的判定依据)
├── 01_架构设计/        系统怎么搭的(整体/各部分;多项目则各一份 + 系统总架构)
│   └── _模板_架构.md
├── 02_需求设计/        一个需求一份:原话 + 解读 + 设计
│   ├── _模板_需求.md
│   └── <需求名>.md
├── 03_开发计划/        技术设计 + 拆了哪些活 + 各做到哪步(放一份,不再拆多目录)
│   ├── _模板_技术设计.md
│   ├── _模板_开发任务.md
│   └── <需求名>-开发.md
├── 04_模块规格/        每个模块的最终说明书(Console / Plugin / 技能 / 内核)
│   └── _模板_模块规格.md
├── 05_经验沉淀/        踩过的坑、复盘、经验
├── 06_决策记录/        ADR:重要决定的 背景/决策/为什么/影响,一决定一篇
├── 99_归档/            退役旧物,只留"去哪找"的索引
├── README / 安装 / 快速上手 / 依赖    入口说明(保留)
└── tmp/                临时存的 791 份旧文档(本设计定稿后再处理)
```

要点:不分 console/plan/plugin(改成文件名带模块);技术设计与任务清单同目录;每个目录配一份模板;一份《文档地图》当总索引。

## 四、`docs/.ccb/` 目录结构(机器层 · 后台)

最大变化:**不再存需求/任务本体**(都搬到 `docs/`)。只剩:

```
docs/.ccb/
├── 目录契约.yaml    规则总表:目录清单 / 各放什么类型 / 每个命令·技能产物落哪 /
│                     索引入口 / 哪些字段能手改·哪些得走命令 —— 由 plugin 声明,两边共读
├── 索引缓存/        "需求X在哪份文档、状态啥、进度多少"的快速查询表(可从 docs 重建)
├── 流水账.jsonl     谁在何时改了什么(审计 + 出事回溯)
├── 锁/              防止两个 agent 同时改一份文档改乱
├── 拆分草稿/        机器拆活时的中间草稿(给人看的清单投影到 docs/03)
└── 校验规则/        检查文档格式对不对的规则
```

一句话:`.ccb/` 从"仓库"变成"卡片柜 + 登记本 + 门锁"。

## 五、管理流程(从头到尾)

```
你提需求(Console 点"新建" 或 直接说)
    │
    ▼
① 落需求 → docs/02_需求设计/<需求>.md(开头原话,下面 AI 解读)
    ▼
② 技术设计 → docs/03_开发计划/<需求>-开发.md(AI 想方案 → Codex 挑刺 → 取舍问你 → 落文档)
    ▼
③ 拆活 → 还在那份 03 文档里列"要干哪几件事"
    ▼
④ 派工+干活 → 每件活派给执行 agent;进度更新在那份文档
    ▼
⑤ 审查 → 看做得对不对 → 过 / 打回
    ▼
⑥ 归档 → 标记完成,旧的挪进 99_归档 留索引
```

每步产物都落在 `docs/` 里你能读的文档;机器只在后台登记。

---

## 六、机制三条规矩

1. **文档当家**:真相永远在你能读能改的文档里。`.ccb/` 只登记成查询表,Console 显示的是这张表;表坏了从文档重建即可,文档丢不了。文档与机器记录冲突时,**以文档为准**。
2. **该手改的手改,该走命令的走命令**:描述性的(需求干啥、怎么设计、经验)随便敲;关键状态(做完没、到哪步、任务流转)走命令/按钮改;手改把关键状态改坏 → 界面**标红"这儿坏了"**,不偷偷修正,人和 AI 都看得到、能修。
3. **规则只写一份**:目录/类型/产物落点/索引入口全声明在"目录契约",plugin 干活和 Console 建索引都照它;以后改一处两边自动一致。

---

## 七、用户拍板记录(本轮已定)

- **方向**:不被历史决策(ADR-0033 + 三份 spec)捆绑、完全重做;从根本治理;吃透参考项目 prediction_market 的结构/模板/内容再融合(非字面合并)。
- **结构**:`docs/01-99` + README 已认可。
- **真相归属**:文档当家(规矩 1)。
- **可改性**:描述自由改 / 关键状态走命令 / 坏了显式标红(规矩 2)。
- **旧文档处理**:暂不议,791 份已搬 `docs/tmp/`(搬动未删),待本设计与模板定稿后再议。
- **ADR 安置**:决策记录(ADR)单独成目录 `06_决策记录/`(不挂 01 下);ADR 模板与那 30 篇的迁移后议。
- **00 文档与状态**:`项目总览` / `文档地图` 已细化为模板;文档状态简化为三档「生效中 / 历史 / 归档」;无单一"主线",并行需求由 Console 实时显示。

---

## 八、不做项(边界)

- 不把机器层 `.ccb/` 也粗暴拍平(它是协调机制,不是文档)。
- 不在本设计内执行任何旧文档删除(只搬不删,处理另议)。
- 不改 Console 网页"读 DB 显示"这条链(DB 仍是缓存)。
- 旧文档内容不逐篇重写(只按新规命名/归类;新文档按新规)。

---

## 九、迁移影响与风险

**量级**:中大型横切迁移(不是全量重写,但面宽)。受影响:Console 的 indexer / document-parser / 需求重建与编辑创建路径、plugin 路径 helper、`su-flow/su-init/su-plan/su-materialize/su-resume/su-status/su-reconcile` 一串技能、内核 schema/manifest、测试夹具。
**打法**:先把"硬编码路径"收敛成读"目录契约"的统一解析器,再分批把需求/设计/状态的落点迁到 `docs/`。可分批、机械。

| 风险 | 对策 |
|---|---|
| 双真相(文档/.ccb/DB/流水账打架谁赢) | 规矩 1:文档为准,其余重建;不一致时标"无效/部分",不静默修 |
| 人手改坏关键状态 | 走校验:只有合格内容进缓存;不合格界面标红 |
| 迁移破坏旧深链/归档语义 | 迁移前做链接审计 + 留可恢复备份 |

## 十、测试策略(大纲)

- 目录契约解析:plugin 与 Console 读同一份得到一致结果。
- 文档→缓存重建:删掉缓存能从 docs 完整重建。
- 坏文档处理:非法关键状态进"无效"态、界面标红、不污染正常数据。
- 端到端:建需求→设计→拆活→执行→审查→归档全程产物落点正确。

## 十三、合一决策 + plugin 优化清单(v0.3 新增)

### 13.1 实体与文档"合一"(用户拍板)
需求/任务等实体**不再"机器 spec + 人读文档"两份**:**真相就是那一份人读文档**(带 frontmatter 状态),`.ccb` 只索引/协调。旧 entity 模板(`requirement-template` / `epic-spec` / `subtask-spec`)在文档驱动下被人读模板(`_模板_需求` / `_模板_开发任务`)取代;`createRequirement` / `materialize` 等改为产/读人读文档。

### 13.2 索引塌缩(4 → 1 自动索引)
旧 4 个手维护、且**当前无人生成**的索引(`architecture/modules/decisions.yaml` + `.catalog.yaml`)→ **塌缩成 1 个自动索引** `00_文档地图`(indexer 从各文档 frontmatter 投影);`.catalog.yaml` 被取代。

### 13.3 plugin 改动清单
- **claude-md / codex-md 模板**:读取原则改为 启动必读 = `00_项目总览` + `00_文档地图` + 目录契约;按需读 = 按契约定位的具体文档;删 4 个旧 yaml 入口。
- **project-scaffold**:旧 `.ccb` 中心结构 → 新 `docs/00-99` + 指向契约 + init 时拷 `templates/docs/**`。
- **保留(机器层)**:`project-yaml` / `capabilities` / `hooks` / kernel schemas。

### 13.4 链路 / 按需
当前读取靠 CLAUDE.md 散文 + 孤儿索引,不算"链路/按需"。契约 + 自动索引才做到:一个入口 → 按契约精准按需读 → 消费方(skills/indexer)读契约不硬编码。**消费方接线 = Codex 迁移**。

### 13.5 谁动什么
- **现在改(模板/结构)**:建 `docs/00-99` 结构 + 拷模板;改 claude-md / codex-md / scaffold 文字。
- **Codex 迁移(代码)**:indexer 生成自动索引、su-init 拷模板逻辑、路径 helper 读契约、`createRequirement` / `materialize` 改为人读文档(合一)。
