# 需求取消指令闭环 技术设计

> 一句话：把已存在但未闭环的取消链路硬化成「**墓碑先行 · 全步幂等可重入 · capability 唯一状态写入口**」的级联操作，UI 侧补原因输入与终态反馈。
>
> **无独立 status** —— 跟随 `requirement_id`（cmpzllxw…913778）。需求来源：`docs/02_需求设计/当前ui界面和plugin似乎缺少取消需求的指令-913778-需求.md`。

> 经 technical_design 节点产出：双路源码勘察（Console 链路 + plugin/kernel 机制）+ 契约文件亲读核验 + slot2_codex 设计协商（job_9a23122a78bd，结论 O1 采纳+四项修正）。**四处用户拍板待定**：见第十一章，均已按建议默认写入设计，否决任一则回改对应节。

## 一、设计概述

| 项 | 说明 |
|----|------|
| 名称 | 需求取消指令闭环（L1 Console UI + L2 指令逻辑硬化） |
| 核心职责 | 让「取消」成为用户一步发起、即时可见、语义闭环、级联完整、符合状态规范的操作 |
| 设计原则 | ① 墓碑先行（授权门先于不可逆销毁）② 每步幂等+journaled+可重入 ③ 状态写只走 `applyCapabilityOutcome` ④ Console 只触发/投影/清理自身运行态，不写业务真相 ⑤ 非抢占（不强杀 running job） |

## 二、方案与架构

```
用户(详情页 ⋯ 菜单·ghost 样式)
  └─ 确认弹窗(级联影响复述 + reason 输入) ──授权=must_ask_9──┐
       ▼                                                      │
POST /anchor-dispatch  payload {command:"su-cancel", payload:{reason}}
  ├─ gate①: 同需求已有 pending|submitted 的 su-cancel → 409 cancel_in_progress
  ├─ gate②: 投影 status=cancelled 且 command ∉{su-cancel,su-reactivate} → 409
  ├─ supersede: 同需求 scope 的 pending 非 cancel 队列行 → status=superseded
  └─ enqueue → sticky slot 串行队列（不写业务状态，语义不变）
       ▼ (slot agent 接管)
/ccb:su-cancel → lib/requirement-cancel cancelRequirement()
  1 前置: delivered→拒 / cancelled→resume-cleanup 模式
  2 append 授权事件(reason+jobId+idempotency_key)
    → applyCapabilityOutcome(requirement.cancel, mustAskRefs+B evidence, expectedHash)
    → frontmatter status=cancelled 【墓碑】
  3 级联: 非终态 dev_task 逐个 applyCapabilityOutcome(subtask.cancel, expectedHash CAS)
  4 deleteBreakdownDraft（无 draft → skip）
  5 worktree state-aware: ready→discard(force) / merged→cleanup(分支已回流,清理而非丢弃) / 缺失·archived·discarded→skip
  6 append requirement_cancel_cascade_completed 汇总事件 + best-effort POST /scan
       ▼ (md 真相落盘)
Console watcher / 10s reindex → 投影 cancelled
  ├─ reconcile: 释放 SlotBinding（busy 则等 job 完成）+ 兜底清 pending
  └─ UI: 「取消执行中」banner 收敛 → 成功 toast / rejected 事件 → 失败 toast
```

| 关键原则 | 说明 |
|----------|------|
| 墓碑先行 | must_ask_9 授权在 capability outcome 内校验，必须先于 worktree 销毁；status=cancelled 同时挡住并发 finalize/dispatch |
| 全步幂等可重入 | cleanup 任一步失败 → 重派 su-cancel 即恢复（步 2 noop、步 3-5 各自 skip/重做），不引入 cleanup_incomplete 持久状态 |
| CAS 兜底 TOCTOU | guard 在锁外只做语义快速失败；原子性靠 expectedHash（guard 读到 reviewing 后被并发归档为 done 的窗口，由 CAS_CONFLICT→重读重判堵死） |
| 非抢占 | cancel job 在 sticky slot 自然排队后行；running job 后续 finalize/dispatch 被墓碑 guard 拒绝；人工兜底=Slots 已有 cancel-current-job（CcbdClientService.cancel） |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `lib/capability-outcome` | `validateExecutableGuards` 增 cancel/defer 分支 + 两个 guard 实现 | apply 主流程、锁、重试、journal 机制不动 |
| `capability-outcome-policy.yaml` | 新增 `subtask.cancel` policy；cancel/defer policy 挂新 guard | 既有 policy 字段语义不动 |
| `skills/su-cancel/SKILL.md` | 重写为精确 lib 合约（对齐 su-archive 工程化程度） | breakdown-draft delete / worktree discard 既有合约保留并入编排 |
| `lib/breakdown-draft`、`lib/worktree` | 仅复用；缺失容忍由新 lib 包一层探测 | 不改其内部语义 |
| su-oriel anchor-dispatch 路由 | 加双 gate + supersede | enqueue 不写业务状态语义不动（task.routes.spec.ts:299 仍成立） |
| su-oriel 投影/reconcile | 投影 cancelled 时释放 SlotBinding+兜底清 pending | Console 不写业务真相边界不动 |
| kernel yaml（state-schema/lifecycle/transition-table） | 枚举漂移修复 + from-list 扩展 | 历史 ADR/terminal 记录/journal 不 sweep |

## 三、关键决策与取舍

- **墓碑先行**：选「先 capability 写 cancelled、后级联清理」；否决现 SKILL.md 的 discard-first——授权门在 capability outcome 内校验，必须先于不可逆销毁；且「cancelled+残留资产可重入」优于「销毁了却还活着」。
- **per-task capability 级联**：选逐任务 `applyCapabilityOutcome(subtask.cancel)`+expectedHash；否决批量原语（O2，新事务语义复杂度不值）与 `writeTaskState` 直写（重蹈绕过 capability 框架覆辙）。
- **非抢占**：否决自动 force-release/杀 job（O3，孤儿 job+半写状态风险，收益仅提前生效）；复用既有 Slots `cancel-current-job` 作人工兜底，弹窗文案指引。
- **cancelled 后白名单而非统一 409**：`su-cancel`（cleanup 重入）与 `su-reactivate`（复活）放行，其余 409——统一 409 会堵死失败恢复与复活（协商修正点）。
- **enqueue 双 gate**：只看投影 status 挡不住「取消排队中」窗口，必须同时查在途 su-cancel job（协商修正点）。
- **授权双保险**：append 授权事件（带 `idempotency_key`）后，apply 同时传 `mustAskRefs:["must_ask_9"]` 与 kind=B `journal_event_exists` evidence——只传 mustAskRefs 无可追溯事件，只传 evidence 依赖形状校验，双传最稳（协商修正点）。
- **registry 对齐**：`capabilities/global.yaml` 现仅注册 `requirement.promote`，policy yaml 却有 cancel/defer/finalize 四个——新增 `subtask.cancel` 时一并补注册，不再扩大 registry/policy 漂移（协商发现）。
- **defer 顺带 terminal guard**：同一改动点+同一注册批次，仅 guard 不扩级联/UI；符合彻底修复原则（待拍板③）。
- **done 子任务保留 done**：已交付工作不可被级联覆盖；standalone 取消 done 任务 → guard 拒绝。
- **worktree 步 state-aware（拆分轮修正）**：`merged` 态分支已回流 target，cancel 该走 `cleanupRequirementWorktree` 而非 force discard（否决"一律 discard"——会误删已合并分支或撞 260604 PR1 的 `ready→discarded` guard）；由此 **913778-pr2 显式依赖 cmpworktreearchive260604 PR1 先合并**（提供 cleanup/split 语义；排期被迫并行时 fallback：pr2 先落 capability/kernel/cancel core，worktree merged 分支延后补，不自造兼容 helper）。
- **退休 disabled `requirement.analyze:analyzed` policy（拆分轮发现，待拍板⑤）**：不退休则枚举 sweep 永久豁免一个死条目（Console generated 镜像会持续生成 `analyzed`）；历史 journal append-only 不受影响。

## 四、核心流程 / 逻辑

```
cancelRequirement({projectRoot, requirementId, reason, sourceActor, dispatchRef})
  ├─ resolve requirement（resolver 定位 md）→ 读 frontmatter {status, hash}
  ├─ status=delivered ──► return REJECTED(delivered_terminal)
  ├─ status=cancelled ──► resumeMode=true（跳过授权+墓碑，直接清理）
  ├─ [!resumeMode] appendEvent(user_cancel_authorized {reason, dispatchRef, idempotency_key})
  │                applyCapabilityOutcome(requirement.cancel, expectedHash=hash,
  │                  mustAskRefs:[must_ask_9], evidence:[A file_exists, B journal_event_exists])
  ├─ listDevTasks(requirementId) → filter status∉{done,cancelled} → 逐个:
  │    读 task state{hash} → applyCapabilityOutcome(subtask.cancel, expectedHash)
  │    └─ CAS_CONFLICT → 重读: 已终态→skip记录, 否则重试一次→仍冲突则收入 issues
  ├─ readBreakdownDraft 探测 → 有则 deleteBreakdownDraft / 无则 skip
  ├─ worktree state-aware 处置: 读 runtime state → ready→discardRequirementWorktree(force)
  │    / merged→cleanupRequirementWorktree(260604 PR1 提供) / missing|archived|discarded→skip
  ├─ appendEvent(requirement_cancel_cascade_completed {reason, cancelled_task_ids,
  │    skipped:{draft,worktree,...}, resumed, issues})
  └─ return {ok, noop?, resumed?, steps[], issues[]}   // 任一步硬失败→中断+结构化上报，重派即重入
```

| 处理规则 | 说明 |
|----------|------|
| 幂等 | 步 2 guard noop（已 cancelled）；步 3 终态 skip；步 4/5 缺失 skip；事件 idempotency_key 防重 |
| 失败恢复 | 中断点之前的效果保留；用户重派 su-cancel（白名单放行）→ resumeMode 续清理 |
| 并发 | requirement md 与 dev_task 写均走 `applyCapabilityOutcome` 内置 withFileLock+CAS；guard 语义检查不承担原子性 |
| 可观测 | 每步落 journal；汇总事件含 issues；skill 末步 best-effort `POST /scan`（WSL2 watcher 兜底，Console 缺席容忍） |
| Console reconcile | 投影 cancelled → SlotBinding 非 busy 即释放，busy 标记待释放等 job 完成；pending 队列行兜底 supersede |

## 十、迁移影响与风险

- **受影响**：取消全链路（UI→路由→队列→skill→lib→kernel yaml→投影）；枚举修复波及 kernel 引用面
- **打法**：3 PR 串行（pr1 UI=Claude → pr2 plugin/kernel 硬化=Codex → pr3 Console server 运行态+E2E 冒烟=Codex；pr3 由原 pr2 按 repo 边界拆出）；**pr2 跨需求依赖 cmpworktreearchive260604 PR1 先合并**（worktree split 语义）；两侧 generated policy 重生分别纳入 pr2/pr3 审查
- **回滚**：git revert 即可；枚举修复回滚需连带回退生成产物；无数据迁移故无数据回滚面

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 投影延迟（capability 写不触发 file_written，WSL2 watcher 漏） | 中 | toast/slot 释放晚 ~10s | 详情页 10s reindex 已兜底 + skill 末步 POST /scan |
| 枚举 sweep 漏引用 | 中 | lint/运行时报未知状态 | 全仓 grep 清单化 + lint_manifest.py + 生成器重生 diff 审查 |
| 级联半完成（agent 中断） | 低 | cancelled 但资产残留 | 全步幂等 + 白名单放行重派重入 + 汇总事件含 issues 可审计 |
| 取消排队期间新工作进入 | 低 | 浪费执行 | enqueue 双 gate + supersede + 墓碑 guard 三层 |
| running job 写一半被取消波及 | 低 | 状态脏写 | 非抢占设计规避；其后续 finalize 被墓碑 guard 拒 |
| 260604 PR1 未先合，pr2 worktree 语义悬空 | 中 | pr2 阻塞或误用旧 API | 显式跨需求依赖；fallback=pr2 先落 core、merged 分支延后补 |

## 十一、用户拍板项（已按建议默认写入，否决则回改）

| # | 事项 | 建议 | 业务语言说明 |
|---|------|------|--------------|
| ① | `deferred → cancelled` 直达 | **允许**（Codex 同票） | 暂缓≠交付完成；想彻底放弃一个已暂缓的需求时，不必先"复活"再取消走仪式 |
| ② | 新增 `subtask.cancel` capability + global.yaml 补注册 cancel/defer/finalize | **批准** | 取消子任务获得与归档同级的防护；顺带消除 registry 与 policy 两本账不一致的存量漂移 |
| ③ | `requirement.defer` 顺带 terminal guard | **批准** | 暂缓与取消同属大状态操作，补同款防误写护栏（仅护栏，不扩暂缓功能） |
| ④ | 非抢占语义 + cancelled 白名单 409 | **确认** | 取消不强杀正在干活的 agent（避免孤儿任务）；已取消需求只接受"重试取消/复活"两种指令 |
| ⑤ | 退休 disabled `requirement.analyze:analyzed` policy | **批准**（Codex 发现） | 一个已停用的死条目会让"清理旧状态命名"永远清不干净（生成产物持续带 `analyzed`）；删除不影响历史记录 |
