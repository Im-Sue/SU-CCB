## 二、方案与架构

```
OverviewPage                                   App.tsx（全局常驻）
┌──────────────────────────────┐    ┌────────────────────────────────────┐
│ ProjectOnboardingBanner      │    │ MainTerminalLauncher（M 入口）      │
│  确认 Modal                  │    │  open: 本地 state（属主不变）       │
│  handleConfirmInit 成功:     │    │  消费 effect:                       │
│   1. setInitJob(轮询,既有)   │    │   request.projectId                 │
│   2. setConfirmOpen(false)   │    │     === selectedProjectId ?         │
│   3. requestOpenMainTerminal │    │    是→ loadMainState+setOpen(true)  │
│      (projectId) ───┐        │    │    否→ 丢弃                         │
└─────────────────────┼────────┘    │   两路均 clear request              │
                      ▼             └──────────────┬─────────────────────┘
        ui-store（zustand，新增独立字段）           ▼
        mainTerminalOpenRequest:            Modal(xl，焦点=关闭按钮)
          { projectId } | null               └─ MainTerminalPanel
        requestOpenMainTerminal(id)              ├─ descriptor 就绪 → claude pane 实时输出
        clearMainTerminalOpenRequest()           └─ 失败/空 → fallback + [重试]（新增）
```

| 关键原则 | 说明 |
|----------|------|
| 请求-消费单向通道 | banner 只表达「请求打开」，不持有/操纵 launcher 的 open 状态；消费后立即 clear，无悬挂状态 |
| 不混入 `modalOpen/modalType` | 既有机制是互斥单 modal 语义（create-project 等），混入会与终端弹窗互踢；独立字段彻底隔离 |
| projectId 防御校验 | 打开请求携带 projectId，launcher 仅在等于当前 `selectedProjectId` 时响应——防御未来 banner 复用到非选中项目上下文（需求文档既定） |
| 焦点安全靠结构而非时机 | launcher 消费在 useEffect（commit 后），`setOpen(true)` 必然落在确认 Modal unmount 之后的新 commit——任意时刻至多一个 Modal open，单 focus trap；焦点由 Modal 默认行为落在关闭按钮，不触 xterm |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `ui-store.ts` | 新增 1 字段 + 2 actions | `modalOpen/modalType` 互斥机制、toast、其余字段全不动 |
| `ProjectOnboardingBanner.tsx` | `handleConfirmInit` 成功路径加发请求 | 轮询/60s 超时/成功失败 toast/确认弹窗文案全不动 |
| `MainTerminalLauncher.tsx` | 新增消费 effect（声明在「项目切换关闭」effect 之后） | open 属主、M 按钮入口、项目切换强制关闭行为不动 |
| `MainTerminalPanel.tsx` | fallback 态加「重试」按钮（`resolverEpoch` 触发重新 fetch） | loading/ready 状态语义、pane tabs、默认 claude pane 不动 |
| `Modal.tsx` / `SlotTerminalSurface.tsx` | 零改动 | 默认聚焦关闭按钮 / 无自动 focus xterm 为既有行为，测试加断言防回归 |
| server | 零改动 | —— |

## 三、关键决策与取舍

- **打开通道：选 A（ui-store 专用 open-request），否决 B（banner 自渲染第二个 Modal+Panel）——双实例/双 websocket/双 focus trap，违背需求「同一个弹窗」语义；否决 C（launcher open 状态整体提升 ui-store）——banner 只需要「请求打开」一个动作，整体提升把 loadMainState/项目切换关闭逻辑全拖进 store，侵入面超出需要。**（Claude/Codex 两轮均一致）
- **重试形态：仅显式「重试」按钮，否决「按钮+自动轻量重试」**——Codex 指出自动重试在 main 会话真不存在时会制造「好像正在恢复」的误导；显式按钮已满足需求验收口径「可自行恢复」，复杂度最低；实测 race 高频时再加自动重试（YAGNI）。
- **时序论证修正（协商实质分歧落点）**：我初版以「React 18 同帧批处理」论证无双 Modal 窗口，Codex 反对将其作为设计依据（依赖实现细节）。重新推导后的正确论证：launcher 消费在 useEffect（commit 后执行），`setOpen(true)` 调度的是下一次 commit——「确认 Modal 先 unmount、main Modal 后 mount」由消费侧异步结构保证，与批处理无关。Codex 建议的 banner 侧 `pendingOpenProjectId` + effect 推迟发请求属双重保险，**不采纳**（多一个状态与 effect 而结构已保证顺序），改以集成断言锁住行为（见五）。
- **重试实现：`resolverEpoch` state 纳入 descriptor fetch 的 useEffect 依赖**，重试=epoch+1 → 复用既有 loading/fallback/ready 状态机，不引入第二条 fetch 路径。

**Claude 4 锚点反思**（协商 job_71df9990d877 后）：

- **我同意的**：放弃同帧批处理表述——它把结论建立在 React 实现细节上，审计者无法从代码结构直接验证；自动重试的「假恢复」误导风险真实存在；测试需补集成断言与 store reset（request 泄漏会污染后续用例）。
- **我不同意的**：banner 侧 pending effect 机制——消费侧 useEffect 已天然保证 commit 顺序，发请求的时机（同 handler 内 vs 下一 commit）不影响双 Modal 窗口的不存在性；加 pending 状态是为错误的论证买保险，修正论证后保险本身多余。
- **我的盲点**：初版时序论证错误但结论碰巧成立——Codex 质疑后我才重新推导出真正的保证机制（消费异步性）；「论证错误+结论正确」比「结论错误」更危险，因为它会在后续变更中静默失效；store reset 防测试泄漏我也没扫到。
- **接下来**：落盘本设计 → step=design 完成自然停 → 等 breakdown 投递（预判单子任务，Claude 亲自实施 web UI）。

**必问扫描（12 类，技术设计常命中项）**：3 依赖不命中（零新依赖，zustand/xterm 均既有）；4 公共 API/schema 不命中（纯 web 内部状态，server 零改动）；2 migration 不命中（无数据面）；11 成本不命中；12 许可证/合规不命中；8 UX 不命中（重试按钮文案/位置为低风险实现细节）。**全部不命中 → 本设计无用户授权事项，实现细节全部 AI 自决**；唯一保留的用户权利项在需求文档（若未来把「自动弹出」降级为 toast CTA 须用户拍板）。

**sc 使用记录**：`/sc:design`、`/sc:analyze` 以逐文件源码核验替代覆盖（ui-store/Modal/Launcher/Panel/Surface/App/OverviewPage 七处装配与行为事实全部直读验证，替代其架构草图与影响面分析目标）；`/sc:research` 不适用——无技术选型（状态机制 zustand 既定、无新领域）；`/sc:business-panel` 替代覆盖——无业务/成本/合规面变化（见必问扫描），需求层已做同维度扫描。

## 四、核心流程 / 逻辑

```
[确认 Modal] 点「确认初始化」
   → initProjectKnowledgeBase(projectId)
       ├─ reject → addToast(error)；确认 Modal 保持打开（现状不变）
       └─ resolve(jobId)
           1. setInitJob(...)                    // banner 轮询启动（既有逻辑）
           2. setConfirmOpen(false)              // commit N：确认 Modal unmount，
           3. requestOpenMainTerminal(projectId) //   cleanup 恢复 previousFocus
                    ↓ launcher 消费 effect（commit N 之后执行）
           request.projectId === selectedProjectId ?
               ├─ 是 → loadMainState() + setOpen(true) → clearRequest()
               │        ↓ commit N+1：main Modal mount，焦点=关闭按钮
               │   MainTerminalPanel: fetchTerminalDescriptor(agentGroup:main)
               │      ├─ panes 就绪 → SlotTerminalSurface（claude pane 实时输出）
               │      └─ 失败/空  → fallback 文案 + [重试]（resolverEpoch+1 → 重新 fetch）
               └─ 否 → clearRequest()（丢弃，防跨项目误弹）
```

| 处理规则 | 说明 |
|----------|------|
| 时序/焦点安全 | 消费在 useEffect ⇒ `setOpen(true)` 必然产生新 commit，晚于确认 Modal unmount；任意时刻至多一个 Modal open（单 focus trap）；焦点链：确认按钮（disabled 时静默落 body）→ main Modal 关闭按钮，全程不触 xterm |
| 重复触发 | request 每次新对象引用 ⇒ 消费 effect 必触发；ready 态「重新初始化知识库」同 handler 自然同行为 |
| 幂等/StrictMode | 消费后立即 clear（request=null 时 effect 为 no-op）；StrictMode 双跑最多重复 loadMainState（只读 fetch）与幂等 setOpen(true) |
| 失败链路 | API reject 不发 request（不弹）；descriptor 失败有显式重试入口，复用既有状态机 |
| 可观测性 | 不新增日志面——banner 既有 jobId 文案 + toast 已覆盖；终端输出本身即过程可见性 |

## 五、测试策略

- [ ] banner spec：确认成功 → ui-store 收到 `{projectId}` 请求；`initProjectKnowledgeBase` reject → request 保持 null
- [ ] launcher spec：注入 request（匹配 projectId）→ 「main agent 组终端」dialog 打开且 request 被 clear；注入不匹配 projectId → 不打开且 request 被 clear
- [ ] **集成断言**（Codex 补充，锁焦点行为）：确认成功后页面无「初始化知识库」dialog、有「main agent 组终端」dialog，且 `document.activeElement` 为 main Modal 关闭按钮
- [ ] panel 重试：descriptor fetch 失败 → fallback 出现「重试」按钮；点击 → 重新 fetch 成功后渲染 surface（优先扩展既有 spec 组织，必要时新增 `MainTerminalPanel.spec.tsx`）
- [ ] 测试卫生：涉及 ui-store 的 spec 在 afterEach reset 新增字段，防 request 泄漏污染后续用例
- 轻量原则（用户既定偏好）：不穷举 pane 切换/websocket/StrictMode 矩阵，主路径 + 上述关键防护即止

## 十、迁移影响与风险

- **受影响**：仅上述 web 前端 4 文件 + 测试；手动 M 入口路径共享 launcher/panel 改动（消费 effect 对手动路径无感，重试按钮对手动路径是正向增强）。
- **打法**：单子任务一次落地（改动面小且强内聚，拆分无收益）。
- **回滚 / 恢复**：纯前端、无数据/接口面，`git revert` 单提交即可完全撤回。

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 焦点时序竞态（双 focus trap） | 低 | 中 | 消费侧异步结构保证顺序 + 集成断言锁 activeElement |
| 手动 M 路径回归 | 低 | 中 | ready 路径零改动；fallback 仅增量 UI；launcher spec 回归 |
| ui-store request 泄漏污染测试 | 中 | 低 | afterEach reset 新增字段 |
| 未来 banner 复用到非选中项目上下文 | 低 | 低 | projectId 校验丢弃为预期防御（需求文档已记） |
