# 需求进度问题 — 技术设计

> 一句话:新增 `requirement.promote:planning` capability-outcome,由 plugin agent 写 canonical;**采用 D1(本期)** —— 在 su-flow planning 入口 promote + `applyRequirementAnalysis` 成功后幂等 affirm,forward-only 只允许 `drafting→planning`。Console enqueue promotion(D2,覆盖无派工的手动 bind)降为**后续跟进**。｜ 最后更新: 2026-06-02
>
> **无独立 status** —— 跟随需求 `cmpvdh2mj3f9c0e75c576e83d`。做什么/为什么见同名《需求设计》。关联 **ADR-0034**(capability-outcome canonical)、**ADR-0037**(文档驱动真相源)、**ADR-0041**(本次契约变更,见下)。

## 一、概述

需求 canonical `status`(真相在需求 md frontmatter)目前在「绑定 slot」「执行需求分析」两条路径都不流转,需求恒留 `drafting`→看板「待处理」。本设计让任一推进事件后 canonical 自动晋升 `drafting→planning`(看板「推进中」)。

核心约束(ADR-0034/0037):**Console 不写 canonical、不调 plugin lib**;canonical 只能由 plugin lib(`applyCapabilityOutcome` + `applyRequirementMarkdownEffects`)在 agent session 内写。Console 只能作为**触发器**enqueue plugin-side command。

## 二、方案(选型:D1 本期交付,D2 后续跟进)

经 Codex 设计协商确认,`planning-anchor/start` 只 bind slot + 写 `planningRuntimeState`,**不 enqueue agent job**;存在「绑定但无 su-flow 运行」的真实路径(手动 `/bind-slot`、直接 `planning-anchor/start`、startup recovery)。**用户拍板本期先做更轻的 D1**:

- **D1(本期)**:晋升做成单一 `requirement.promote:planning` capability-outcome,在 plugin agent 内两处幂等触发:
  - (a) **su-flow planning 入口**(requirement_analysis / technical_design 节点进入时)promote 一次 —— 覆盖「主按钮绑定→派 planning anchor→su-flow 运行」的主流程绑定路径。
  - (b) **`applyRequirementAnalysis` 成功后** affirm 一次(已 planning 则 no-op)—— 覆盖「执行分析」路径,独立满足。
- **D2(后续跟进,本期不做)**:Console 绑定后经共享 helper enqueue 轻量 plugin promotion command,覆盖**无派工的纯手动 bind / startup recovery**。这些路径在 D1 下暂不晋升,作为已知缺口留待后续需求。
- **拒绝 D3**(仅分析触发):会漏主流程绑定路径。

两触发共用同一 outcome,天然幂等收敛。

> **D1 已知覆盖缺口**:未经 su-flow agent 运行的绑定(手动 `/bind-slot`、startup recovery)本期不晋升,需用户后续触发分析或等 D2。

## 三、关键决策

| 决策 | 结论 | 依据 |
|---|---|---|
| 目标态 | `planning`(非 delivering / 已废弃 analyzed) | 用户拍板;delivering 留给子任务执行后 rollup |
| 解绑 slot | 不回退 | 用户拍板;晋升单向语义 |
| canonical 写入方 | 仅 plugin agent;Console 只 enqueue | ADR-0034/0037 |
| 触发组合 | 本期 D1(su-flow 入口 + 分析 affirm);D2 后续 | 用户拍板先轻量交付;无派工 bind 缺口留待 D2 |
| 是否新 ADR | 是(ADR-0041) | 改"绑定不晋升"受测试守护契约,属契约变更非纯补丁 |

## 四、核心流程

```
[主流程绑定路径 — D1(a)]
  用户点主按钮 → planning-anchor/start 绑 slot + 标 runtime → anchor-dispatch 派 planning anchor
    → plugin agent 运行 su-flow → planning 节点入口 promote
      → applyCapabilityOutcome(requirement.promote, planning)
      → forward-only guard 校验 → applyRequirementMarkdownEffects 写 md status=planning(CAS)
      → EventJournal 追加 → watcher/scan → Console 投影 rollup 收敛 → 看板「推进中」

[执行分析路径 — D1(b) affirm]
  su-flow analysis 节点 → applyRequirementAnalysis 成功
    → 同 session 调 applyCapabilityOutcome(requirement.promote, planning)（幂等 affirm）

[无派工 bind(手动 /bind-slot、startup recovery)] — 本期不晋升,留待 D2
```

## 五、capability-outcome policy 设计

新增 policy(仿 disabled `requirement.analyze:analyzed` 模板,但 enabled):

```yaml
- policy_id: requirement.promote:planning:requirement
  capability_id: requirement.promote     # 新 capability,需同步 capability registry
  outcome_type: planning
  subject_type: requirement
  write_target: requirement_md
  state_effects:
    status: set:planning
  evidence_required:
    mode: all
    items:
      - kind: A
        source: requirement_md
        check_id: hash_matches            # 用 CAS hash,非 schema_valid（见六）
  must_ask_refs: []
  guards:
    - no_self_referential_event
    - requirement_promote_forward_only    # 新增专用 guard（见七）
```

## 六、evidence 决策

- **不**用 `schema_valid`:`requirement-md-frontmatter` schema 要求 `analysis_input_hash/analysis_applied_at`,planning 入口(未分析)时缺失会失败;且 schema_valid 不能证明 forward-only。
- 采用 `hash_matches`(`file_exists + expectedHash/base_hash` CAS),证明 AI 读到的 md 未被并发改动。
- **不**把 slot binding 当 policy evidence:plugin evidence registry 读 `docs/.ccb/events/journal.jsonl`,看不到 Console DB 的 `slot_bound`,强行要求会引入新跨层 evidence check。

## 七、forward-only guard 设计

新增 executable guard `requirement_promote_forward_only`(不复用 `requirement_not_cancelled_or_deferred`,后者放过 `delivering/delivered` 会导致降级):

- 要求 `subjectRef.canonical_path` + `expectedHash/base_hash`。
- 读当前 md status:`drafting → planning` 允许;`planning` 幂等 no-op;`delivering|delivered|deferred|cancelled` 拒绝/no-op,不覆盖。
- 不破坏 anchor.routes.ts:356 planning anchor 启动门控(`["drafting","planning"]` 仍允许)。

## 九、迁移影响与风险

- **测试契约变更**:`anchor-requirement-dispatch.routes.spec.ts:139` 等断言「绑定后仍 drafting」需更新;补 forward-only / 幂等用例。
- **重复入队幂等**:promote command 需幂等 key(如 `promote-planning:<requirementId>`),避免重复 dispatch。
- **投影延迟**:md 写完到 UI 收敛依赖 watcher/scan,可能短暂未收敛;必要时触发 scan。
- **reanalyze 降级风险**:`requirement.reanalyze` 允许 `delivering/deferred`,forward-only guard 必须覆盖防误降级。
- **新 capability 边界**:`requirement.promote` 需同步 capability registry,或在 ADR 中明确 policy 扩展边界。

## 十、测试策略

- 单测:forward-only guard 全状态矩阵(drafting→planning / planning 幂等 / delivering|delivered|deferred|cancelled 拒绝)。
- 集成:绑定 slot(三条 bind 路径)后 canonical 晋升;分析成功后 affirm;重复触发幂等。
- 回归:更新并保留 planning anchor 启动门控相关断言。
