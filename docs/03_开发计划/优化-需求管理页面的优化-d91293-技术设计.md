# 优化：需求管理页面的优化 技术设计

> 一句话：在 su-oriel/web 需求看板把「推进中」拆成「规划中 / 推进中」、「已归档」改名「已搁置」、列内容区限高滚动；纯前端分组 / 标题 / 布局调整。｜ 最后更新：2026-06-08
>
> **无独立 status** —— 跟随 `requirement_id` cmq4szijx6b73983193d91293 指向的需求。

## 一、设计概述

**目标对齐**：需求看板现在把需求按生命周期粗分成 4 个块,有三处不顺手——「推进中」块混了"规划中"和"推进中"两种状态(卡片 badge 早已分开,只有列没分);每个块没有高度上限,需求一多就被无限撑高、整页越拖越长;「已归档」这个名字装的其实是"已暂缓 + 已取消",叫"已搁置"更贴切。本设计把看板列从 4 拆到 5、给列内容区加限高滚动、修正标题用词,并同步修掉首页"活跃需求"统计在拆桶后会漏算"规划中"的隐患。全程只动 su-oriel/web 展示层,不碰需求状态枚举、DB、接口或 kernel。

| 项 | 说明 |
|----|------|
| 名称 | 需求看板分组优化(RequirementsPage) |
| 核心职责 | status → 看板列 的分桶 / 标签 / 布局投影 |
| 设计原则 | 单一分桶真相源(ui-mapping)、内部 key 稳定只改可见 label、改动局部不外溢任务看板 / 治理 |
| 需求来源 | `docs/02_需求设计/优化-需求管理页面的优化-d91293-需求.md` |
| 覆盖范围 | 看板列定义、status→列映射、列标题、列内滚动 CSS、首页活跃口径连带、相关测试 |
| 不覆盖 | 需求 status 枚举 / DB / 接口 / kernel、任务看板、文档治理"历史"分组、卡片 badge 文案、点击跳转、数据 / 接口 / 依赖变更(均无) |

## 二、方案与架构

分桶真相源在 `ui-mapping.ts`,看板与首页都复用它,改这一处即同源生效:

```
Requirement.status (canonical · 不动)
   │
   ▼
classifyRequirementTab(status)  ── ui-mapping.ts(本次:4 桶 → 5 桶)
   │   drafting   → pending
   │   planning   → planning     ← 新增独立桶
   │   delivering → delivering   ← 原 inProgress 窄化并重命名
   │   delivered  → delivered
   │   deferred|cancelled → archived(label 改"已搁置")
   │   未知       → archived(安全兜底)
   ├──────────────────────────────┬────────────────────────────┐
   ▼                              ▼                            ▼
REQUIREMENT_BOARD_COLUMNS      OverviewPage 活跃口径         (无第三消费者)
看板 5 列 · RequirementsPage    首页统计 · 本次须纳入 planning
```

| 关键原则 | 说明 |
|----------|------|
| 单一分桶真相源 | 看板列与首页统计都走 `classifyRequirementTab`,不各自定义,避免漂移 |
| 内部 key 稳定 | 只重命名语义已变的 `inProgress→delivering`;`archived` key 不动,仅改可见 label |
| 改动不外溢 | "已归档 / archived" 在任务看板、文档治理、归档节点另有语义,本次一律不碰 |

**与现有系统的关系 / 边界**:

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `lib/ui-mapping.ts` | 改 `RequirementTabKey` / `REQUIREMENT_TAB_LABELS` / `classifyRequirementTab`;`getRequirementAction` 的 archived label 同改"已搁置";新增活跃谓词 | badge 文案、任务相关函数 |
| `lib/requirement-board-config.ts` | 列定义 4→5、buckets、JSDoc 注释 | 排序 / 投影逻辑结构 |
| `pages/overview/OverviewPage.tsx` | 活跃口径纳入 planning + delivering | 其它统计 |
| `pages/requirements/RequirementsPage.module.css` | 列内容区限高 + 滚动 | 卡片样式 |
| 任务看板 / 治理 / 归档节点 | **不动** | 其各自的"已归档 / archive"语义与快照 |

## 三、关键决策与取舍

- **分桶 key 命名**:选 `pending | planning | delivering | delivered | archived`。`inProgress` 旧义 = planning + delivering,本次窄化为仅 delivering,旧名会误导后人,故重命名为 `delivering`,使 key 与 status 枚举名对齐;没选"保留 inProgress"(留 `delivering→inProgress` 语义错配)。`archived` 保留 key 只改 label——它和 pending / delivered 一样是英文语义桶名,且要继续承担未知 status 的安全兜底;没选"改名 shelved"(无用户可见收益、纯 churn)。Codex 协商确认 `archived` 样式走 `column.muted → data-archived`,不绑 key 字符串,改 label 不影响降权样式。
- **列内滚动**:选"列内容区局部限高 `max-height: calc(100vh - N)` + `overflow-y:auto`",与仓内既有 `DocumentsPage.module.css`(`calc(100vh - 160px)`)、`BreakdownReviewEmbedded.module.css`(`calc(100vh - 240px)`)同一 calc idiom;没选"整页 / 看板高度链重构"(更正统但要改 app-shell 高度链、影响面大,超出本需求);没选"固定 px"(不随视口自适应)。
- **首页活跃口径连带**:把"是否活跃"判定从 `OverviewPage` 内联散写抽成可单测的小谓词 `isActiveRequirementTab`(含 pending / planning / delivering),直接给最关键的静默回归(拆桶后漏算 planning)上单测保护——这是 Codex 点出的本设计头号风险口,当前 Overview 无任何测试覆盖。
- **archived action label**:`getRequirementAction(deferred/cancelled)` 旧返回 label "已归档"(需求页不渲染但测试命中),同步改"已搁置",消除残留旧词。

## 四、核心流程 / 逻辑

```
渲染需求看板:
requirements[] ──map──▶ classifyRequirementTab(status) ──▶ buckets{5 桶}
        ──REQUIREMENT_BOARD_COLUMNS(顺序)──▶ 5 列, 列内按 createdAt 倒序
        ──▶ 列内容区超过 max-height 时纵向滚动(整页不被撑长)

首页活跃统计:
requirements[] ──▶ classifyRequirementTab ──▶ isActiveRequirementTab(tab)?
        ──▶ active 计数(pending + planning + delivering)
```

**模拟示例**:一条 `status=planning` 的需求 → `classifyRequirementTab` 归 `planning` 桶 → 渲染进第 2 列「规划中」;同一条在首页 `isActiveRequirementTab("planning")=true` → 计入"活跃需求"(修复前它落 inProgress 桶也算活跃,修复后仍算活跃,但看板列展示更细、首页口径有测试兜底)。反例:`status=cancelled` → `archived` 桶 → 第 5 列「已搁置」(muted 灰显),首页不计活跃。

| 处理规则 | 说明 |
|----------|------|
| 列顺序 | 固定 pending → planning → delivering → delivered → archived,与 status 生命周期同序 |
| 未知 status | 兜底归 archived,不让数据从 UI 消失 |
| 列内滚动 | 仅 `.columnBody` 滚动,列头固定;`scrollbar-gutter: stable` 防滚动条出现时布局抖动 |

## 十、迁移影响与风险

- **受影响**:仅 su-oriel/web 需求看板 + 首页活跃统计 + 上述测试。无 DB / 接口 / 数据迁移。
- **打法**:一处分桶真相源(ui-mapping)+ 看板列定义 + 首页连带一并改并同测,避免分桶语义漂移。
- **回滚**:纯前端,`git revert` 即可;无数据 / schema 副作用。

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 拆桶后首页漏算 planning 活跃 | 中 | 规划中需求从首页"活跃"静默消失 | 抽 `isActiveRequirementTab` 谓词 + 补 Overview 单测 |
| 误改任务看板 / 治理的"已归档 / archived" | 中 | 误伤任务看板快照 / 归档语义 | 仅改需求相关函数 label,内部 key 不动,任务看板"已归档"不碰 |
| 先 `-u` 快照掩盖真实回归 | 中 | 回归被快照吞掉 | 先实现 + 非快照断言 → 确认只剩快照差 → 再 -u |
| CSS 在 happy-dom 测不到滚动 | 高 | 滚动 / 窄屏视觉问题漏网 | 浏览器 / Playwright 桌面 + 窄屏视觉验证 |
| 5 列窄屏更挤 | 低 | auto-fit 列变窄 / 换行 | 沿用现有 `minmax(180px,1fr)` 响应式 + 窄屏解除列限高 |
