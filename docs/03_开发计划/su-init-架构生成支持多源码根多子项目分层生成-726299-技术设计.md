## 二、方案与架构

```
/ccb:su-init (agent 层 SKILL)
  │
  ├─① initProjectScaffold (lib · 确定性 · 无 LLM)
  │     ├─ 既有三步脚手架 —— 不动
  │     └─ detectArchitectureCandidates()            ← 替换旧单数 gate
  │           └─ summary.architectureCandidates = {
  │                mode, reason, candidates[], overviewTargetPath,
  │                overviewExisting, excluded[], scopeConflicts[],
  │                existingArchitectureDocs, capLimit }
  │
  ├─② init.mjs stdout 同步打印 architectureCandidates
  │
  └─③ agent 按 mode 分支:
        ├─ skip          → 回执 reason(no_source / architecture_exists)
        ├─ single        → 现行单份路径(证据门槛→渲染→wx 写)+ 新增 scope frontmatter
        ├─ layered       → per-candidate 证据门槛→子架构×N(逐个 wx)
        │                   → 总架构(输入=实际写成功清单,固定渲染 profile)
        └─ overview_only → 仅总架构 + 全部候选清单 + 点名补生成提示
```

| 关键原则 | 为什么 |
|----------|--------|
| 候选发现只在 lib 一处,agent 不复制算法 | 沿 1db492:CLI/agent 共用单一判定,杜绝割裂 |
| lib 判「项目形状+候选+置信度」,agent 判「证据是否够写正文」 | 形状是确定性事实;「AI 能否抽出足够证据」是 agent 层判断(evidence_insufficient 不进 lib enum) |
| `generate` 与 `list_only` 二分 | 用户拍板「看能否独立运行」:有运行/部署证据(high)才单独成篇;library-like(medium)与 submodule 只入总架构清单,防 shared libs 过量生成(Codex 修正:workspace member 只是边界证据≠可独立运行) |
| scope 元数据 = 分层身份地基 | 防重复生成 + per-candidate 补缺;以 `scope_source_roots` 物理路径做匹配锚点(slug 算法演进不破补缺) |
| 全输出排序(目录遍历/glob 展开/evidence/候选序) | missing-only + 补缺语义要求同一 fixture 双跑 bit-identical(Codex 补强为硬规则) |

**与现有系统的关系 / 边界**:

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `lib/su-init/index.mjs` | 删旧 `detectArchitectureCandidate`(单数),新导出 `detectArchitectureCandidates`;summary 字段更名 `architectureCandidates` | 三步脚手架、无 LLM、IGNORE_DIRS/MARKER_FILES 基础设施(扩展复用) |
| `skills/su-init/scripts/init.mjs` | stdout 字段更名 | 既有 counts/warnings/verification 输出 |
| `skills/su-init/SKILL.md` | 「旧项目架构生成」改写为三模式规程 + scope frontmatter + 总架构渲染 profile + 回执契约 | init lib 调用契约、节点 manifest 声明、wx 写盘纪律 |
| `templates/docs/01_架构设计/_模板_架构.md` | frontmatter 区注释加 scope 字段可选说明(轻,additive) | 章节结构、「多项目→各一份+总架构」既有指引 |
| 契约三副本(plugin canonical + 项目副本 + `su-oriel/server/src/indexer/default-docs-structure-contract.yaml`) | `01_架构设计/` 条目加注释级 scope 字段声明,**三份同步**(触碰则三份,防漂移;Codex 共识) | 结构/治理字段/`split_by_part` 既有语义 |
| Console / indexer | 不碰代码;scope 字段语义仅 su-init lib 自身解析 | Oriel 行级 frontmatter parser 现状(决定 D4 的 inline list 写法) |
| 消费方确认 | 旧接口仓内消费者仅 SKILL 文档/init.mjs/测试,无外部 npm 消费(Codex 复核确认)→ 删旧导出安全,本仓同步改 | —— |

## 三、关键决策与取舍(D1–D9 · Claude×Codex 协商结论)

> 协商:`slot1_codex`(child job `job_c4779eb94687`,mode=consult,**1 轮达成共识**,Codex 给出 4 个真盲点修正 + 2 个 open question 裁决,非橡皮图章)。4 锚点反思见本节末 + EventJournal。

- **D1 接口签名**:删旧单数导出,新导出 `detectArchitectureCandidates({projectRoot,resolver})`,`initProjectScaffold` 挂 `summary.architectureCandidates`,init.mjs stdout 同步。返回模型(Codex 方案 A 补强):`{ mode, reason, candidates[], overviewTargetPath, overviewExisting, excluded[], scopeConflicts[], existingArchitectureDocs, capLimit }`;candidate = `{ id, path, kind: "source_root"|"submodule", disposition: "generate"|"list_only", confidence: "high"|"medium", evidence: string[], targetPath, existing: bool }`。`multiple_source_roots` reason 删除;`excluded[]`(framework_shell_merged / root_aggregator / nested_absorbed)透明可审计。否决:保留兼容 wrapper(无外部消费者,两形状长期并存有害);收缩为只总览(低于用户拍板)。

- **D2 候选发现与置信度**:**运行/部署证据才 high**(scripts.start|dev|serve、bin/main 字段、Dockerfile、compose service、pyproject `[project.scripts]`、入口源文件 main.go|src/main.rs|main.py|manage.py|app.py 等);仅 marker / workspace member 无入口 → medium(library-like)。**workspace member 只算边界证据不抬置信度**(Codex 最大修正,防 shared libs 成篇)。置信度两档,否决三档(low 无明确语义=假精度)与 0-1 数值(假精确)。disposition:high→generate;medium→list_only。

- **D3 排除与归并规则**:
  (a) **框架壳归并**:模式=「壳目录+宿主证据」对(非裸名单,呼应用户拍板「看运行证据」):`src-tauri/`+根 tauri.conf.json|@tauri-apps 依赖;`android|ios/`+根 react-native|expo|capacitor 证据。宿主证据不存在则不归并(独立 Rust 工程恰好叫 src-tauri 仍成候选)。壳 path 并入宿主候选 `scope_source_roots`。名单为 lib 内数据表,可扩;漏网壳(flutter/electron 等)代价=多一份带置信度的浅文档+回执可删,可接受。
  (b) **root aggregator**:根声明 workspaces → 默认排除;**但根有硬运行证据(Dockerfile / 入口源文件)→ 仍纳入候选 id="root"**(Codex open question 裁决:纳入;monorepo-root-app 不误杀。scripts.start 在 monorepo 根常为 turbo/lerna 转发,不作 root 纳入依据)。
  (c) **嵌套吸收**:已有祖先候选的 marker 目录被吸收,除非自身是显式 workspace member。
  (d) **submodule**:`.gitmodules` 条目→`kind=submodule`、`disposition=list_only`(不深入生成,需求边界),计入总架构清单,**不占 cap**;其路径从 marker 扫描中整体排除(Codex 抓的重复枚举盲点)。
  (e) examples/fixtures/testdata 沿用 IGNORE_DIRS。

- **D4 scope 元数据**:`architecture_scope: <path 派生 slug>`(根候选="root",总架构固定="overview")+ `scope_source_roots: ["repo/相对路径", ...]`。两字段不冗余:scope 是身份/命名 key,source_roots 是补缺匹配的物理锚点。**frontmatter 必须写单行 inline list**——Oriel parser 是行级 `key: value`,多行 YAML 块会 partial(Codex 抓的盲点,`su-oriel/server/src/indexer/document-parser.ts:90`)。slug=路径小写、非字母数字→`-`(apps/web→apps-web);冲突时 lib 断言并附 deterministic 短 hash(必要,非过度设计)。**single 模式也写 scope**(身份统一;新文档重跑走 existing 精准幂等,旧 1db492 无 scope 文档仍整体跳过,向后安全)。schema 不改(架构 md 无 schema,1db492 D4 结论沿用;indexer 容忍额外字段已验证)。

- **D5 判定顺序与补缺状态机**(Codex 顺序修正:无 scope 已有架构优先于一切):
  1. 收集 `docs/01_架构设计/` 非模板 md 并解析 frontmatter → **任一缺 `architecture_scope`** ⇒ `mode=skip, reason=architecture_exists`(用户拍板保守语义,零写入);
  2. 全部带 scope(或无文档)→ 候选发现;scoped 文档按 `scope_source_roots`∩候选 path 匹配 → 对应候选 `existing=true`(per-candidate missing-only,重跑 su-init 自然补缺);`scope=overview` 文档 → `overviewExisting=true`,总架构不重写;
  3. **已知限制**:overview 已存在而本轮补写了新子架构 → 旧总架构不会更新链接(missing-only 不破),回执显式提示用户对话补改。

- **D6 mode 推导**(generatable = kind=source_root && disposition=generate,含 existing;cap 按 generatable 计,submodule/list_only 不占):`0 候选→skip(no_source)`;`generatable==1→single`(medium 候选回执列出,不写总架构);`2..8→layered`;`>8→overview_only`(全转 list_only,只写总架构+候选清单);`generatable==0 && 候选>0→overview_only`(纯库仓:总架构=清单+可证依赖,不让用户空手)。**cap=8**:C4 全景可读性 7±2、token 成本、>8 的仓更需总览先行;lib 常量不做配置项。否决 5(常见 web 全家桶即 5)与 12(成本/幻觉线性放大)。

- **D7 命名**:子架构 `<subsystem-slug>-架构.md`,总架构 `<project-slug>-总架构.md`(对齐契约注释「前端架构/后端架构/… + 系统总架构」与现存命名风格);lib 断言全部 targetPath 互不冲突且不与既有文件冲突,冲突附 deterministic 后缀。**不新建 `_模板_总架构.md`**(1db492 D4 跨仓多副本维护教训;现模板已自带多项目指引)——但 SKILL 必须给**固定渲染 profile**(Codex 补强,防各项目总架构结构漂移):总架构固定填模板五章:一概述(系统全景)、二整体结构图(子系统框图,仅证实连线)、五核心模块(=全部候选清单表:子系统/路径/职责/置信度/状态/文档链接)、六关键流程(仅可证实关联:import 方向/HTTP client 指向/配置引用;推不出标「未推断」)、十一相关文档(**仅链接实际写成功的子架构**)。

- **D8 契约/模板改法**:`_模板_架构.md` frontmatter 注释加 scope 可选字段说明(单处,真实价值:人/AI 手写时可标 scope 参与补缺);契约 `01_架构设计/` 条目加注释级 scope 声明,**三副本同步**(plugin canonical + 项目副本 + Oriel fallback;Codex open question 裁决:「触碰则三份同步」,半同步=未来 reconcile 噪声;Oriel 侧为 3 行注释级改动,走 su-oriel 子仓流程)。用户拍板「契约/模板同步小改」由此完整落地,不打折。

- **D9 写入顺序与失败恢复**:先子架构(按候选 id 排序逐个 final-path `wx` 独占写,单个 EEXIST/失败→记 skipped 继续其余),后总架构(输入=实际写成功清单,失败者不链接、回执如实);**写前二次 detect 的中止条件=出现新的「无 scope 非模板 md」**(外部干扰)——本轮自写的 scoped 子架构走 existing 语义,不算形状变化(Codex 抓的状态机自踩盲点);无事务回滚(missing-only 下半成品合法,重跑自然补缺)。

## 四、核心流程 / 逻辑

**`detectArchitectureCandidates` 算法**(全步骤排序,bit-identical):

```
0. existingDocs = 递归(docs/01_架构设计/) 非 _模板_ .md,解析 frontmatter(行级容忍)
     任一缺 architecture_scope ⇒ { mode:"skip", reason:"architecture_exists", existingArchitectureDocs }
1. submodulePaths = parse .gitmodules(排序)→ kind=submodule 候选(list_only,不占 cap)
2. workspaceMembers = package.json#workspaces ∪ pnpm-workspace.yaml ∪ Cargo.toml#[workspace].members
     ∪ go.work use(glob 展开排序;目录须存在且含 marker)
3. markerDirs = depth≤3 扫描(IGNORE_DIRS + submodulePaths 整体排除)
4. 候选池 = workspaceMembers ∪ markerDirs;嵌套吸收(祖先候选吸收子孙,显式 member 除外)
5. root:声明 workspaces ⇒ 默认 aggregator 排除(记 excluded);
     但有硬运行证据(Dockerfile/入口源文件)⇒ 纳入 id="root"
6. 壳归并:壳模式表 [{shellDirPattern, hostEvidencePredicate}] 命中 ⇒ 移除候选、
     记 excluded(framework_shell_merged)、path 并入宿主 scope_source_roots
7. evidence 收集(排序)+ confidence:run/deploy 证据→high;否则 medium
8. disposition:high→generate;medium|submodule→list_only
9. existing 匹配:scoped 文档 scope_source_roots ∩ 候选 path
10. cap:|generate| > 8 ⇒ mode=overview_only(全转 list_only)
11. mode:0 候选→skip(no_source);generatable==1→single;2..8→layered;
      generatable==0 且候选>0→overview_only
12. targetPath 派生 + 全局唯一断言(冲突附 deterministic 短 hash)
```

**生成文档 frontmatter**(单行 inline list,Oriel 行级 parser 兼容):

```yaml
---
doc_type: architecture
updated: 2026-06-06
generated_by: su-init-ai
human_verified: false
architecture_scope: apps-web
scope_source_roots: ["apps/web", "src-tauri"]
---
```

**agent 层三模式**(eligible 判断全部以 lib 返回为准,SKILL 不复制算法):

| mode | 规程 |
|------|------|
| single | 现行路径(证据门槛四块→渲染→二次 detect→wx)+ scope frontmatter |
| layered | per-candidate 证据门槛(以候选目录为根:目录树+≥1 grounding 源+四块可填)→不满足记 evidence_insufficient 继续其余→子架构逐个 wx(「覆盖范围/不覆盖」章互指)→总架构(D7 固定 profile,输入=写成功清单)→wx |
| overview_only | 仅总架构:全部候选清单表(含 list_only/submodule/excluded)+ 仅证实关联 + 点名补生成提示 |

**回执契约**:生成清单 / 每候选终态(generated · existing · evidence_insufficient · list_only · submodule · shell_merged · aggregator_excluded)/ 总架构状态(generated|existing|stale-link 提示)/ AI 生成提醒 + review 建议 / 超限时全部候选+点名补生成指引。
