# 文档中心治理重构 — 技术设计

> 一句话:把扁平文档列表重构为「文档治理中心」——`documents` 列表轻扩展 `governance` 投影 + 客户端派生覆盖度/健康度,保留 Markdown 阅读器。｜ 最后更新: 2026-05-29

## 一、设计概述

| 项 | 说明 |
|----|------|
| 名称 | 文档中心治理重构(文档治理中心) |
| 核心职责 | 文档中心从"按 doc_type 扁平筛选"转为"按目录/档位导航 + 文档覆盖度 + 健康度",且不与 Requirements/Tasks 看板重叠 |
| 设计原则 | ① 无 DB migration ② 无新增重依赖 ③ 文档覆盖 ≠ 交付状态 ④ 派生规则单一来源 ⑤ 复用现有看板 client-projection 模式 |
| 需求来源 | `docs/02_需求设计/文档中心的ui重构-2318ee-需求.md` |
| 覆盖范围 | `documents` API 增 `governance` 投影;web `DocumentsPage` 重构为治理三栏;client 端派生覆盖度/分组/健康度 |
| 不覆盖 | DB migration、Prisma 关系、正文编辑、看板状态流转、树/图重依赖、agent 专用 UI、`stale` 健康度(无 per-doc 真相) |

## 二、方案与架构

```
                 docs/*.md (人读真相源)
                       │  scan
                       ▼
   ┌─────────────────────────────────────────────┐
   │ indexer: parse → DB Document(frontmatterJson) │
   │          + buildDocumentMap → cache(.ccb)     │
   └───────────────┬───────────────┬───────────────┘
                   │ (二者共用同一规则)│
                   ▼               ▼
        [NEW] deriveDocumentGovernance(纯 builder)
        tier / requirementId / entityStatus / taskId / healthFlags
                   │
                   ▼
   GET /documents → DocumentView + governance{...}   ← 单一来源,项内已带 documentId
                   │  fetchDocuments(已有)
                   ▼
   web project-store.documents (已有,无需第二份状态)
                   │  client projection (mirror node-board-config)
        ┌──────────┼────────────────┐
        ▼          ▼                ▼
   左:目录/档位  中:覆盖度卡+健康度   右:MarkdownViewer(保留)
```

| 关键原则 | 说明 |
|----------|------|
| 单一派生来源 | tier/entityStatus 等规则抽成 `deriveDocumentGovernance` 纯函数,indexer 写 cache 与 route 响应共用,杜绝地图/UI 分叉 |
| 不暴露扫描模型 | 不 export 吃 `ParsedDocumentRecord` 的 `buildDocumentMapEntries`;builder 吃归一化最小形,route 从 DB `Document` 适配 |
| 单源单请求 | governance 挂在现有 `/documents` 列表项上(已带 `documentId`),不新增端点/store 状态/不按 path 回连 |
| 覆盖 ≠ 交付 | 覆盖度只判"文档是否存在";需求状态仅作上下文标签,不进覆盖评分 |

**与现有系统的关系 / 边界**:

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `server document.routes.ts` | [MODIFY] list 响应每项 additive 加 `governance` | 既有字段形状不变 |
| `server project-indexer.ts` | [REFACTOR] 抽 `deriveDocumentGovernance` 共用 | `document-map.json` 缓存输出语义不变(回归兜底) |
| `web DocumentsPage.tsx` | [REWRITE] 三栏治理视图 | 复用 `Card/Badge/EmptyState/MarkdownViewer` |
| `web types/document.ts` `console-api.ts` | [MODIFY] `DocumentView` 加 `governance` | fetchDocuments/detail 流程不变 |
| Requirements/Tasks 看板 | 不动 | 治理中心只读跳转,不复制状态流转 |

## 三、关键决策与取舍（Claude / Codex 协商结论）

- **投影位置**:选【扩展 `/documents` 加 `governance` 子对象】;否决【新增 `/document-map` 端点】——后者引双请求/双 store 状态/按 path 回连,关注点分离收益不足(Codex 质疑,Claude 采纳)。
- **派生来源**:选【DB 现算 + 抽共用纯 builder】;否决【route 直读 `docs/.ccb/index/document-map.json` 磁盘缓存】——缓存可能缺失/过期且会与 DB 派生分叉;缓存仅作调试/派生产物。
- **不 export 现成函数**:否决【直接 export `buildDocumentMapEntries`】——它吃扫描期 `ParsedDocumentRecord`,会把扫描模型泄漏到 API 层(Codex 指出)。
- **覆盖度语义**:选【只判"文档存在"】——"有设计"=∃ `technical_design` 且 `requirement_id`==该需求;"有任务"=∃ `dev_task` 且 `requirement_id`==该需求;"缺口"=生效中需求缺设计或任务。需求状态仅上下文标签。**依据用户 Q1 选"治理中心 over 交付链",隐含确认"覆盖 ≠ 交付"**。
- **左栏 MVP**:选【top-level 目录 + tier section + 搜索】;可折叠树用本地 state 轻实现或后置,不引库。
- **`stale` 撤除**:无 per-doc stale 真相,不承诺(Codex 指出);健康度只保留 `parseStatus=error`。
- **list/detail 类型分层**:拆 `DocumentBaseView` 公共形,`governance` **仅挂 list(`DocumentView`)**,`DocumentDetailView` 不带——否则 `DocumentDetailView extends DocumentView` 会让 required `governance` 传染到 detail 链路(preview drawer / reconcile reader / mocks)被动要求该字段(Codex 执行视角指出,为切片最危险假设)。

## 四、核心流程 / 逻辑

```
list 请求 → findMany(documents) → 对每项:
   parse frontmatterJson → deriveDocumentGovernance(doc, ctx)
   ctx = { requirementStatusById(全量需求扫一遍), resolver }
   → DocumentView{ ...既有, governance:{tier,requirementId,entityStatus,taskId,healthFlags} }
client(派生,不在 API 评分):
   documents → group by (目录, tier)              → 左栏导航
   documents → group by requirementId, 判 docType 集合 → 覆盖度卡(缺口高亮)
   documents.filter(parseError | 未绑定)           → 健康度/异常面板
   选中 → /documents/:documentId → MarkdownViewer
```

| 处理规则 | 说明 |
|----------|------|
| 命名归一 | `requirement_id↔requirementId`、`task_id↔taskKey` 在 builder 集中转换,前端只见 camelCase |
| 未绑定文档 | `requirementId` 为 null 的文档进"未绑定"一等分组,不静默丢弃 |
| 覆盖判定 | 仅统计 `requirement_id` 已填的绑定文档;缺绑定显式提示,不误判为"需求缺设计" |
| 性能 | 本项目 docs 数十份,list 现算可接受;量级增长再引缓存快路 |

## 七、接口设计

| 端点 | 方法 | 作用 | 变更 |
|------|------|------|------|
| `/api/projects/:projectId/documents` | GET | 文档列表 | [MODIFY] 每项 additive 加 `governance` |

`governance` 形状(每个 `DocumentView` 项):

```ts
governance: {
  tier: "生效中" | "历史" | "归档";
  requirementId: string | null;   // null = 未绑定
  entityStatus: string | null;    // 该文档或其所属需求状态(仅上下文)
  taskId: string | null;
  healthFlags: { parseError: boolean };  // stale 暂不提供(无 per-doc 真相)
}
```

覆盖度不在 API 评分,由 client 从 `documents[]` 派生。

> `governance` **仅挂 list 项(`DocumentView`)**;`DocumentDetailView` 不带,二者共用新 `DocumentBaseView`。`healthFlags.parseError` 由 route 读 `parseStatus` 派生,**不暴露 `parseError` 原文**到 list。

## 十、迁移影响与风险

- **受影响**:`documents` 列表 API 形状(additive)、indexer 派生路径(重构)、文档中心前端(重写)
- **打法**:先抽 builder + 回归缓存语义 → 再加 API governance → 再重写前端;每步独立可测
- **回滚 / 恢复**:纯 additive + 无 migration,`git revert` 即可;无数据迁移风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| `requirement_id` 缺失致覆盖误判 | 中 | 中 | 显式"未绑定"分组,只统计已绑定 |
| 重构使 cache 与 route 派生分叉 | 低 | 中 | 抽单一 builder 共用 + 缓存语义回归测试 |
| 文案混入交付状态再撞看板边界 | 中 | 中 | UI 统一叫"文档覆盖",需求状态仅上下文标签 |
| list route 现算遍历成本 | 低 | 低 | docs 数十份可接受;预留缓存快路 |
