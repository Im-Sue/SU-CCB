## 一、设计概述

| 项 | 说明 |
|----|------|
| 名称 | 需求详情子任务阅读弹窗 |
| 核心职责 | 需求详情页点击子任务 → 打开只读 reader 弹窗，内联渲染该子任务 canonical dev_task markdown 正文，不再跳转任务详情页 |
| 设计原则 | 全程 additive；镜像现有 requirement reader；canonical 文档解析放后端（前端不挑文档）；只读；复用 Modal/状态机/MarkdownViewer |
| 需求来源 | `docs/02_需求设计/需求详情里的子任务交互效果-560b82-需求.md` |
| 覆盖范围 | 需求详情页子任务点击交互 + 一个后端只读取文端点 + 共享 markdown/路径 helper 抽取 |
| 不覆盖 | 不动任务详情页 TaskDetailPage 及其它入口；不在弹窗内提供编辑/流转/工作区操作；不做相对图片资产解析；不改任务列表 API 负载；不改 schema |

## 二、方案与架构

```
需求详情页 RequirementDetailPage
  子任务 <button> onClick
     ── 旧: navigate(`/tasks/:id`)            ← 删除
     ── 新: setSubtaskReader(task)            ← TaskView
        │
        ▼  useEffect([subtaskReader]) + seq-ref 防竞态
   fetchTaskMarkdown(projectId, task.id)  (console-api)
        │  GET /api/projects/:projectId/tasks/:taskId/markdown
        ▼
   loadTaskMarkdownBody(prisma, projectId, taskId)   (server: task-markdown.service)
        │  1. task = findFirst{ id, projectId } include project   → 无→NotFound
        │  2. canonical 候选列表:
        │       primary  = doc{ id:primaryDocumentId, projectId, kind:dev_task, taskKey } (若有)
        │       fallback = doc[]{ projectId, taskKey, kind:dev_task } orderBy path asc, 排除已入选 id
        │  3. 逐候选: resolveProjectPath(localPath, doc.path) 守卫 + readFile → 首个成功者胜
        │       全部失败 → NotFound
        │  4. return { path: doc.path, content: extractMarkdownBody(raw) }   (body-only)
        ▼
   { path, content }  →  MarkdownReaderState(ready/empty) | 404→not-found | err→error
        ▼
   <Modal size="reader" title="任务文档 · {task.title}">  MarkdownViewer(content)
```

| 关键原则 | 说明 |
|----------|------|
| additive | 仅新增端点/helper + 替换一个 onClick；不删任务详情页、不改既有契约 |
| 镜像 requirement reader | 端点形态 `{path, content}` body-only、前端状态机/Modal/MarkdownViewer 全部沿用，降低认知与回归面 |
| canonical 后端解析 | 「对应任务文档」的判定（primaryDocumentId）落后端，前端只传 task.id，不暴露文档选择器 |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| `web/.../RequirementDetailPage.tsx` | 改子任务 onClick + 新增 subtask reader state/effect/Modal | 子任务按钮视觉/结构、其它 navigate(870/1002)、ai/design reader |
| `web/.../console-api.ts` | 新增 `fetchTaskMarkdown` | 既有 fetch 函数 |
| `server/.../task` 模块 | 新增 `task-markdown.service.ts` + 路由 | `GET /api/tasks/:taskId` 等既有端点不动 |
| `server/src/lib/`（新） | 抽中立 `extractMarkdownBody` / `resolveProjectPath` | requirement service 以 re-export 保持测试兼容 |
| 任务详情页 / `/tasks/:id` 路由 | **不动** | 看板/迭代/我的工作/活动流/命令面板入口照旧 |

## 三、关键决策与取舍

- **取文走后端端点（选 A）**：否决 B（前端 `fetchTaskDetail` + `linkedDocuments[0]` + `/api/documents/:id`）——泄漏 canonical 选择到 UI、多一次往返、`/api/documents/:id` 返回含 frontmatter 的 raw（document.routes.ts:103-118）需前端再剥、且拉入整套 task detail；否决 C（任务列表 API 塞 linkedDocuments）——膨胀所有列表消费方负载。
- **canonical 解析 = primaryDocumentId + taskKey 守卫 + 候选降级**：primary 查询带 `projectId + kind:dev_task + taskKey: task.taskKey`（不只 id），避免脏 primary 指到同项目其它 taskKey 时读错文档（比 fallback 失败更坏）。primary 文件缺失不直接 404，而是构造「primary 候选 + fallback 候选」列表逐个 containment+readFile，全失败才 NotFound（容忍投影漂移）。
- **共享 helper 抽中立 lib**：`extractMarkdownBody`（现已在 requirement-edit.service.ts:103 导出，另在 reindex/indexer/conformance 各有复制）抽到 `server/src/lib/markdown.ts`，requirement service re-export 保持测试兼容；task 模块从中立 lib import，避免 task→requirement-edit 语义耦合。路径 containment 复用 `resolveProjectPath`（现为 requirement-reindex.service.ts:564 局部函数）抽到 `server/src/lib/project-path.ts`。**本需求只抽这两处，indexer/conformance 的其余复制留作后续清理，不在本范围扩散。**
- **空 vs 不存在分离**：弹窗区分「任务文档正文为空」（读到但 body 空）与「任务文档不存在或尚未进入索引」（404），与现有 requirement/design reader 一致。
- **状态类型泛化**：`RequirementMarkdownState` → 改名 `MarkdownReaderState`（ai/design/subtask 三处 reader 共用），避免命名误导。

> **4 锚点反思（Codex design consult job_5ba027f9c678 后）**
> - 我同意：primary 加 taskKey 守卫、候选列表降级、empty/not-found 文案分离、状态类型泛化——都比我初稿更稳更清晰。
> - 我不同意/修正：我初稿把 `renderTaskMarkdownContent` 的 empty 与 not-found 文案合并是错的，已拆分。
> - 我的盲点：**漏了路径 containment**——直接 `join(projectRoot, doc.path)` 在 doc.path 为脏数据（`../` / 绝对路径）时可越界读盘。已采纳 `resolveProjectPath` 守卫并列入必测。
> - 接下来：写设计文档（本篇）→ 自然停在设计完成态；建议下一步 task_breakdown（规模小，倾向 1 全栈子任务）。

## 四、核心流程 / 逻辑

```
后端 loadTaskMarkdownBody:
  findFirst Task{id,projectId}+project ─无→ 404
        │
   候选 = [primary?(守卫 taskKey/kind/projectId)] ++ [fallback dev_task by taskKey, 排除已选]
        │
   for doc in 候选:
       try { abs = resolveProjectPath(localPath, doc.path); raw = readFile(abs) }   // 越界/缺文件→跳过
       catch → 下一个
       成功 → return { path: doc.path, content: extractMarkdownBody(raw) }
        │
   候选全失败 → 404
```

| 处理规则 | 说明 |
|----------|------|
| 路径 containment | 每个候选 `doc.path` 必经 `resolveProjectPath` 守卫，越界/绝对路径直接判失败，绝不读盘 |
| canonical 优先 | primary 候选在前；primary 存在且可读时不看 fallback；不向前端暴露选择器 |
| 投影漂移容忍 | primary 文件缺失/脏指向 → 降级到 fallback 候选；全部不可读才 NotFound |
| 空 vs 404 | 文件可读但 body 空 → 200 `{path, content:""}` → 前端「正文为空」；无任何可读候选 → 404 → 前端「不存在或尚未进入索引」 |
| 前端防竞态 | subtask reader 用 seq-ref：A→B 快速点击时，A 的迟到响应被丢弃；关闭后迟到响应也丢弃 |

## 五、测试策略

- [ ] 单元 `loadTaskMarkdownBody`：primary 命中且 body-only（去 frontmatter）；primary 脏指向同项目其它 taskKey → 不读它、走 fallback；primary 文件缺失 → 降级 fallback；`doc.path` 越界（`../`/绝对路径）→ 判失败不读盘；无任何 dev_task 文档 → NotFound；body 空 → 返回空 content。
- [ ] 集成（端点）：`GET /api/projects/:pid/tasks/:tid/markdown` 200 返回 `{path, content}` body-only；task 跨 project → 404；无可读文档 → 404。
- [ ] 前端 `RequirementDetailPage.spec.tsx`：点击子任务 → 弹窗打开并渲染内容、**不发生路由跳转**；快速点击 A→B 只显示 B（迟到响应丢弃）；404 → 「任务文档不存在或尚未进入索引」；空 body → 「任务文档正文为空」；loading 态。

## 十、迁移影响与风险

- **受影响**：requirement-edit.service（re-export）、requirement-reindex.service（import 改源）、RequirementDetailPage（新增 state + 改 1 个 onClick）。
- **打法**：全 additive；helper 抽取保持原导出名，既有测试零改动验证；前端改动局限于子任务交互与新 Modal。
- **回滚**：纯增量，`git revert` 即可；无数据迁移、无状态变更。

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| `doc.path` 脏数据越界读盘 | 低 | 高（任意文件读取） | `resolveProjectPath` containment，必测越界用例 |
| primaryDocumentId 脏指向 | 低 | 中（读错文档） | primary 查询带 taskKey/kind 守卫 + 候选降级，必测 |
| 投影过期 / 文件缺失 | 中 | 低 | 候选降级 + 404 弹窗内联文案，不 toast 不跳转 |
| A→B 快速点击迟到响应错配 | 中 | 低 | seq-ref 丢弃过期响应，必测 |
| 相对图片资产不渲染 | 中 | 低 | 与现有 reader 同限制，明确不在本需求扩 |
