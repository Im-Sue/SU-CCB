# 归档按钮门控修复 · 技术设计（简版）

> 一句话：把归档按钮门控从【已交付才可点】改成【推进中(delivering)即可点】，配确认弹窗提示；能否真正 finalize 由 plugin 既有的唯一关卡判定 ｜ 最后更新: 2026-06-08
>
> **无独立 status** —— 跟随 `requirement_id: cmq4tn9ub44963ca7baa654c0`。

---

## 一、设计概述

**目标对齐**：那个一直点不动的"归档"按钮，根因是门控写反了——它要求需求"已交付(delivered)"才可点，可"已交付"恰恰是点完这个按钮才有的结果，于是死循环、永远灰。简版修法：**按钮只要需求"推进中(delivering)"就可点**；点下去到底能不能真归档，交给 plugin 那道**既有的硬关卡**判断（子任务没跑完合并会被拒绝并提示）。不在前端/后端再造一套门控，避免多处规则打架、相互漂移。

| 项 | 说明 |
|----|------|
| 名称 | 归档按钮门控修复（requirement archive gate · 简版） |
| 核心职责 | 解前端门控死循环；finalize 安全由 plugin 唯一关卡承担 |
| 设计原则 | 单一真相源（plugin 关卡）；前端只解死循环、不复制门控；最小改动 |
| 需求来源 | `docs/02_需求设计/归档按钮门控修复-a654c0-需求.md` |
| 覆盖范围 | 前端按钮门 `delivered→delivering` + 确认弹窗提示 + 配套测试 |
| 不覆盖 | worktree 运行态投影进 DB；后端 dispatch 归档专用守卫；kernel 两段式 archive 本体；su-batch 自动 finalize |

## 二、方案与架构

```
前端按钮  gate: status == "delivering"   （唯一前端判据，status 已在 API）
   │ 点击
确认弹窗  "确认归档需求" + 提示"请确认子任务已全部完成合并后再归档"
   │ 确认
POST .../dispatch  /ccb:su-archive(subject=requirement)
   │  后端：推进中即放行入队（不加归档专用守卫）
   ▼
anchor 执行 su-archive ──► plugin 唯一关卡（既有硬代码 + 测试）
   ├─ 合并/清理关卡 cleanupGateAllows：worktree 必 merged/archived，否则拒
   └─ finalize 关卡 requirement.finalize：子任务全终态 + md hash CAS，否则拒、保持原状
        │ 通过
        ▼  cleanup + requirement.finalize ─► delivered
```

| 关键原则 | 说明 |
|----------|------|
| 单一关卡 | finalize 安全只在 plugin 一层判定，前后端不重复实现 |
| 前端最小 | 按钮只读已有的 `status`，不引入新字段/投影 |
| 能点≠能成 | 推进中即可点；真正能否归档由 plugin 拒绝/放行并反馈 |

**与现有系统的关系 / 边界**：

| 涉及模块 | 本设计如何动它 | 保留 / 不动什么 |
|----------|----------------|------------------|
| 前端 `RequirementDetailPage` | 改归档门 + 确认弹窗加提示 | 其它生命周期按钮门不动 |
| 后端 dispatch（`anchor.routes.ts`） | **不动**——"推进中"已可派发 | 不加归档专用守卫 |
| plugin archive 关卡 | **只复用、不改** | cleanupGateAllows / requirement.finalize policy 全保留 |
| DB / 投影链（indexer/reindex） | **不动** | 不新增 worktree 运行态列 |

## 三、关键决策与取舍

> 本节是用户拍板 + Claude 核实的结论。

- **D1 门控口径取宽口径 `delivering`**：选 `status==="delivering"` 即可点；否决窄口径 a+（`delivering 且 worktree∈{merged,archived}`），因为它需要把 worktree 运行态投影进 DB + 加后端守卫，与 plugin 既有关卡重复、且制造"前端门 / 后端门 / plugin 门"三处同判据的漂移面。用户明确选简单 + 单一真相源。
- **D2 不投影 worktree 运行态**：前端只需 `status`（详情接口已返回），无需新增 DB 列、无需改 scan/reindex。
- **D3 后端不加归档专用守卫**：dispatch 对"推进中"放行即可；finalize 安全完全交给 plugin。**已核实 plugin 有两道硬代码关卡（非仅文档）**：`lib/worktree/index.mjs:751-753` `cleanupGateAllows`（要求 worktree merged/archived）；`lib/capability-outcome/index.mjs:113` + `generated-policy.mjs:215` `requirement.finalize`（要求子任务全终态 + md hash CAS），均带单测。
- **D4 早点用确认弹窗提示承接**：复用现有"确认归档需求"弹窗（`confirmActionText`），加一句"请确认子任务已全部完成合并后再归档"，不做前端硬门预防——子任务未完成时点击由 plugin 拒绝并提示。
- **D5 archived 与 merged 一视同仁**：按钮不分支、只派 `/ccb:su-archive`；archive 节点内部决定 `merged→cleanup+finalize` / `archived→finalize-only recovery`（`archived` 不会被二次 cleanup，Codex 已确认）。

## 四、核心流程 / 逻辑

```
正常路径：su-batch 跑完 → worktree merged（需求仍 delivering）
   用户进详情页 → 按钮(delivering)亮 → 点击 → 确认弹窗 → 派 su-archive
   → plugin：cleanup 关卡(merged ✓) → finalize 关卡(子任务全终态 ✓ + hash ✓) → delivered

早点路径：子任务还没跑完（worktree 非 merged，需求 delivering）
   按钮仍亮 → 用户点击 → 确认弹窗 → 派 su-archive
   → plugin：cleanup 关卡(非 merged ✗) → 拒绝 + 明确原因 → 需求保持 delivering（不误标 delivered）
```

**模拟示例**（端到端，命中多角色 + 状态机 + UI 流程，须给）：

需求 `R`（`cmq4tn9ub…a654c0`）处于 `delivering`。

- **场景甲（就绪）**：su-batch 已 merge-only，`worktrees/R.json` 为 `merged`。用户开详情页→`lifecycleDisabledReason("su-archive")` 见 `status==="delivering"`→返回 `null`（按钮亮）。点击→"确认归档需求"弹窗（含提示）→确认→`POST .../R/dispatch {command:"su-archive"}`→后端放行入队→anchor 跑 su-archive→cleanup 关卡过(merged)→requirement.finalize 过(子任务全终态+hash)→`R` 变 `delivered`。与命令行 `/ccb:su-archive R` 终态、证据一致（验收 #3/#5）。
- **场景乙（早点）**：子任务仍在跑，`worktrees/R.json` 为 `ready`。按钮同样亮（宽口径），用户点击→确认→派发→anchor 跑 su-archive→cleanup 关卡 `cleanupGateAllows=false`→拒绝并报"worktree 非 merged/archived，拒绝 finalize"→`R` 保持 `delivering`，不误标（验收 #4）。

| 处理规则 | 说明 |
|----------|------|
| 前端门 | 仅 `status==="delivering"` 可点；其余态禁用 + 准确 tooltip |
| 确认提示 | 弹窗含"先确认子任务已完成合并"，承接早点风险 |
| finalize 安全 | 由 plugin cleanup/finalize 两关卡判定，前后端不复制 |

## 协商与反思

**Codex 协商**：前一轮（slot4_codex, `job_c14edc846601`）针对的是复杂版（投影 + 后端守卫），Codex 同意新列并确认 `archived` 不二次 cleanup、guard 放 enqueue 前不污染 runtime。**简版是用户随后拍板收敛**——去掉投影与后端守卫。我据此重新核实 plugin 两道关卡确为硬代码兜底（`cleanupGateAllows` / `requirement.finalize` policy + 单测），确认简版安全后落档。

**4 锚点反思**：
- 我同意：plugin 是唯一权威关卡，前端只需解死循环，复制门控是过度设计。
- 我不同意（对自己早前方案）：撤回"投影 + 后端 guard"——它与 plugin 关卡重复、增双源漂移面，不如单一关卡干净。
- 我的盲点：起初把"按钮只在可 finalize 时才亮"当硬需求；它其实是 UX 增强，用户更要简单与单一真相。
- 接下来：等用户审完→进入 task_breakdown，本简版基本是单文件前端改动 + 测试，拆分会很轻。

**用户授权事项分类（③.6）**：
- 已由用户拍板：宽口径门控（`delivering`）+ 去投影/去后端守卫 + 保留确认弹窗提示。
- AI 自决实现细节：前端门写法、tooltip/提示文案、测试用例。
- 无命中：新依赖、schema/DB 变更（简版连 DB 改动都没有了）、公共 API 破坏、成本、许可证、合规。
